### TL;DR
#### 이 연구를 시작하게 된 배경과 동기는 무엇입니까?

그래프 데이터베이스는 지난 수십 년간 산업과 학계에서 관계형 데이터베이스보다 더 유연한 데이터 모델을 제공하면서 큰 주목을 받아왔습니다. 많은 개발자들이 기존 관계형 데이터베이스 애플리케이션을 그래프 데이터베이스로 이전하려는 관심을 보이고 있으며, Apache Age 같은 시스템들이 이러한 전환을 돕고 있습니다. 그러나 두 데이터베이스 모델 사이를 오가는 작업에는 본질적인 어려움이 따릅니다. 개발자가 한 모델에서 작성된 질의를 다른 모델로 변환해야 할 때, 조인과 관계의 차이, 집계 의미론의 차이 등 여러 복잡성으로 인해 잘못된 변환이 빈번하게 발생합니다. StackOverflow나 Neo4j 포럼 등 온라인 게시판에서 이러한 어려움을 호소하는 글이 다수 발견되며, 더욱 심각한 점은 기존 문헌에서조차 "동치"라고 주장된 질의들이 실제로는 동치가 아닌 사례가 여러 건 확인되었다는 것입니다.

관계형 데이터베이스 질의들 사이의 동치성 검증에 대해서는 상당한 연구 진전이 있었지만, 관계형 질의와 그래프 질의 사이의 동치성 검증 문제를 다룬 연구는 이전에는 존재하지 않았습니다. 이 문제의 핵심적인 난점은 두 데이터베이스가 근본적으로 다른 데이터 모델을 사용한다는 데 있습니다. 관계형 데이터베이스는 데이터를 테이블 형태로 조직하여 행과 열에 대해 조인이나 집계 같은 잘 정의된 연산을 수행하는 반면, 그래프 데이터베이스는 데이터를 노드와 엣지로 표현하며 질의는 주로 그래프 구조를 통한 관계와 순회를 표현합니다. 이처럼 데이터 표현 방식 자체가 다르기 때문에, 동치성을 정의하는 일과 그것을 검증하는 일 모두가 까다로워집니다.

#### 이 연구에서 제시하는 새로운 해결 방법은 무엇입니까?

이 논문은 SQL로 작성된 관계형 질의와 Cypher로 작성된 그래프 데이터베이스 질의 사이의 동치성을 검사할 수 있는 자동 추론 기법을 제시합니다. 핵심 통찰은 **그래프 데이터베이스 인스턴스에서의 경로가 관계형 데이터베이스에서 행들의 조인에 대응된다**는 점입니다. 즉, 그래프에서 노드와 엣지를 따라가며 패턴을 매칭하는 작업은 관계형 세계에서 여러 테이블을 조인하는 작업과 본질적으로 같은 일을 한다는 것입니다. 이 대응 관계 덕분에 Cypher 질의를 SQL로 구문 지향적으로 번역할 수 있을 뿐 아니라, 두 질의 사이의 동치성 검사도 가능해집니다.

전체 접근법은 세 가지 핵심 구성 요소로 이루어집니다. 첫째, **데이터베이스 변환기**라는 개념을 도입하여 한 데이터 모델의 데이터베이스 인스턴스를 다른 모델의 동치 인스턴스로 변환할 수 있게 합니다. 둘째, 이 변환기를 바탕으로 **표준 데이터베이스 변환기(SDT)**를 정의하여 그래프 요소를 관계형 테이블로 사상하는 변환 규칙의 집합을 만들고, 임의의 Cypher 질의를 유도 관계형 스키마 위의 SQL 질의로 변환하는 구문 지향적 트랜스파일레이션 규칙을 정의합니다. 셋째, 실제 현장에서는 목표 관계형 데이터베이스가 유도 관계형 스키마와는 다른 스키마를 사용하는 경우가 많기 때문에, **잔여 데이터베이스 변환기**를 구성하여 임의의 데이터베이스 변환기를 기준으로 그래프 질의와 관계형 질의의 동치성을 검사하는 방법론을 제안합니다. 이 방법론의 핵심은 문제를 **서로 다른 스키마 위의 SQL 질의들 사이의 동치성 검사**로 환원하여, SQL 동치성 검사를 위해 이미 개발된 기존 기법과 도구들을 활용할 수 있다는 점입니다.

#### 제안된 방법은 어떻게 구현되었습니까?

이 접근법은 **Graphiti**라는 새로운 도구로 구현되었으며, 410개의 벤치마크에 대해 광범위한 실험적 평가가 수행되었습니다. 이 벤치마크는 StackOverflow, 튜토리얼, 학술 논문 등 공개 플랫폼에서 수집한 45개, Cypher 경험이 있는 학생들이 SQL로부터 번역한 160개, 그리고 ChatGPT를 이용해 번역한 205개로 구성됩니다. 평가에서 Graphiti는 유계 검증을 위한 SQL 백엔드인 VeriEQL과 결합되었을 때 34개 벤치마크에서 동치성 위반을 찾아냈으며, 여기에는 실제 현장에서 수집된 사례, 수작업 번역 사례, GPT 생성 번역 사례가 모두 포함됩니다. 또한 집계가 없는 부분 집합에 대해서는 연역적 SQL 검증기인 Mediator와 결합하여, 테이블과 그래프의 크기에 제한이 없는 무계 동치성 검증을 수행할 수 있었습니다.

구현 과정에서 데이터베이스 변환기를 작성하는 부담이 크지 않음이 확인되었습니다. 각 질의 쌍에 대해 변환기 하나는 평균적으로 약 5.9개의 규칙으로 이루어지며 작성에 약 1분 정도가 걸렸습니다. 트랜스파일레이션 자체는 거의 시간이 들지 않아 평균 6.3밀리초 내에 완료되었으며, Graphiti가 생성한 SQL 질의는 실행 효율 면에서 수작업으로 작성된 질의에 견줄 만합니다. 실제로 33.3%의 벤치마크에서 Graphiti의 트랜스파일 결과가 수작업 작성 질의보다 더 빨랐으며, 절반 이상(51.1%)의 벤치마크에서 1.2배 이내의 속도 저하에 머물렀습니다.

#### 이 연구의 결과가 가지는 의미는 무엇입니까?

Graphiti는 동치성을 입증하는 검증뿐 아니라 동치가 아님을 밝히는 반박에도 모두 유용하며, Cypher 튜토리얼과 학술 논문에 숨어 있던 미묘한 버그까지 찾아낼 수 있음이 입증되었습니다. 특히 GPT로 트랜스파일된 질의 중 13%에서 동치성에 대한 반례가 발견되었으며, 전문가가 작성한 것으로 여겨지는 Neo4j 공식 튜토리얼 예제에서도 버그가 발견되었습니다. 이는 그래프 질의와 관계형 질의의 의미론적 차이가 실제 현장에서 얼마나 흔하게 잠복해 있는지를 잘 드러냅니다. 종합하면, 이 논문은 데이터베이스 변환기 기준 동치성이라는 형식적 정의 위에서 그래프 질의와 관계형 질의의 동치성을 추론하는 최초의 기법을 제안하고, 이를 Graphiti라는 도구로 구현하여 검증과 반박 양쪽에서의 실용성을 실증했다는 점에서 의의를 가집니다.

이 연구는 이질적인 두 데이터 모델을 동시에 추론하는 난제를 피하고, 근본적으로 다른 두 데이터 모델의 질의 동치성 문제를 잘 연구된 동질적인 SQL 동치성 문제로 환원하는 우아한 전략을 제시합니다. 이는 복잡한 문제를 더 다루기 쉬운 하위 문제로 환원하는 철학을 구현한 것으로, 향후 SQL 동치성 검증 분야에서 이루어질 어떠한 발전도 그대로 흡수하여 활용할 수 있는 모듈식 설계를 제공합니다. 또한 형식적 기반을 통해 트랜스파일레이션의 건전성과 완전성을 증명함으로써, 자동 번역 결과의 신뢰성을 보장하는 이론적 토대를 마련했습니다.

# Graphiti: 그래프 데이터베이스 질의와 관계형 데이터베이스 질의의 가교

## 연구의 배경과 동기

지난 수십 년간 그래프 데이터베이스는 산업계와 학계 모두에서 큰 주목을 받아왔습니다. 그래프 데이터베이스는 관계형 데이터베이스와 비교했을 때 더 유연한 데이터 모델을 제공하면서도 서로 다른 장단점을 가지고 있습니다. 이러한 흐름 속에서 많은 개발자들이 기존의 관계형 데이터베이스 애플리케이션을 그래프 데이터베이스로 이전(migration)하는 데 관심을 두고 있으며, Apache Age와 같은 시스템은 관계형 데이터베이스에 그래프 구성 요소를 통합하여 이러한 전환을 돕고자 합니다.

그러나 두 데이터베이스 모델 사이를 오가는 작업에는 본질적인 어려움이 따릅니다. 개발자가 한 모델에서 작성된 질의를 다른 모델로 변환해야 하는 경우가 많은데, 이 과정에서 조인(join)과 관계(relationship)의 차이, 집계(aggregation) 의미론의 차이 등 여러 복잡성으로 인해 잘못된 변환이 빈번하게 발생합니다. StackOverflow나 Neo4j 포럼 등 여러 온라인 게시판에서 이러한 어려움을 호소하는 글이 다수 발견됩니다. 더욱 심각한 점은, 기존 문헌에서조차 "동치(equivalent)"라고 주장된 질의들이 실제로는 동치가 아닌 사례가 여러 건 확인되었다는 것입니다. 이는 그래프 질의와 관계형 질의의 동치성을 엄밀하게 추론할 수 있는 방법론에 대한 수요가 점점 커지고 있음을 보여줍니다.

관계형 데이터베이스 질의들 사이의 동치성 검증에 대해서는 상당한 연구 진전이 있었지만, 관계형 질의와 그래프 질의 사이의 동치성 검증 문제를 다룬 연구는 이 논문 이전에는 존재하지 않았습니다. 이 문제의 핵심적인 난점은 두 데이터베이스가 근본적으로 다른 데이터 모델을 사용한다는 데 있습니다. 관계형 데이터베이스는 데이터를 테이블 형태로 조직하여 행(row)과 열(column)에 대해 조인이나 집계 같은 잘 정의된 연산을 수행합니다. 반면 그래프 데이터베이스는 데이터를 노드(node)와 엣지(edge)로 표현하며, 질의는 주로 그래프 구조를 통한 관계와 순회(traversal)를 표현합니다. 이처럼 데이터 표현 방식 자체가 다르기 때문에, 동치성을 정의하는 일과 그것을 검증하는 일 모두가 까다로워집니다.

## 핵심 아이디어와 접근 방법

이 논문은 SQL로 작성된 관계형 질의와 Cypher로 작성된 그래프 데이터베이스 질의 사이의 동치성을 검사할 수 있는 자동 추론 기법을 향한 첫걸음을 제시합니다. Cypher는 가장 널리 사용되는 그래프 데이터베이스 질의 언어입니다. 이 기법의 토대가 되는 핵심 통찰은 **그래프 데이터베이스 인스턴스에서의 경로(path)가 관계형 데이터베이스에서 행들의 조인에 대응된다**는 점입니다. 즉, 그래프에서 노드와 엣지를 따라가며 패턴을 매칭하는 작업은 관계형 세계에서 여러 테이블을 조인하는 작업과 본질적으로 같은 일을 한다는 것입니다. 이 대응 관계 덕분에 Cypher 질의를 SQL로 구문 지향적(syntax-directed)으로 번역할 수 있을 뿐 아니라, 두 질의 사이의 동치성 검사도 가능해집니다.

전체 접근법은 세 가지 핵심 구성 요소로 이루어집니다.

### 그래프-관계형 데이터베이스 동치성의 형식적 기반

앞서 언급했듯이, 두 데이터 모델 사이에 직관적인 대응 관계가 없다는 점이 동치성 추론을 어렵게 만듭니다. 이를 해결하기 위해 이 논문은 **데이터베이스 변환기(database transformer)** 라는 개념을 도입합니다. 이는 스키마 매핑(schema mapping)에 관한 기존 연구, 예컨대 Miller 등(2000)의 연구로부터 착안한 것으로, 한 데이터 모델(그래프)의 데이터베이스 인스턴스를 다른 모델(관계형)의 동치 인스턴스로 변환할 수 있게 해줍니다. 이 변환이 곧 그래프 질의와 관계형 질의 사이의 동치성을 정의하는 토대가 됩니다. 직관적으로 말하면, 두 질의가 동치라는 것은 변환기를 통해 서로 대응되는 데이터베이스 인스턴스들에 대해 두 질의가 항상 같은 결과를 내놓는다는 의미입니다.

### 구성에 의한 올바름이 보장되는 트랜스파일레이션

데이터베이스 변환기 개념을 바탕으로, 논문은 기본 정확성 명세(correctness specification)로 작동하는 **표준 데이터베이스 변환기(standard database transformer, SDT)** 개념을 제시합니다. 간단히 말해 SDT는 그래프 요소(노드와 엣지)를 관계형 테이블로 사상하는 변환 규칙의 집합으로, 그래프의 구조와 의미를 관계형 모델 안에서 유지합니다. 예를 들어 그래프 스키마의 노드는 관계형 스키마의 테이블로 변환되어 노드의 속성이 열이 되고, 엣지는 외래 키(foreign key)를 통해 이 테이블들 사이의 관계로 표현됩니다. 이렇게 만들어진 관계형 스키마를 **유도 관계형 스키마(induced relational schema)** 라고 부릅니다.

이 SDT를 바탕으로, 임의의 Cypher 질의를 유도 스키마 위의 SQL 질의로 변환하는 구문 지향적 트랜스파일레이션 규칙이 정의됩니다. 핵심은 부분 그래프에 대한 패턴 매칭을 수행하는 Cypher 경로 질의를 관계형 조인으로 사상할 수 있다는 것입니다. 다만 Cypher가 유연한 다단계 패턴 매칭을 지원하기 때문에, 단순한 패턴이든 복잡한 패턴이든 원래 그래프 질의의 의미가 SQL 조인에서 정확히 보존되도록 세심하게 처리해야 한다는 점에서 실제 번역은 상당히 까다롭습니다.

### 임의 스키마에 대한 동치성 검사 방법론

위의 트랜스파일레이션은 동치인 SQL 질의를 생성할 수 있지만, 그 동치성은 어디까지나 SDT를 기준으로 한 것입니다. 그런데 실제 현장에서는 목표 관계형 데이터베이스가 유도 관계형 스키마와는 다른 스키마를 사용하는 경우가 많기 때문에, 구문 지향적 트랜스파일레이션만으로는 충분하지 않습니다. 이 간극을 메우기 위해 논문은 임의의 데이터베이스 변환기를 기준으로 그래프 질의와 관계형 질의의 동치성을 검사하는 검증 방법론을 제안합니다.

이 방법론의 핵심 통찰은, Cypher와 SQL 질의 사이의 동치성을 직접 추론하는 대신 — 그렇게 하려면 그래프 구조와 관계형 구조를 결합한 복잡한 SMT 인코딩이 필요합니다 — 문제를 **서로 다른 스키마 위의 SQL 질의들 사이의 동치성 검사**로 환원한다는 데 있습니다. 이렇게 환원하면 SQL 동치성 검사를 위해 이미 개발된 기존 기법과 도구들을 활용할 수 있으며, 근본적으로 다른 두 데이터 모델을 동시에 추론해야 하는 난제를 피할 수 있습니다.

## 전체 접근법의 흐름

다음 그림은 Cypher와 SQL 질의의 동치성을 검사하는 전체 접근법의 구조와 데이터 흐름을 보여줍니다.

![접근법 개요](https://arxiv.org/html/2504.03182/x1.png)

이 방법은 네 가지 입력을 받습니다. 그래프 질의인 Cypher 질의 $Q\_G$, 관계형 질의인 SQL 질의 $Q\_R$, 그래프 및 관계형 데이터베이스의 스키마, 그리고 데이터베이스 변환기 형태의 정확성 명세 $\Phi$입니다. $\Phi$를 기준으로 $Q\_G$와 $Q\_R$의 동치성을 확립하기 위해, 방법은 먼저 유도 관계형 스키마와 SDT를 도출합니다. 그런 다음 구성에 의한 올바름이 보장되는 트랜스파일레이션 기법을 적용하여, SDT 기준으로는 $Q\_G$와 증명 가능하게 동치이지만 $\Phi$ 기준으로는 아직 동치가 아닌 SQL 질의 $Q\_R'$을 생성합니다. 마지막 단계에서는 유도 스키마와 목표 관계형 스키마 사이의 의미론적 "차이(diff)"를 계산하여 두 관계형 인스턴스를 정렬하는 잔여 데이터베이스 변환기(residual database transformer)를 구성하고, 기성품(off-the-shelf) SQL 동치성 검사기를 사용해 $Q\_R'$과 $Q\_R$의 동치성을 검사합니다.

이 방법론은 두 가지 큰 장점을 가집니다. 첫째, 사용자가 단지 Cypher 질의를 SQL로 번역하고 싶고 기반이 되는 관계형 스키마에는 신경 쓰지 않는 경우(또는 원하는 스키마가 유도 관계형 스키마와 같은 경우), 구성에 의한 올바름이 보장되는 트랜스파일레이션을 그대로 활용할 수 있습니다. 둘째, 데이터베이스 변환기 형태의 임의의 정확성 명세가 주어지면, 제안된 트랜스파일레이션과 기존 SQL 자동 추론 도구를 결합하여 임의의 Cypher–SQL 질의 쌍의 동치성을 추론할 수 있습니다.

## 구현과 평가 개요

이 접근법은 **Graphiti**라는 새로운 도구로 구현되었으며, 410개의 벤치마크에 대해 광범위한 실험적 평가가 수행되었습니다. 이 벤치마크는 StackOverflow, 튜토리얼, 학술 논문 등 공개 플랫폼에서 수집한 45개, Cypher 경험이 있는 학생들이 SQL로부터 번역한 160개, 그리고 ChatGPT를 이용해 번역한 205개로 구성됩니다. 평가에서 Graphiti는 유계(bounded) 검증을 위한 SQL 백엔드인 VeriEQL과 결합되었을 때 34개 벤치마크에서 동치성 위반을 찾아냈으며, 여기에는 실제 현장에서 수집된 사례, 수작업 번역 사례, GPT 생성 번역 사례가 모두 포함됩니다. 또한 집계가 없는(aggregation-free) 부분 집합에 대해서는 연역적(deductive) SQL 검증기인 Mediator와 결합하여, 테이블과 그래프의 크기에 제한이 없는 무계(unbounded) 동치성 검증을 수행할 수 있었습니다. 마지막으로 Graphiti가 생성한 SQL 질의가 실행 효율 면에서 수작업으로 작성된 질의에 견줄 만하다는 점도 확인되었습니다. 이러한 결과들은 Graphiti가 동치성을 입증하는 검증(verification)뿐 아니라 동치가 아님을 밝히는 반박(refutation)에도 모두 유용하며, Cypher 튜토리얼과 학술 논문에 숨어 있던 미묘한 버그까지 찾아낼 수 있음을 보여줍니다.

종합하면, 이 논문은 데이터베이스 변환기 기준 동치성이라는 형식적 정의 위에서 그래프 질의와 관계형 질의의 동치성을 추론하는 최초의 기법을 제안하고, 이를 Graphiti라는 도구로 구현하여 검증과 반박 양쪽에서의 실용성을 실증했다는 점에서 의의를 가집니다. 이어지는 절들에서는 동기를 부여하는 구체적 예제, 형식적 기반, 동치성 검사 알고리즘의 세부 내용이 차례로 전개됩니다.
## 동기를 부여하는 예제: 겉보기에 같지만 실제로는 다른 두 질의

앞서 데이터베이스 변환기, 유도 관계형 스키마, 잔여 변환기 같은 핵심 개념을 큰 그림에서 소개했다면, 이 절에서는 실제 생의학(biomedical) 연구 데이터베이스에서 가져온 구체적인 예제를 통해 "동치처럼 보이는 두 질의가 사실은 동치가 아닐 수 있다"는 문제를 손에 잡히도록 보여줍니다. 이 예제는 그래프 SQL 분석을 연구한 선행 연구(Lin et al., 2016)에서 가져온 것으로, 그래프와 관계형이라는 두 데이터 모델이 동시에 등장하기 때문에 문제의 본질을 드러내기에 매우 적합합니다.

### 그래프 스키마와 관계형 스키마의 대응 관계

예제의 데이터는 세 가지 종류의 노드로 구성됩니다. 의학 개념을 나타내는 `CONCEPT`, 술어 논항(predication argument)을 의미하는 `PA`, 그리고 문장을 나타내는 `SENTENCE`입니다. 이들은 두 종류의 엣지로 연결되는데, `CS` 엣지는 개념과 술어 논항을 잇고, `SP` 엣지는 술어 논항과 문장을 잇습니다. 즉 `CONCEPT → PA → SENTENCE`로 이어지는 경로가 데이터의 기본 골격을 이룹니다. 각 노드 타입은 고유의 속성 키를 가지며, 예컨대 `CONCEPT`는 `CID`와 `Name`을, `PA`는 `PID`와 `CSID`를, `SENTENCE`는 `SID`와 `PMID`를 가집니다.

이 그래프 모델은 다섯 개의 테이블 `Concept`, `Cs`, `Pa`, `Sp`, `Sentence`로 이루어진 관계형 스키마로 표현됩니다. 흥미로운 점은 이 다섯 테이블이 그래프의 노드뿐 아니라 엣지에도 대응한다는 것입니다. 즉 노드 타입인 `CONCEPT`·`PA`·`SENTENCE`는 각각 테이블이 되고, 엣지 타입인 `CS`·`SP` 역시 별도의 테이블 `Cs`·`Sp`가 되어 기본 키(primary key)와 외래 키(foreign key)로 연결 관계를 표현합니다. 이러한 대응을 직관적으로 이해하기 위해, 같은 "Atropine"과 "Aspirin"이라는 항목을 담은 그래프 인스턴스와 관계형 인스턴스가 함께 제시됩니다. 그래프 측에서는 Aspirin과 Atropine이 두 개의 `Concept` 노드이고, `PA0`·`PA1`이 두 개의 `PA` 노드, `S0`·`S1`이 두 개의 `Sentence` 노드로 표현되며, `:CS`와 `:SP` 엣지로 서로 연결됩니다. 관계형 측에서는 이 동일한 정보가 다섯 테이블의 행으로 흩어져 저장됩니다.

### 두 질의가 같은 데이터에서 서로 다른 결과를 내는 이유

이제 핵심 장면입니다. 선행 연구에서 동치라고 주장된 SQL 질의와 Cypher 질의를 살펴봅니다. SQL 질의는 `CID = 1`인 개념 `c1`에 연결된 모든 개념을 찾고, `Cs`·`Pa`·`Sp` 테이블의 조인을 통해 그 개념들이 `c1`에 연결되는 경로의 빈도를 세는 것을 목표로 합니다. SQL 질의는 다음과 같습니다.

```sql
SELECT c2.CID, Count(*)
FROM   Cs AS c2, Pa AS p2, Sp AS s2
WHERE  s2.PID = p2.PID
  AND  p2.CSID = c2.CSID
  AND  s2.SID IN (
        SELECT s1.SID
        FROM   Cs AS c1, Pa AS p1, Sp AS s1
        WHERE  s1.PID = p1.PID
          AND  p1.CSID = c1.CSID
          AND  c1.CID = 1)
GROUP BY CID
```

여기서 주목할 부분은 안쪽의 서브쿼리가 `c1`과 연결된 문장들의 `SID` 집합을 구한 뒤, 바깥 질의가 `s2.SID IN (...)` 형태로 그 집합에 속하는지를 **멤버십 조건**으로만 검사한다는 점입니다. 즉 어떤 문장이 조건을 만족하는지 여부만 따지고, 그 문장에 도달하는 경로가 몇 개인지는 바깥쪽 계산에 영향을 주지 않습니다.

반면 Cypher 질의는 다음과 같습니다.

```cypher
MATCH (c1:CONCEPT {CID:1})-[r1:CS]->(p1:PA)-[r2:SP]->(s:SENTENCE)
WITH s
MATCH (s:SENTENCE)<-[r3:SP]-(p2:PA)<-[r4:CS]-(c2:CONCEPT)
RETURN c2.CID, Count(*)
```

이 질의는 먼저 `c1`에서 `CS-SP` 경로를 따라 도달 가능한 문장 `s`를 찾고, `WITH s` 절로 그 중간 결과를 다음 단계로 넘긴 뒤, 다시 그 문장들로부터 거꾸로 거슬러 올라가 모든 개념 `c2`에 이르는 경로의 빈도를 셉니다. 여기서 비동치성의 근원이 드러납니다. 첫 번째 `MATCH`가 동일한 문장 `s`에 도달하는 여러 경로를 만들어내면, `WITH s`를 거치면서 같은 `s`가 여러 번 중복되어 다음 단계로 전달될 수 있습니다. SQL의 `IN` 멤버십 검사는 이러한 중복을 자연스럽게 제거하지만, Cypher의 경로 기반 의미론은 그렇지 않기 때문에 두 질의의 빈도 계산이 어긋나게 됩니다.

실제로 앞서 제시한 동일한 데이터 인스턴스에서 두 질의를 실행하면, SQL 질의는 `c2.CID = 1`에 대해 `Count(*) = 2`를 반환하는 반면 Cypher 질의는 `Count(*) = 4`를 반환합니다. 두 결과는 관련된 개념의 `CID` 자체는 일치시키지만, 관계의 빈도를 나타내는 `Count` 열에서 명백히 갈립니다. 참고로 이 SQL 질의와 진짜로 동치인 올바른 Cypher 질의는 부록에 따로 제시되는데, 원래 질의의 버그는 중첩된 `MATCH`를 사용한 데서 비롯되며, 이를 `EXISTS` 조건으로 바꾸면 개별 매치 수준에서 올바르게 필터링되어 이중 집계가 사라집니다. 이처럼 표면적으로 같아 보이는 질의가 집계와 경로 매칭의 미묘한 의미론 차이 때문에 다른 답을 내놓는다는 사실이, 그래프 질의와 관계형 질의의 의미론적 차이를 자동으로 드러내는 추론 도구의 필요성을 강하게 뒷받침합니다.

### 데이터 동치성을 정의하는 장치: 데이터베이스 변환기

두 질의가 의미적으로 다르다고 결론짓기 위해서는, 같은 데이터에 대해 두 질의가 어떻게 동작하는지를 비교해야 합니다. 그런데 그래프와 관계형은 데이터 모델 자체가 다르기 때문에 두 입력 인스턴스가 결코 글자 그대로 동일할 수 없습니다. 따라서 질의 동치성을 추론하기에 앞서 먼저 **데이터 동치성**을 정의할 메커니즘이 필요하며, 이 역할을 데이터베이스 변환기가 담당합니다. 데이터베이스 변환기는 스키마 $\Psi$ 위의 그래프 인스턴스 $D$를 입력받아 관계형 스키마 $\Psi'$ 위의 관계형 인스턴스 $D'$를 생성합니다. 이 개념은 Clio로 대표되는 스키마 매핑 및 데이터 교환(data exchange) 연구에서 착안하여, 그래프와 관계형 데이터베이스 사이의 대응을 모델링하도록 일반화한 것입니다. 스키마 매핑을 논리적 제약식으로 표현할 수 있다는 [Clio](https://doi.org/10.1145/3183713.3190657) 등의 통찰이 여기서 그대로 활용되어, 변환기가 일종의 형식 명세로 기능하게 됩니다.

이 예제에 대한 변환기는 각 관계형 테이블이 그래프의 노드·엣지로부터 어떻게 생성되는지를 규칙으로 기술합니다. 예를 들어 다음 규칙을 봅시다.

$$
\texttt{CONCEPT}(\mathsf{cid}, \_),\ \texttt{CS}(\mathsf{cid,csid,cid,pid}),\ \texttt{PA}(\mathsf{pid},\mathsf{csid}) \to \texttt{Cs}(\mathsf{cid,csid})
$$

이 규칙은 직관적으로 "그래프 안에 두 노드 `CONCEPT`와 `PA`를 잇는 `CS` 엣지가 존재하고, 그 `CS`의 첫 두 속성이 $\mathsf{cid}$와 $\mathsf{csid}$이면, 관계형 테이블 `Cs`에 행 $(\mathsf{cid}, \mathsf{csid})$가 존재한다"는 의미입니다. 여기서 `CS`의 나머지 두 속성은 해당 엣지의 출발 노드와 도착 노드를 가리키는 외래 키 역할을 합니다. 이런 규칙들의 집합이 그래프 데이터와 관계형 데이터를 정렬(align)하는 사상(mapping)을 정의하며, 모든 변수는 암묵적으로 전칭 한정(universally quantified)됩니다.

### 유도 관계형 스키마와 표준 변환기

동치성 추론은 주어진 Cypher 질의를 **유도 관계형 스키마(induced relational schema)** 위의 SQL 질의로 먼저 트랜스파일하는 것에서 출발합니다. 유도 관계형 스키마는 그래프 데이터베이스를 자연스럽게 관계형으로 표현한 것으로, 두 단계로 얻어집니다. 첫째, 그래프 스키마의 노드 타입과 엣지 타입을 모두 테이블로 변환합니다. 둘째, 노드와 엣지 사이의 결합(incidence) 및 인접(adjacency) 정보를 함수적 종속성(functional dependency)으로 변환합니다. 예컨대 `CONCEPT` 노드는 `Concept` 테이블로, `CS` 엣지는 `Cs` 테이블로 사상됩니다.

함수적 종속성을 다루기 위해서는 테이블에 외래 키를 도입해야 합니다. 앞서 본 일반 관계형 스키마와 비교하면, 유도 관계형 스키마는 기존 속성을 그대로 보존하면서 `SRC`(출발)와 `TGT`(도착)라는 추가 속성을 외래 키로 사용해 함수적 종속성을 표현한다는 점이 다릅니다. 가령 `CS` 타입 엣지의 출발 노드는 `CONCEPT` 타입이므로, 유도 스키마의 `Cs'` 테이블에는 `Concept` 테이블의 `CID`를 가리키는 외래 키 `SRC`가 생기고, 도착 노드 쪽으로는 `Pa` 테이블을 가리키는 외래 키 `TGT`가 생깁니다. 이렇게 원래 그래프 스키마로부터 자동으로 구성되는 변환기를 **표준 데이터베이스 변환기** $\Phi\_{\mathsf{sdt}}$라 부르며, 이는 주어진 그래프 스키마의 임의의 인스턴스를 그 유도 스키마 위의 관계형 데이터베이스로 변환할 수 있게 해줍니다.

### 구문 지향적 트랜스파일레이션의 실제

표준 변환기 $\Phi\_{\mathsf{sdt}}$는 그래프 데이터베이스의 원소와 유도 관계형 데이터베이스의 항목 사이에 일대일 대응을 확립합니다. 이 대응이 "원자적 원소" 수준에서 사상을 고정해 주기 때문에, 노드와 엣지에 대한 Cypher의 원자 질의를 SQL 테이블에 대한 원자 질의로 번역할 수 있으며 이것이 귀납적 트랜스파일레이션의 기저 사례(base case)가 됩니다. 핵심 통찰은 앞서 강조한 대로 Cypher 경로 질의가 관계형 조인에 대응한다는 점입니다. 예를 들어 `MATCH (u)-[:CS]->(v)` 같은 경로 질의는, 변환기가 `CS` 엣지 타입을 `Cs` 테이블로 사상한다고 명시하므로 `Concept`, `Cs`, `Pa` 테이블 사이의 SQL 조인 질의로 번역됩니다.

개념은 단순하지만 실제 트랜스파일 결과는 상당한 세심함을 요구합니다. 앞서 본 Cypher 질의의 트랜스파일 결과는 `WITH` 절을 활용한 다단계 SQL로 표현됩니다.

```sql
WITH T1 AS (
  SELECT c1.CID AS c1_CID, ..., s.SID AS s_SID
  FROM Concept AS c1 JOIN CS AS r1 JOIN PA AS p1
       JOIN SP AS r2 JOIN Sentence AS s
  ON c1.CID = 1 AND c1.CID = r1.SRC AND ... AND r2.TGT = s.SID ),
T2 AS ( SELECT s_SID FROM T1 ),
T3 AS (
  SELECT s.SID AS s_SID, ..., c2.CID AS c2_CID
  FROM Sentence AS s JOIN SP AS r3 JOIN PA AS p2
       JOIN CS AS r4 JOIN Concept AS c2
  ON s.SID = r3.TGT AND ... AND r4.SRC = c2.CID ),
T4 AS ( SELECT * FROM T2 JOIN T3 ON T2.s_SID = T3.s_SID )
SELECT T4.c2_CID, Count(*) FROM T4 GROUP BY T4.c2_CID
```

여기서 번역의 대응 관계를 따라가 보면, 첫 번째 패턴 매칭이 `T1`로, `WITH` 절이 중간 결과를 `T2`로 전파하는 부분이, 두 번째 패턴 매칭이 `T3`로 번역됩니다. 두 경로 패턴이 공유하는 노드 `s:SENTENCE` 때문에 `T2`와 `T3`를 조인하여 `T4`를 만들고, 마지막 `RETURN` 절의 집계 표현식이 `GROUP BY`로 번역됩니다. 이처럼 Cypher의 다양한 경로 패턴 유형과, 경로 질의가 집계 같은 다른 Cypher 연산자와 합성되는 경우를 정확히 다루어야 하므로, 개념적 단순함과 별개로 구현에는 상당한 주의가 필요합니다.

### 검증을 위한 마지막 조각: 잔여 변환기

구성에 의한 올바름이 보장되는 트랜스파일이 가능하더라도 이것만으로는 충분하지 않은 세 가지 이유가 있습니다. 첫째, 트랜스파일 결과가 가장 효율적이라는 보장이 없어 사용자가 직접 손으로 최적화한 SQL을 쓰고 싶을 수 있습니다. 둘째, 사용자가 기본 유도 스키마가 아닌 다른 관계형 스키마를 사용하고 싶을 수 있습니다. 셋째, 트랜스파일 규칙은 Cypher에서 SQL로의 방향만 다루고 그 반대 방향은 다루지 않습니다. 이러한 한계 때문에, 임의의 Cypher–SQL 질의 쌍에 대한 검증을 수행하려면 추가 장치가 필요합니다.

이때 핵심 아이디어는 **잔여 데이터베이스 변환기(residual database transformer)**를 추론하는 것입니다. 잔여 변환기는 유도 스키마 위의 관계형 데이터베이스와, 사용자가 제공한 변환기로 명세된 목표 관계형 데이터베이스 사이의 관계를 기술합니다. 이 예제의 경우 잔여 변환기는 유도 스키마의 인스턴스(`Concept'`, `Cs'`, `Pa'`, `Sp'`, `Sentence'`)를 원하는 목표 스키마의 인스턴스로 변환하는 규칙으로 표현되며, 형태상 앞서 본 표준 변환기 규칙에서 그래프 레이블을 유도 스키마의 테이블 이름으로 치환한 것과 유사합니다. 이렇게 잔여 스키마가 주어지면, 이제 문제는 완전히 SQL 영역 안의 동치성 검사로 환원됩니다. 따라서 [VeriEQL](https://doi.org/10.1145/3649849) 같은 기존 SQL 동치성 검사기를 그대로 활용하여 두 질의의 동치성을 반박하고, 앞서 본 `Count(*)` 값이 2와 4로 갈리는 반례 인스턴스를 자동으로 얻을 수 있습니다. VeriEQL은 SMT 기반으로 심볼릭 튜플 위에서 질의 의미론을 인코딩하고, 동치성이 깨질 때 구체적인 데이터베이스 인스턴스를 반례로 생성하는 능력을 갖추고 있어, 이처럼 미묘한 번역 버그를 드러내는 데 특히 잘 들어맞습니다.

결국 이 동기 예제는 서로 다른 두 데이터 모델 위의 질의 동치성이라는 다루기 어려운 이질적(heterogeneous) 문제를, 데이터베이스 변환기와 트랜스파일레이션, 그리고 잔여 변환기를 거쳐 잘 연구된 동질적(homogeneous) SQL 동치성 문제로 환원하는 전체 전략을 한 편의 작은 이야기처럼 압축해 보여줍니다.
# 형식적 기반 다지기: 그래프 데이터베이스, 관계형 데이터베이스, 그리고 두 질의 언어

앞선 동기 예제가 "겉보기에 같은 두 질의가 사실은 다르다"는 문제를 직관적으로 보여주었다면, 이제 그 문제를 엄밀하게 다루기 위한 형식적 토대를 마련할 차례입니다. 이 절에서는 그래프 데이터베이스와 관계형 데이터베이스를 각각 수학적으로 정의하고, 두 세계의 질의 언어인 Featherweight Cypher와 Featherweight SQL을 형식 문법으로 제시합니다. 이렇게 다져 둔 정의들은 이후 데이터베이스 변환기를 통한 동치성 정의와 트랜스파일레이션 규칙을 기술하기 위한 공통 형식 언어를 수립합니다.

## 속성 그래프의 형식적 정의

그래프 데이터베이스 인스턴스는 노드와 엣지가 데이터를 담고 있는 **속성 그래프(property graph)** 입니다. 일반적으로 노드는 개체(entity)를, 엣지는 관계(relationship)를 모델링하며, 각 노드와 엣지는 속성 키(property key)와 값(value)의 쌍으로 표현된 데이터를 저장합니다. 또한 각 노드·엣지에는 그것이 어떤 종류의 개체나 관계인지를 나타내는 **레이블(label)** 이 부여됩니다. 잘 정의된 속성 그래프는 그래프 스키마를 따라야 하는데, 논문은 이를 다음과 같이 형식화합니다.

먼저 타입(type)의 개념이 도입됩니다. 노드 타입 $t\_{\mathsf{node}}$는 튜플 $(l, K\_1, \ldots, K\_n)$으로, $l$은 노드의 레이블(예: $\mathsf{Actor}$)이고 $K\_1, \ldots, K\_n$은 그 노드 타입이 가지는 속성 키들(예: $\mathtt{name}$, $\mathtt{dob}$)입니다. 엣지 타입 $t\_{\mathsf{edge}}$ 역시 튜플 $(l, t\_{\mathsf{src}}, t\_{\mathsf{tgt}}, K\_1, \ldots, K\_m)$으로 정의되는데, 여기서 $l$은 레이블(예: $\mathsf{ACTS\_IN}$), $t\_{\mathsf{src}}$와 $t\_{\mathsf{tgt}}$는 각각 출발 노드와 도착 노드의 타입, 그리고 $K\_1, \ldots, K\_m$은 그 엣지 타입의 속성 키(예: $\mathtt{role}$)입니다.

이 타입들로부터 정보를 꺼내는 보조 함수들이 정의됩니다. 노드 타입 $t\_{\mathsf{node}} = (l, K\_1, \ldots, K\_n)$에 대해 $\mathsf{label}(t\_{\mathsf{node}})$는 레이블 $l$을 돌려주고, $\mathsf{keys}(t\_{\mathsf{node}}) = \{K\_1, \ldots, K\_n\}$은 그 노드 타입과 연관된 속성 키들의 집합을 돌려줍니다. 여기서 한 가지 미묘한 점에 주목할 필요가 있습니다. 노드 타입 자체는 순서가 있는 튜플로 정의되며 $K\_1$이 **기본 속성 키(default property key)** —관계형 데이터베이스의 기본 키(primary key)처럼 전역적으로 유일한 값을 갖는 키—로 약정됩니다. 그런데 $\mathsf{keys}$ 함수는 순서가 없는 집합을 반환하므로, "$K\_1$이 기본 키"라는 정보는 타입의 튜플 표현에서만 추출할 수 있고 $\mathsf{keys}$의 결과만 보아서는 잃어버린다는 점입니다. 엣지 타입 $t\_{\mathsf{edge}} = (l', t\_{\mathsf{src}}, t\_{\mathsf{tgt}}, K'\_1, \ldots, K'\_m)$에 대해서도 $\mathsf{label}(t\_{\mathsf{edge}}) = l'$와 $\mathsf{keys}(t\_{\mathsf{edge}}) = \{K'\_1, \ldots, K'\_m\}$이 같은 방식으로 정의되며, $K'\_1$이 기본 속성 키가 됩니다.

엣지 타입에는 출발·도착 노드 타입을 꺼내는 접근자도 정의됩니다. 다만 그 정의가 직관에 반하는 것처럼 보일 수 있습니다. 논문은 $\mathsf{dstType}(t\_{\mathsf{edge}}) = t\_{\mathsf{src}}$와 $\mathsf{srcType}(t\_{\mathsf{edge}}) = t\_{\mathsf{tgt}}$로 정의하는데, 이름과 정의가 서로 뒤바뀐 것처럼 보입니다. 이는 논문이 채택한 표기법을 그대로 옮긴 것으로, 이 접근자들은 이후 트랜스파일레이션 규칙에서 엣지 타입으로부터 노드 타입 방향으로의 역방향 조회를 지원하기 위해 이런 방식으로 정의된 것으로 이해할 수 있습니다.

이러한 타입들을 모으면 스키마가 됩니다. 그래프 데이터베이스 스키마 $\Psi\_G$는 쌍 $(T\_N, T\_E)$로, $T\_N$은 노드 타입의 집합이고 $T\_E$는 엣지 타입의 집합입니다. 논문은 두 가지 단순화 가정을 둡니다. 첫째, 각 노드·엣지 타입의 레이블이 그 타입을 유일하게 결정합니다. 즉 $\forall t\_1, t\_2 \in T\_N \cup T\_E.\ \mathsf{label}(t\_1) \neq \mathsf{label}(t\_2)$이며, 따라서 타입과 레이블을 서로 바꿔 써도 무방합니다. 둘째, 한 스키마 안의 모든 속성 키는 유일하여 서로 다른 타입 사이에서 이름 충돌이 없습니다.

마지막으로 스키마의 실제 인스턴스가 정의됩니다. 스키마 $\Psi\_G = (T\_N, T\_E)$의 인스턴스는 튜플 $G = (N, E, P, T)$로, $N$은 노드의 집합, $E \subseteq N \times N$은 엣지의 집합, $P : (N \cup E) \times \textbf{Keys} \to \textbf{Values}$는 노드나 엣지의 속성 키에 값을 대응시키는 함수, 그리고 $T : N \cup E \to T\_N \cup T\_E$는 각 노드 $n$이나 엣지 $e$에 그 타입을 부여하는 함수입니다. 표기상 $P(n, k)$는 노드 $n$의 속성 키 $k$의 값을, $P(e, k)$는 엣지 $e$에 대해 같은 의미를 갖습니다. 또한 $G \triangleright \Psi\_G$는 $G$가 스키마 $\Psi\_G$의 인스턴스임을 나타내며, $G$의 임의의 부분 그래프를 속성 그래프라 부릅니다.

이처럼 추상적인 네 쌍 정의는 코드로 옮겨 보면 한층 또렷해집니다. 다음은 정의 $G = (N, E, P, T)$가 실제 데이터에서 어떤 모습인지 보여 주는 의사 코드입니다.

```python
# 보충 예시: Definition 3.3의 G = (N, E, P, T)를 구체화
N = {"a1", "m1"}                        # 노드 집합 N
E = {("a1", "m1")}                      # E ⊆ N × N (하나의 엣지)
P = {("a1", "name"): "Cate Blanchett",  # P: (N∪E) × Keys → Values
     ("m1", "title"): "Blue Jasmine",
     (("a1","m1"), "role"): "Jasmine"}  # 엣지의 속성도 P로 표현
T = {"a1": t_Actor, "m1": t_Movie,      # T: N∪E → T_N∪T_E
     ("a1","m1"): t_ACTS_IN}            # 각 원소에 타입을 부여
```

여기서 노드 `a1`(배우)과 `m1`(영화)이 하나의 `ACTS_IN` 엣지로 연결되고, 노드와 엣지가 각각 속성을 갖는 모습이 함수 $P$와 $T$로 어떻게 부호화되는지 직관적으로 드러납니다.

## 그래프 질의 언어: Featherweight Cypher

이 방법론을 형식화하기 위해 본 논문은 널리 사용되는 그래프 질의 언어 Cypher의 한 부분 집합에 집중하며, 이를 **Featherweight Cypher** 라 부릅니다. 이 언어의 형식 문법은 다음과 같은 비단말 기호들로 구성됩니다. 질의 $Q$는 반환 질의 $R$의 합집합이거나 그러한 반환 질의들에 정렬을 적용한 형태로, $\mathsf{OrderBy}(R, k, b)$, $\mathsf{Union}(Q, Q)$, $\mathsf{UnionAll}(Q, Q)$를 포함합니다. 반환 질의 $R$은 $\mathsf{Return}(C, \overline{E}, \overline{k})$의 형태로, 절(clause) $C$, 표현식의 리스트 $\overline{E}$, 속성 키 이름의 리스트 $\overline{k}$를 입력으로 받아 그래프들의 리스트를 테이블로 빚어냅니다.

절 $C$는 입력 속성 그래프에 대한 패턴 매칭을 나타내는데, 단일 매칭 $\mathsf{Match}(PP, \phi)$, 이전 절에 이어지는 매칭 $\mathsf{Match}(C, PP, \phi)$, 선택적 매칭 $\mathsf{OptMatch}(C, PP, \phi)$, 그리고 $\mathsf{With}(C, \overline{X}, \overline{X})$로 구성됩니다. 여기서 $\mathsf{With}$ 절은 중간 결과를 다음 단계로 전달하는 동시에, Cypher에서 변수의 스코프(scope)를 재설정하여 이후 $\mathsf{Match}$가 새로운 패턴을 시작할 수 있게 하는 역할을 합니다. 이 스코프 재설정 의미는 이후 SQL의 `WITH`(공통 테이블 식, CTE) 절로 자연스럽게 번역됩니다. 패턴은 경로 패턴 $PP$로 명시되는데, 노드 패턴 하나($NP$)이거나 노드 패턴·엣지 패턴·경로 패턴이 이어진 형태($NP, EP, PP$)입니다. 노드 패턴 $NP$는 $(X, l)$, 즉 변수 이름과 레이블의 쌍이고, 엣지 패턴 $EP$는 $(X, l, d)$로 방향 $d \in \{\rightarrow, \leftarrow, \leftrightarrow\}$를 추가로 갖습니다.

표현식 $E$는 속성 키 $k$, 값 $v$, 술어를 값으로 변환하는 $\mathsf{Cast}(\phi)$, 집계 $\mathsf{Agg}(E)$, 그리고 산술 연산 $E \oplus E$로 구성되며, 집계 연산자 $\mathsf{Agg}$는 $\{\mathsf{Count}, \mathsf{Avg}, \mathsf{Sum}, \mathsf{Min}, \mathsf{Max}\}$ 중 하나입니다. 술어 $\phi$는 참 $\top$, 거짓 $\bot$, 비교 $E \odot E$, $\mathsf{IsNull}(E)$, 멤버십 $E \in \overline{v}$, 존재 검사 $\mathsf{Exists}(PP)$, 그리고 논리 결합 $\phi \land \phi$, $\phi \lor \phi$, $\neg\phi$로 이루어집니다. 특히 $\mathsf{Exists}(PP)$ 술어는 어떤 경로 패턴이 존재하는지를 검사하는데, 이후 부록에서 이 술어가 투영된 서브쿼리에 대한 SQL의 `IN` 술어로 번역됨을 보게 됩니다. 이는 앞선 동기 예제에서 중첩 $\mathsf{Match}$ 대신 $\mathsf{Exists}$를 쓰면 이중 집계 버그가 사라진다는 사실과 직접 맞닿아 있습니다. 여기서 $\oplus$와 $\odot$은 각각 산술 연산자와 논리 연산자에 대응하며, 이 부분 집합의 표시적 의미론(denotational semantics)은 부록에서 형식적으로 기술됩니다. 다만 Cypher의 무제한 길이 경로 질의나 `shortestPath` 같은 그래프 도달 가능성(reachability) 원시 연산은 이 논문이 다루는 핵심 SQL 조각으로 표현할 수 없으므로 제외됩니다.

이 표시적 의미론에서 한 가지 주목할 점은, 술어 $\phi$가 단순한 이치 논리가 아니라 삼치 논리(true/false/null)로 평가된다는 것입니다. 이렇게 세 번째 값 `null`이 필요한 이유는 $\mathsf{OptMatch}$ 때문입니다. 선택적 매칭이 실패하면 바인딩되지 못한 변수가 `null`로 처리되어야 하는데, 이는 정확히 SQL의 좌외부 조인(left outer join) 의미론에 대응합니다. 즉 삼치 논리는 그래프 질의의 선택적 매칭과 SQL의 외부 조인을 의미적으로 정렬하기 위한 장치인 셈입니다.

추상적인 문법을 구체적인 Cypher 질의와 연결해 보면 이해가 한결 쉬워집니다. 다음 스니펫은 위 문법의 각 구성 요소가 실제 질의에서 어디에 해당하는지 주석으로 표시한 예입니다.

```cypher
-- Featherweight Cypher: Q = Return(Match(PP, φ), Ē, k̄)
MATCH (a:Actor)-[e:ACTS_IN]->(m:Movie)   -- PP: NP, EP, NP
WHERE a.name = 'Cate Blanchett'          -- φ: E ⊙ E (비교 술어)
RETURN m.title AS title, e.role AS role  -- Ē(표현식), k̄(반환 키 이름)
```

논문은 이 문법이 실제 Cypher 질의를 어떻게 포착하는지를 구체적 예제로 보여 줍니다. 다음 Cypher 질의를 생각해 봅시다.

```cypher
MATCH (n:EMP)-[:WORK_AT]->(m:DEPT)
RETURN m.dname AS name, Count(n) AS num
```

이 질의는 부서 이름과 그 부서에 속한 직원 수를 담은 테이블을 반환합니다. Featherweight Cypher 구문으로는 다음과 같이 표현됩니다.

$$
\mathsf{Return}(\mathsf{Match}([(\texttt{n},\texttt{EMP}),(\texttt{e},\texttt{WORK\_AT},\rightarrow),(\texttt{m},\texttt{DEPT})],\top),[\texttt{m.dname},\mathsf{Count}(\texttt{n.id})],[\texttt{name},\texttt{num}])
$$

여기서 매칭 절은 `WORK_AT` 타입의 엣지로 연결된 모든 길이 1짜리 경로를 `EMP` 노드에서 `DEPT` 노드로 향하는 방향으로 찾아냅니다. 그런 다음 반환 절은 매칭된 경로들의 집합을 `name`과 `num`이라는 두 열을 가진 테이블로 다시 빚어냅니다. `name` 열은 `DEPT` 노드의 속성 키 `dname`의 값으로 채워지고, `num` 열은 `n.id`의 개수로 채워집니다. 패턴 내 위치를 기준으로 보면 `n`은 `EMP` 타입의 출발 노드이고 `m`은 `DEPT` 타입의 도착 노드에 해당합니다.

## 관계형 데이터베이스와 SQL

이제 대응되는 관계형 세계를 형식화합니다. 관계형 데이터베이스 스키마는 쌍 $\Psi\_R := (S, \xi)$로, $S : \mathcal{R} \to [\mathcal{A}]$는 관계 이름의 집합 $\mathcal{R}$을 속성의 리스트로 대응시키는 사상이고, $\xi$는 **무결성 제약(integrity constraint)** 입니다. 모든 속성 이름은 스키마 안에서 유일하다고 가정합니다. 무결성 제약은 세 종류의 원자적 제약의 논리곱으로 표현됩니다. 첫째, 기본 키 제약 $\mathsf{PK}(R) = a$는 속성 $a$가 관계 $R$의 기본 키임을, 즉 $a$에서 값이 일치하는 두 튜플이 존재할 수 없음을 명시합니다. 둘째, 외래 키 제약 $\mathsf{FK}(R.a) = R'.a'$는 관계 $R$의 속성 $a$에 저장된 값들이 관계 $R'$의 속성 $a'$에 저장된 값들의 부분 집합이어야 함을 명시합니다. 셋째, 널 불허 제약 $\mathsf{NotNull}(R, a)$는 관계 $R$의 속성 $a$의 값이 $\mathsf{Null}$이어서는 안 됨을 명시합니다. 이 세 종류의 제약은 우연히 선택된 것이 아니라, 이후 그래프 스키마를 관계형 스키마로 변환하는 스키마 추론 절차에서 모두 자동으로 생성되는 핵심 재료가 됩니다. 앞선 동기 예제에서 엣지가 외래 키를 통해 표현되던 장면이 바로 이 형식적 정의에 근거를 둡니다.

관계형 데이터베이스 인스턴스 $R$은 튜플의 모임 $\{r\_1, \ldots, r\_m\}$으로, 각 $r\_i$는 $(a\_1 : v\_1, \ldots, a\_n : v\_n)$의 형태입니다. 여기서 $a\_1, \ldots, a\_n$은 속성이고 $v\_1, \ldots, v\_n$은 값입니다. $\mathsf{Attrs}(r\_i)$는 속성 리스트 $a\_1, \ldots, a\_n$을 순서대로 돌려주고, $r\_i.a$는 튜플 $r\_i$의 속성 $a$에 저장된 값을 나타냅니다. 그래프와 마찬가지로 $R \triangleright \Psi\_R$는 $R$이 $\Psi\_R$의 인스턴스임을 나타냅니다.

앞서 본 속성 그래프 의사 코드와의 일관성을 위해, 같은 직원-부서 예제를 관계형 인스턴스로 표현하면 다음과 같습니다.

```python
# 보충 예시: Definition 3.5/3.6의 관계형 스키마와 인스턴스 R ▷ ΨR
# ΨR = (S, ξ): S = {EMP: [id, name, dept_id], DEPT: [id, dname]}
#              ξ = PK(EMP)=id ∧ PK(DEPT)=id ∧ FK(EMP.dept_id)=DEPT.id
R_EMP  = {(id:1, name:"Alice", dept_id:10),
          (id:2, name:"Bob",   dept_id:10)}
R_DEPT = {(id:10, dname:"Engineering")}

# r_i.a 표기:  R_EMP[0].dept_id = 10
# Attrs(r_i):  R_EMP[0]의 속성 리스트 = [id, name, dept_id]
```

여기서 외래 키 $\mathsf{FK}(\texttt{EMP.dept\_id}) = \texttt{DEPT.id}$가 그래프에서 직원이 부서에 속한다는 관계를 어떻게 대신 표현하는지가 분명히 드러납니다. 그래프에서 엣지로 표현되던 연결이 관계형에서는 외래 키 참조로 부호화되는 것입니다.

질의 언어로는 SQL을 사용합니다. Featherweight SQL 문법은 핵심 관계 대수—투영 $\Pi$, 선택 $\sigma$, 이름 변경 $\rho$, 조인 $\otimes$, 집합 합집합 $\cup$, 백(bag) 합집합 $\uplus$—를 확장하여 `GroupBy`, `OrderBy`, `With` 절을 포함합니다. 구체적으로 질의 $Q$는 관계 $R$, 투영 $\Pi\_L(Q)$, 선택 $\sigma\_\phi(Q)$, 이름 변경 $\rho\_R(Q)$, $Q \cup Q$, $Q \uplus Q$, $Q \otimes Q$, 그리고 $\texttt{GroupBy}(Q, \overline{E}, L, \phi)$, $\texttt{With}(\overline{Q}, \overline{R}, Q)$, $\texttt{OrderBy}(Q, a, b)$로 구성됩니다. 속성 리스트 $L$은 표현식 $E$, 이름 변경 $\rho\_a(E)$, 혹은 그들의 나열 $L, L$이며, 속성 표현식 $E$는 속성 $a$, 값 $v$, $\texttt{Cast}(\phi)$, $\texttt{Agg}(E)$, 산술 $E \oplus E$입니다. 술어 $\phi$는 불리언 $b$, 비교 $E \odot E$, $\texttt{IsNull}(E)$, 멤버십 $E \in \overline{v}$, 서브쿼리 멤버십 $\overline{E} \in Q$, 그리고 논리 결합으로 구성됩니다.

특히 주목할 부분은 조인 연산자 $\otimes$입니다. 이는 카티전 곱 $\times$, 내부 조인(inner join) $\bowtie\_\phi$, 그리고 세 가지 외부 조인—좌외부 조인 $⟕\_\phi$, 우외부 조인 $⟖\_\phi$, 완전 외부 조인 $⟗\_\phi$—을 포함합니다. 외부 조인이 명시적으로 포함된 것은 우연이 아닙니다. 앞서 Featherweight Cypher에서 $\mathsf{Match}$가 내부 조인에 대응하는 반면 $\mathsf{OptMatch}$는 선택적 매칭의 실패 시 `null`을 만들어 내야 하므로 좌외부 조인에 대응하기 때문입니다. 다시 말해 이 조인 연산자들의 집합은 Cypher의 두 가지 매칭 의미론을 SQL 쪽에서 정확히 받아낼 수 있도록 신중히 선택된 것이며, 앞서 언급한 삼치 논리와 함께 두 언어의 의미적 정렬을 떠받칩니다. 이 SQL 연산자들의 의미론은 표준적이며, 관계형 질의 동치성 검증을 다룬 He et al.의 선행 연구를 비롯한 기존 연구에서 형식적으로 정의된 바 있습니다. 또한 Featherweight Cypher의 기반이 되는 Cypher 언어 자체는 Francis et al.이 제안한 사양을 따릅니다.

이로써 그래프와 관계형 양쪽의 데이터 모델과 질의 언어가 모두 형식적으로 갖추어졌습니다. 한쪽에는 노드·엣지·속성·타입으로 이루어진 속성 그래프와 경로 패턴 중심의 Featherweight Cypher가, 다른 한쪽에는 관계·속성·무결성 제약으로 이루어진 관계형 인스턴스와 관계 대수 기반의 Featherweight SQL이 놓여 있습니다. 이 두 형식 체계가 마련되었으므로, 다음 단계에서는 이들을 잇는 데이터베이스 변환기를 1차 논리 형식으로 정의하고 변환기를 기준으로 한 질의 동치성 문제를 엄밀하게 진술할 수 있게 됩니다.
# 문제 정의: 데이터베이스 변환기의 언어와 동치성 검사 문제

앞서 그래프와 관계형 양쪽의 데이터 모델과 질의 언어를 형식적으로 정의했으므로, 이제 두 세계를 연결하는 매개체인 데이터베이스 변환기를 1차 논리(first-order logic) 형식으로 엄밀히 정의하고, 이를 기준으로 한 질의 동치성 문제를 정밀하게 진술할 차례입니다. 이 절은 동기 예제에서 직관적으로만 소개되었던 데이터베이스 변환기를 비로소 수학적으로 못 박는 부분입니다.

## 데이터베이스 변환기를 위한 도메인 특화 언어

논문은 데이터베이스 변환기를 표현하기 위한 작은 도메인 특화 언어(DSL)를 도입합니다. 이 DSL은 Wang et al. (2020)의 선행 연구를 따르며, 관계형 데이터베이스를 위한 표준적인 스키마 매핑(schema mapping) 개념, 특히 Miller et al. (2000) 및 Fagin et al. (2009)의 스키마 매핑 및 데이터 교환 연구를 한층 유연한 형태로 일반화한 것입니다. 이 DSL의 문법은 다음과 같이 정의됩니다.

$$
\begin{array}{l l c l}
\text{Transformer}&\Phi&::=&P,\ldots,P\to P~|~\Phi~\Phi\\
\text{Predicate}&P&::=&E(t,\ldots,t)\\
\text{Term}&t&::=&c~|~v~|~\_\\
\end{array}
$$

여기서 기호 $E$는 테이블 이름, 노드 레이블, 또는 엣지 레이블 중 하나에 대응되고($E \in \textbf{Table Names} \cup \textbf{Node Labels} \cup \textbf{Edge Labels}$), $c$는 상수(constant), $v$는 변수(variable)입니다.

이 문법이 말하는 바를 풀어 보면, 하나의 데이터베이스 변환기 $\Phi$는 다음과 같은 형태의 1차 논리 식들의 집합으로 표현됩니다.

$$
P\_1, \ldots, P\_n \to P\_0
$$

각 $P\_i$는 데이터베이스 요소—관계형 데이터베이스의 테이블이나 그래프 데이터베이스의 노드 또는 엣지—를 나타내는 술어(predicate)이며, $E(t\_1, \ldots, t\_n)$의 형태를 띱니다. 여기서 $E$는 테이블 이름이나 노드·엣지 레이블에 대응하고, 각 $t\_j$는 항(term), 즉 변수나 상수입니다. 특별히 기호 $\_$는 **사용되지 않는 새로운(fresh) 변수**를 나타내는 와일드카드입니다. 모든 변수는 암묵적으로 전칭 한정(universally quantified)되지만 간결함을 위해 한정 기호는 문법에서 생략됩니다.

이 와일드카드 $\_$의 의미를 구체적인 미니 예제로 짚어 보면 이해가 빨라집니다. 가령 `Person(x, _) → Employee(x)`라는 규칙을 생각해 보면, `Person` 술어의 두 번째 속성(예컨대 나이)은 $\_$로 무시되고 오직 첫 번째 속성 `x`(예컨대 이름)만이 출력 술어 `Employee(x)`에 참여합니다. 즉 $\_$는 "이 위치에는 무엇이 와도 상관없으며 출력에서 쓰이지 않는다"는 표현입니다.

직관적으로 식 $P\_1, \ldots, P\_n \to P\_0$은 "만약 어떤 데이터베이스 인스턴스 $D$ 위에서 술어 $P\_1, \ldots, P\_n$이 모두 성립한다면, 또 다른 데이터베이스 인스턴스 $D'$ 위에서 술어 $P\_0$이 성립한다"는 함의를 표현합니다. 바로 이 함의 구조가 한 데이터 모델의 데이터가 어떻게 다른 모델의 데이터로 사상되는지를 규칙으로 기술하는 핵심 장치입니다.

## Herbrand 의미론으로 풀어내는 변환기의 의미

이 DSL의 의미론을 본격적으로 정의하기에 앞서, 그 토대가 되는 **Herbrand 의미론(Herbrand semantics)** 의 직관을 먼저 잡아 두면 이후 형식적 정의가 한결 자연스럽게 다가옵니다. 일반적인 1차 논리에서 식의 진리값은 추상적인 임의의 해석 영역(domain of interpretation) 위에서 평가됩니다. 그러나 Herbrand 의미론에서는 해석 영역을 문법에 등장하는 상수와 함수 기호로부터 구성되는 구체적인 구문 객체들로 고정하고, 진리값을 **그라운드 술어(ground predicate)의 집합**이 식을 충족하는지 여부로 따집니다. 데이터베이스 맥락에서 이는 매우 자연스러운데, 데이터베이스 인스턴스 자체가 곧 구체적인 사실(fact)들의 모임이기 때문입니다. 즉 각 데이터베이스 인스턴스를 그라운드 술어들의 집합으로 표현한 뒤, 그 집합이 변환기에 해당하는 1차 논리 식을 충족하는지를 검사하는 것이 의미론의 핵심 아이디어입니다.

이를 정밀하게 만들기 위해 논문은 데이터베이스 인스턴스를 그것의 구조와 내용을 나타내는 그라운드 술어 집합으로 사상하는 함수 $\mathcal{C}$를 도입합니다.

$$
\mathcal{C}(D) = \{E(t\_1, \ldots, t\_n) \mid E \in D\}
$$

이 사상이 구체적으로 어떻게 동작하는지는 $D$가 관계형 데이터베이스인지 그래프 데이터베이스인지에 따라 달라집니다.

먼저 **관계형 데이터베이스 인스턴스** $D$의 경우, $D$ 안의 테이블 $R$이 레코드 집합 $\{(a\_1, \ldots, a\_n)\}$을 가진다면, $R$은 다음과 같은 그라운드 술어 집합으로 변환됩니다.

$$
\{R(a\_1, \ldots, a\_n) \mid (a\_1, \ldots, a\_n) \in R\}
$$

여기서 $R$은 테이블 이름이 되고, 각 $a\_i$는 레코드 안의 값에 해당하는 상수입니다. 즉 테이블의 각 행이 그대로 하나의 그라운드 사실이 됩니다.

다음으로 **그래프 데이터베이스 인스턴스** $D$의 경우는 노드와 엣지를 각각 처리합니다. 레이블 $l$과 속성 키 $K\_1, \ldots, K\_n$에 대한 값 $a\_1, \ldots, a\_n$을 가진 노드를 $N(l, a\_1, \ldots, a\_n)$으로 표기할 때, 노드들은 다음 그라운드 사실 집합으로 변환됩니다.

$$
\{l(a\_1, \ldots, a\_n) \mid \text{node } N(l, a\_1, \ldots, a\_n) \in D\}
$$

즉 노드의 레이블 $l$이 술어 이름이 되고 속성 값들이 인자가 됩니다. 엣지의 경우도 유사한데, 레이블 $l$을 가지고 노드 $s$와 $t$를 잇는 엣지가 속성 값 $a\_1, \ldots, a\_n$을 가질 때 이를 $E(l, s, t, a\_1, \ldots, a\_n)$으로 표기하면, 엣지들은 다음과 같이 변환됩니다.

$$
\{l(a\_1, \ldots, a\_n, s, t) \mid \text{edge } E(l, s, t, a\_1, \ldots, a\_n) \in D\}
$$

여기서 주목할 점은 엣지를 술어로 변환할 때 출발 노드 $s$와 도착 노드 $t$가 인자 목록의 끝에 덧붙여진다는 것입니다. 이는 동기 예제에서 엣지가 관계형 모델로 옮겨질 때 외래 키를 통해 출발·도착 노드를 가리키던 장면과 정확히 맞닿아 있습니다. 즉 그래프의 연결 정보가 술어의 인자로 보존되어, 관계형 측의 외래 키 참조와 의미론적으로 정렬될 수 있게 됩니다.

이 추상적인 $\mathcal{C}$ 함수의 동작을 동기 예제의 데이터에 적용해 구체화하면 다음과 같이 이해할 수 있습니다.

```
-- 그래프 측: CONCEPT 노드와 CS 엣지가 주어졌을 때
-- 노드 CONCEPT(cid=1, name="Aspirin")  →  그라운드 사실
CONCEPT(1, "Aspirin")
-- 엣지 CS(label, s, t, csid)           →  s, t가 인자 끝에 덧붙음
CS(csid, 1, pid)        -- 출발 노드(1)와 도착 노드(pid)를 인자로 보존

-- 변환기 규칙(동기 예제의 Cs 테이블 생성 규칙):
CONCEPT(cid, _), CS(cid, csid, cid, pid), PA(pid, csid) → Cs(cid, csid)
-- 위 규칙이 적용되면 관계형 측에 다음 그라운드 사실이 생성됨
Cs(1, csid)
```

이처럼 $\mathcal{C}$는 그래프 인스턴스든 관계형 인스턴스든 모두 동일한 형식의 그라운드 술어 집합으로 환원해 주며, 변환기 규칙은 이 두 집합 사이의 논리적 관계를 기술하게 됩니다.

이제 변환기 $\Phi$의 의미론을 정의할 수 있습니다. 변환기 $\Phi$를 전칭 한정된 1차 논리 식들의 집합으로 표현한 것을 $\llbracket\Phi\rrbracket$이라 표기할 때, 의미론은 다음과 같이 주어집니다.

$$
\Phi(D)=D'\quad\Leftrightarrow\quad\mathcal{C}(D)\cup\mathcal{C}(D')\models\llbracket\Phi\rrbracket
$$

여기서 $S \models \varphi$는 그라운드 술어 집합 $S$가 식 $\varphi$의 Herbrand 모델임을 의미합니다. 풀어 말하면, 변환기 $\Phi$가 인스턴스 $D$를 $D'$으로 변환한다는 것은, $D$와 $D'$을 각각 그라운드 술어 집합으로 바꾼 뒤 그 합집합이 $\llbracket\Phi\rrbracket$을 충족한다는 것과 동치입니다. 동기 예제로 돌아가면, 그래프 인스턴스 $G$와 관계형 인스턴스 $R$에 대해 해당 변환기 $\Phi$가 $\Phi(G) = R$을 만족한다는 사실(Example 4.1)이 바로 이 정의에 의해 형식적으로 뒷받침됩니다.

## 동치성 검사 문제의 형식적 정의

변환기의 의미론이 갖추어졌으므로, 이제 변환기 $\Phi$를 기준(modulo)으로 한 그래프 질의와 관계형 질의의 동치성을 정의할 수 있습니다. 이를 위해 몇 가지 정의가 차례로 도입됩니다.

다음 그림은 사용자가 제공한 스키마 변환기 $\Phi$를 매개로 한 Cypher 질의와 SQL 질의 사이의 동치성 정의를 개념적으로 시각화합니다.

![Cypher와 SQL 질의의 동치성 정의](https://arxiv.org/html/2504.03182/x2.png)

이 그림은 그래프 데이터베이스, Cypher 질의, 스키마 변환기 $\Phi$, SQL 질의, 그리고 각 데이터베이스 인스턴스 위에서 질의를 실행해 얻은 결과 테이블 사이의 관계를 보여 줍니다. 핵심은 한 쌍의 Cypher 질의와 SQL 질의가 동치라는 것이, $\Phi$를 충족하는 데이터베이스 인스턴스 쌍 위에서 실행되었을 때 (열 이름의 재명명을 무시하고) 항상 같은 테이블을 산출한다는 의미라는 점입니다.

먼저 **데이터베이스 질의(Definition 4.2)** 가 정의됩니다. 스키마 $\Psi$ 위의 데이터베이스 질의 $Q$는 $D \triangleright \Psi$를 만족하는 인스턴스 $D$를 입력받아 테이블을 산출하며, 그 의미론을 $\llbracket Q\rrbracket\_D$로 표기합니다. 이 정의는 SQL과 Cypher 질의를 모두 포괄할 만큼 일반적인데, 두 질의 언어 모두 결과로 테이블을 반환하기 때문입니다.

다음으로 **데이터베이스 동치성(Definition 4.3)** 이 정의됩니다. 스키마 $\Psi, \Psi'$ 위의 인스턴스 $D, D'$과, $\Psi$의 인스턴스를 $\Psi'$의 인스턴스로 변환하는 변환기 $\Phi$가 주어졌을 때, $\Phi(D) = D'$이면 $D$는 $\Phi$를 기준으로 $D'$과 동치라고 하며 $D \sim\_\Phi D'$으로 표기합니다. 즉 그래프 인스턴스 $G$가 관계형 인스턴스 $R$과 동치라는 것은, $G$에 변환기 $\Phi$를 적용하여 $R$의 내용을 얻을 수 있다는 뜻입니다.

질의의 출력이 같은지를 판단하려면 테이블 사이의 동치성을 정의해야 합니다. **테이블 동치성(Definition 4.4)** 은 두 테이블 $T$와 $T'$ 사이에 열들 간의 전단사 사상(bijective mapping) $\pi$가 존재하여, 중복도(multiplicity) $n$을 가진 각 튜플 $r \in T$에 대해 같은 중복도 $n$을 가진 유일한 튜플 $r' \in T'$이 존재하고 다음 조건을 만족할 때 성립하며, $T \equiv T'$로 표기합니다.

$$
\forall a \in \mathsf{Attrs}(r).~r.a = r'.\pi(a)
$$

이 정의의 핵심은 속성의 순서와 이름을 모두 무시한다는 점인데, 이는 더 견고한 질의 동치성 개념을 가능하게 합니다. 구체적인 미니 예제로 보면, 열 $(\texttt{name}, \texttt{age})$를 가진 테이블 $T$와 열 $(\texttt{이름}, \texttt{나이})$를 가진 테이블 $T'$은, $\pi$가 $\texttt{name} \mapsto \texttt{이름}$, $\texttt{age} \mapsto \texttt{나이}$로 사상하고 튜플 $(\texttt{"Alice"}, 30)$이 양쪽에서 일치하는 한 동치로 간주됩니다. 열 이름이 다르고 순서가 달라도, 적절한 전단사 사상만 존재하면 내용이 같다고 보는 것입니다.

여기서 한 가지 미묘한 예외가 있습니다. 질의에 `OrderBy` 절이 포함된 경우에는 리스트 의미론(list semantics)이 적용되어 결과가 순서 있는 튜플의 리스트가 됩니다. 이 경우 동치성은 더 엄격해지는데, 여전히 속성 간 전단사 사상 $\pi$는 허용되지만(즉 열의 재명명은 가능), **같은 인덱스에 위치한** 튜플 쌍 $r \in T$와 $r' \in T'$에 대해 $\forall a \in \mathsf{Attrs}(r).~r.a = r'.\pi(a)$가 성립해야 합니다. 다시 말해 정렬이 명시되면 행의 순서 자체가 동치성 조건의 일부가 되어 더 이상 행을 임의로 재배치할 수 없으며, $\pi$는 오직 열 방향으로만 작용합니다. 이는 정렬된 결과의 순서가 질의의 의미에 본질적이기 때문에 자연스러운 강화입니다.

이제 모든 조각이 모여 **그래프-관계형 질의 동치성(Definition 4.5)** 이 정의됩니다. 그래프 스키마 $\Psi\_G$와 관계형 스키마 $\Psi\_R$, 그리고 $\Psi\_G$에서 $\Psi\_R$로의 변환기 $\Phi$가 주어졌을 때, $\Psi\_G$ 위의 질의 $Q$가 $\Psi\_R$ 위의 질의 $Q'$과 $\Phi$를 기준으로 동치라는 것($Q \simeq\_\Phi Q'$)은 다음 조건으로 정의됩니다.

$$
\forall G, R.\ (G\triangleright\Psi\_G\land R\triangleright\Psi\_R\land G\sim\_\Phi R)\ \Rightarrow\ \llbracket Q\rrbracket\_G\equiv\llbracket Q'\rrbracket\_R
$$

즉 $G \sim\_\Phi R$을 만족하는 모든 인스턴스 쌍 $G, R$에 대해, $Q$를 $G$에서 실행한 결과와 $Q'$을 $R$에서 실행한 결과가 (열의 재명명·재배치를 무시하고) 같은 테이블을 산출하면 두 질의는 동치입니다. 이 정의가 바로 위 그림이 시각화하는 내용입니다.

마지막으로 **동치성 검사 문제(Definition 4.6)** 가 진술됩니다. 그래프 질의 $Q$, 관계형 질의 $Q'$, 그리고 그래프 스키마 $\Psi\_G$에서 관계형 스키마 $\Psi\_R$로의 변환기 $\Phi$가 주어졌을 때, 동치성 검사 문제는 $Q \simeq\_\Phi Q'$이 성립하는지를 결정하는 문제입니다.

이로써 동기 예제에서 직관적으로만 다루었던 "변환기를 기준으로 한 동치성"이 완전히 형식적으로 정의되었습니다. 변환기 $\Phi$가 두 이질적 데이터 모델의 인스턴스를 1차 논리 위에서 정렬해 주는 다리 역할을 하고, 그 다리 위에서 두 질의가 항상 같은 테이블을 내놓는지가 동치성의 기준이 되는 것입니다. 이 형식적 토대는 다음 단계에서 전개될 동치성 검사 알고리즘—유도 관계형 스키마와 표준 변환기 추론, 구문 지향적 트랜스파일레이션, 그리고 SQL 동치성 검사로의 환원—이 무엇을 증명해야 하는지를 명확히 규정하는 기준점이 됩니다.
# 동치성 검사 알고리즘: 세 단계로 풀어내는 그래프-관계형 질의 검증

앞선 문제 정의에서 변환기 $\Phi$를 기준으로 한 동치성 $Q\_G \simeq\_\Phi Q\_R$이 무엇을 의미하는지를 형식적으로 못 박았다면, 이제 그 동치성을 실제로 **어떻게 판정하는가**를 다룰 차례입니다. 이 절은 논문의 알고리즘적 심장부로, 이질적인 두 데이터 모델의 질의 동치성 문제를 잘 연구된 SQL 동치성 문제로 환원하는 구체적인 절차를 단계별로 제시합니다.

## 전체 알고리즘의 골격

핵심 절차는 `CheckEquivalence`라는 프로시저로 요약됩니다. 이 프로시저는 그래프 스키마 $\Psi\_G$, 관계형 스키마 $\Psi\_R$, Cypher 질의 $Q\_G$, SQL 질의 $Q\_R$, 그리고 사용자가 제공한 변환기 $\Phi$를 입력으로 받아, 두 질의가 동치이면 $\top$(참)을, 검증에 실패하면 $\bot$을 반환합니다. 알고리즘의 작동은 정확히 세 개의 연속된 단계로 구성됩니다.

첫 번째 단계는 **스키마와 변환기 추론**입니다. 그래프 스키마 $\Psi\_G$만 주어지면 `InferSDT` 함수를 호출하여 두 가지 산출물을 동시에 얻어냅니다. 하나는 그래프를 가장 자연스럽게 관계형으로 표현한 유도 관계형 스키마 $\Psi\_R' = (S, \xi)$이고, 다른 하나는 그래프 인스턴스를 이 유도 스키마의 관계형 인스턴스로 변환하는 표준 데이터베이스 변환기 $\Phi\_{\mathsf{sdt}}$입니다. 코드로 표현하면 $(\Phi\_{\mathsf{sdt}}, \Psi\_R') \leftarrow \textsc{InferSDT}(\Psi\_G)$가 됩니다.

두 번째 단계는 **구문 지향적 트랜스파일레이션**입니다. 추론된 변환기와 무결성 제약을 활용하여 Cypher 질의 $Q\_G$를 SQL 질의 $Q\_{R'}$로 번역하는데, 이때 $\textsc{Transpile}(Q\_G, \Phi\_{\mathsf{sdt}}, \Psi\_R')$의 결과는 표준 변환기 $\Phi\_{\mathsf{sdt}}$를 기준으로 $Q\_G$와 **증명 가능하게 동치**임이 보장됩니다.

세 번째 단계는 **SQL 동치성 검사**입니다. 알고리즘은 유도 스키마 $\Psi\_R'$의 인스턴스를 목표 스키마 $\Psi\_R$의 인스턴스로 변환하는 잔여 데이터베이스 변환기 $\Phi\_{\mathsf{rdt}}$를 계산한 뒤, 두 SQL 질의 $Q\_R$과 $Q\_{R'}$이 이 잔여 변환기 $\Phi\_{\mathsf{rdt}}$를 기준으로 동치인지를 검사하며, 이 작업은 `ReduceToSQL` 프로시저에 위임됩니다.

### 왜 직접 비교하지 않고 이렇게 우회하는가

여기서 자연스럽게 떠오르는 의문은 "차라리 그래프 질의 $Q\_G$와 관계형 질의 $Q\_R$의 동치성을 직접 추론하면 더 간단하지 않은가?"입니다. 논문은 이 직접적 접근을 채택하지 않은 두 가지 분명한 이유를 제시합니다. 첫째, 그래프와 관계형 질의의 동치성을 직접 검증하려면 그래프와 관계 양쪽 모두에 대한 적절한 SMT 인코딩이 필요한데, 이는 결과적으로 풀어야 할 제약 충족 문제를 훨씬 어렵게 만듭니다. 둘째, 문제를 관계형 동치성 검사로 환원하면 SQL을 위해 이미 개발되어 온 다양한 도구—[Chandra et al.](https://www.semanticscholar.org/paper/43faccfff448d78e108f9bb7643bd024fb61ebbb)의 테스팅 도구, [He et al.](https://doi.org/10.1145/3649849)의 유계 모델 검사기(VeriEQL), [Wang et al.](https://doi.org/10.1145/3158144)의 연역적 검증기(Mediator)—를 그대로 활용할 수 있습니다. 즉 근본적으로 다른 두 데이터 모델을 동시에 추론하는 난제를 피하고, 동질적인 SQL 영역에서 검증을 수행하는 전략적 선택인 셈입니다. 이는 [SPES](https://www.semanticscholar.org/paper/b8a0c7ca6266e60177c89005902492709db4db88)나 [VeriEQL](https://doi.org/10.1145/3649849) 같은 선행 연구들이 공통적으로 채택한 "복잡한 문제를 더 다루기 쉬운 하위 문제로 환원"하는 철학과 정확히 맞닿아 있습니다.

## 유도 관계형 스키마와 표준 변환기의 추론

첫 단계인 `InferSDT` 절차는 다음과 같은 형태의 추론 규칙(inference rule)으로 형식화됩니다.

$$
\Psi\_G \hookrightarrow (S, \xi, \Phi\_{\mathsf{sdt}})
$$

이 판단(judgment)은 그래프 스키마의 요소($\Psi\_G$, 즉 노드·엣지·부분 그래프)가 유도 관계형 스키마 $\Psi\_R = (S, \xi)$와 표준 변환기 $\Phi\_{\mathsf{sdt}}$로 사상됨을 의미합니다. 추론 규칙이라는 형식은 마치 "전제가 성립하면 결론이 성립한다"는 논리 규칙을 분수 형태로 적어 내려가는 방식인데, 분수의 위쪽(분자)이 전제이고 아래쪽(분모)이 도출되는 결론입니다.

### 노드 규칙: 노드 타입을 테이블로

가장 기본이 되는 **Node 규칙**은 각 노드 타입을 하나의 관계형 테이블로 변환합니다. 레이블 $l$과 속성 키 $K\_1, \ldots, K\_n$을 가진 노드 타입 $t\_{\mathsf{node}} = (l, K\_1, \ldots, K\_n)$이 주어지면, 속성 $K\_1, \ldots, K\_n$을 가진 테이블 $R\_l$이 유도 스키마에 도입됩니다. 이때 앞선 형식적 정의에서 약속했던 기본 속성 키 $K\_1$이 그대로 이 테이블의 기본 키(primary key)가 되어, 무결성 제약 $\xi = (\mathsf{PK}(R\_l) = K\_1)$이 생성됩니다. 동시에 변환기 항목으로는 $\Phi = \{l(K\_1, \ldots, K\_n) \to R\_l(K\_1, \ldots, K\_n)\}$이 만들어집니다. 직관적으로 이 변환기 규칙은 "그래프 안에 레이블 $l$을 가진 원소가 속성 값 $(K\_1, \ldots, K\_n)$으로 존재하면, 관계형 테이블 $R\_l$에 같은 값을 가진 튜플이 존재한다"는 뜻입니다.

### 엣지 규칙: 외래 키로 연결 정보를 보존

**Edge 규칙**은 한층 흥미롭습니다. 엣지는 단순히 데이터를 담을 뿐 아니라 두 노드 사이의 연결 정보를 품고 있기 때문에, 이를 관계형으로 옮기려면 외래 키(foreign key)가 필요합니다. 레이블 $l$, 출발 노드 타입 $t\_{\mathsf{src}}$(레이블 $s$), 도착 노드 타입 $t\_{\mathsf{tgt}}$(레이블 $t$), 그리고 속성 키 $K\_1, \ldots, K\_m$을 가진 엣지 타입 $t\_{\mathsf{edge}} = (l, t\_{\mathsf{src}}, t\_{\mathsf{tgt}}, K\_1, \ldots, K\_m)$에 대해, 테이블 $R\_l$이 속성 $K\_1, \ldots, K\_m, \mathsf{fk}\_s, \mathsf{fk}\_t$를 가지고 도입됩니다. 여기서 $K\_1$은 다시 기본 키가 되고, 새롭게 추가된 $\mathsf{fk}\_s$와 $\mathsf{fk}\_t$는 각각 출발 노드와 도착 노드에 대응하는 테이블의 기본 키를 가리키는 외래 키입니다. 따라서 생성되는 무결성 제약은 세 개의 논리곱으로 표현됩니다.

$$
\mathsf{PK}(R\_l) = R\_l.K\_1 \land \mathsf{FK}(R\_l.\mathsf{fk}\_s) = \mathsf{PK}(R\_s) \land \mathsf{FK}(R\_l.\mathsf{fk}\_t) = \mathsf{PK}(R\_t)
$$

여기서 $R\_s$와 $R\_t$는 각각 출발·도착 노드에 해당하는 테이블입니다. 변환기 항목은 노드 규칙과 유사하되 출발·도착 외래 키가 인자 끝에 덧붙은 $\Phi = \{l(K\_1, \ldots, K\_m, \mathsf{fk}\_s, \mathsf{fk}\_t) \to R\_l(K\_1, \ldots, K\_m, \mathsf{fk}\_s, \mathsf{fk}\_t)\}$의 형태가 됩니다. 이 장면은 앞서 형식적 기반에서 살펴본 $\mathcal{C}$ 함수가 엣지를 그라운드 술어로 옮길 때 출발·도착 노드를 인자 끝에 붙이던 방식과 정확히 일관됩니다.

### 집합 규칙과 스키마 규칙: 부분 결과의 결합

나머지 두 규칙은 개별 타입의 변환 결과를 합쳐 전체 스키마를 구성하는 역할을 합니다. **Set 규칙**은 두 타입 집합 $T\_1$과 $T\_2$가 각각 $(S\_1, \xi\_1, \Phi\_1)$과 $(S\_2, \xi\_2, \Phi\_2)$로 변환될 때, 이들의 분리합집합(disjoint union) $T\_1 \uplus T\_2$가 각 구성 요소를 합친 $(S\_1 \uplus S\_2, \xi\_1 \land \xi\_2, \Phi\_1 \cup \Phi\_2)$로 변환됨을 명시합니다. 무결성 제약은 논리곱으로, 스키마와 변환기는 합집합으로 결합되는 점에 주목할 필요가 있습니다. **Schema 규칙**은 노드 타입 집합 $T\_N$과 엣지 타입 집합 $T\_E$의 결합 $(T\_N \uplus T\_E)$가 $(S, \xi, \Phi)$로 변환되면, 그래프 스키마 $(T\_N, T\_E)$ 전체도 같은 결과로 변환됨을 마무리짓습니다.

이 추론 규칙들이 어떻게 동작하는지는 구체적인 예제로 보면 한결 명확합니다. 직원과 부서를 다루는 그래프 스키마를 생각해 봅시다. 이 스키마는 `EMP`와 `DEPT`라는 두 노드 레이블과 그 사이를 잇는 `WORK_AT` 엣지 레이블로 구성되며, `EMP`는 속성 키 `id`, `name`을, `DEPT`는 `dnum`, `dname`을, `WORK_AT`는 `wid`를 가지고 굵게 표시된 것이 기본 속성 키입니다. 이 그래프 스키마에 위 규칙들을 적용하면, `emp`(속성 `id`, `name`), `dept`(속성 `dnum`, `dname`), 그리고 `work_at` 테이블이 유도되는데, 특히 `work_at` 테이블은 자신의 기본 키 `wid` 외에 `emp`의 `id`를 가리키는 외래 키 `SRC`와 `dept`의 `dnum`을 가리키는 외래 키 `TGT`를 추가로 갖게 됩니다. 이렇게 그래프의 엣지가 관계형 세계에서 외래 키를 품은 테이블로 자연스럽게 표현됩니다.

표준 변환기 $\Phi\_{\mathsf{sdt}}$가 실제 데이터를 어떻게 옮기는지도 직관적으로 이해할 수 있습니다. 두 직원 노드(`id`가 1인 A, 2인 B)와 두 부서 노드(`CS`, `EE`)가 `:WORK_AT` 엣지로 연결된 그래프 인스턴스가 있다면, SDT는 이를 `emp` 테이블의 두 행 `(1,A)`, `(2,B)`, `dept` 테이블의 두 행 `(1,CS)`, `(2,EE)`, 그리고 출발·도착 외래 키를 담은 `work_at` 테이블의 행들로 변환합니다. 노드와 엣지의 각 출현(occurrence)이 대응 테이블의 튜플로 일대일 사상되는 것입니다.

## 구문 지향적 트랜스파일레이션

표준 변환기를 토대로, 이제 Cypher 질의를 유도 스키마 위의 SQL 질의로 번역하는 핵심 작업으로 넘어갑니다. 이 작업은 여러 도전을 내포합니다. Cypher에는 부분 그래프에 대한 패턴 매칭이나 선택적 매칭처럼 SQL에 직접 대응물이 없는 기능이 있고, Cypher는 매칭된 부분 그래프에 대해 집계하는 반면 SQL은 그룹화된 튜플에 대해 집계하며, 질의 전반에 걸쳐 그래프 노드와 엣지의 일관된 사상을 유지해 참조 무결성을 지켜야 하기 때문입니다. 그 핵심 아이디어는 [Cypher 언어](https://doi.org/10.1145/3183713.3190657)의 경로 패턴이 SQL의 관계형 조인으로 표현될 수 있다는 점이며, 구체적으로 `Match` 절은 내부 조인(inner join)으로, `OptMatch`(선택적 매칭) 절은 외부 조인(outer join)으로 번역됩니다.

### 질의 수준의 번역 규칙

질의 번역 규칙은 $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash Q \overset{\textsf{query}}{\longrightarrow} Q'$ 형태의 판단을 사용합니다. 이는 SDT $\Phi\_{\mathsf{sdt}}$와 유도 스키마 $\Psi\_R$가 주어졌을 때 Cypher 질의 $Q$가 SQL 질의 $Q'$으로 사상됨을 뜻합니다. 이 중 `Return`의 처리가 특별한 주의를 요하는데, 집계 함수의 유무에 따라 두 갈래로 나뉘기 때문입니다.

집계 함수가 없는 경우(즉 $\neg\mathsf{hasAgg}(\overline{E})$인 경우)에는 **Q-Ret 규칙**이 적용되어, Cypher 질의 $\mathsf{Return}(C, \overline{E}, \overline{k})$가 단순한 SQL 투영(projection) $\Pi\_{\rho\_{\overline{k}}(\overline{E'})}(Q)$로 번역됩니다. 여기서 $Q$는 절 $C$를 번역한 결과이고, $\overline{E'}$은 표현식 $\overline{E}$를 번역한 것이며, 모든 속성은 $\overline{k}$로 이름이 변경됩니다.

반면 집계 함수가 등장하는 경우($\mathsf{hasAgg}(\overline{E})$)에는 **Q-Agg 규칙**으로 전환되어 `GroupBy` 질의가 생성됩니다. 이는 SQL이 집계를 처리할 때 집계되지 않은 열을 기준으로 행을 분할(partition)하는 `GroupBy`를 사용하기 때문입니다. Cypher 질의 안에서 집계가 아닌 표현식들 $\overline{A} = \mathsf{filter}(\lambda x. \neg\mathsf{IsAgg}(x), \overline{E'})$이 그룹화 키(grouping key)로 작동하고, 집계 표현식들은 각 그룹에 대한 올바른 결과 계산을 담당합니다. 이렇게 그룹화 키를 "집계가 아닌 표현식"으로 자동 식별하는 방식은, 앞선 동기 예제에서 보았던 집계 의미론의 미묘함을 정확히 다루기 위한 핵심 장치입니다. 이 밖에 정렬을 다루는 Q-OrderBy, 집합 합집합을 $\cup$로 옮기는 Q-Union, 백 합집합을 $\uplus$로 옮기는 Q-UnionAll 규칙이 있습니다.

구체적인 예를 보면, 직원 수를 부서별로 세는 다음 Cypher 질의를 생각해 봅시다.

$$
\mathsf{Return}(\mathsf{Match}([(\texttt{n},\texttt{EMP}),(\texttt{e},\texttt{WORK\_AT},\rightarrow),(\texttt{m},\texttt{DEPT})],\top), [\texttt{m.dname}, \mathsf{Count}(\texttt{n.id})], [\texttt{name}, \texttt{num}])
$$

여기에는 $\mathsf{Count}$ 집계가 있으므로 Q-Agg 규칙이 적용됩니다. 먼저 매칭 절이 SQL 질의 $Q$로 번역되고, 반환 표현식 `m.dname`과 $\mathsf{Count}(\texttt{n.id})$가 각각 대응되는 SQL 표현식으로 옮겨집니다. 이 중 집계를 포함하지 않는 `m.dname`이 `GroupBy`의 그룹화 키로 선택되며, 집계 결과에 대한 필터링이 없으므로 `Having` 절은 생성되지 않습니다. 따라서 최종 번역 결과는 $\texttt{GroupBy}(Q, [\texttt{m.dname}], \rho\_{[\texttt{name}, \texttt{num}]}([\texttt{m.dname}, \texttt{Count(n.id)}]), \top)$이 됩니다.
## 절(clause) 수준의 번역 규칙

질의 전체를 다룰 때와 달리, 개별 절을 번역할 때는 절 안에서 사용된 노드·엣지 변수에 관한 추가 정보를 추적해야 합니다. 그 이유는 직관적입니다. 같은 변수가 서로 다른 절에 여러 번 등장할 때, 이들이 모두 SQL에서 **같은 튜플**을 가리키도록 번역되어야 정합성이 유지되기 때문입니다. 이를 위해 절 번역 판단은 $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash C \overset{\textsf{clause}}{\longrightarrow} \mathcal{X}, Q'$의 형태를 띠는데, 여기서 Cypher 절 $C$가 SQL 질의 $Q'$으로 번역되며 $\mathcal{X}$는 사용된 모든 노드·엣지 변수의 집합입니다. 즉 앞선 질의 규칙이 출력 형태(투영, 그룹화)에 집중했다면, 절 규칙은 변수 추적이라는 추가 책무를 짊어진다는 점이 핵심 차이입니다.

$\mathsf{Match}$와 $\mathsf{OptMatch}$ 절의 번역 규칙은 "Cypher의 그래프 패턴 매칭이 SQL의 조인 연산 시퀀스로 모사될 수 있다"는 관찰에 근거합니다. $\mathsf{Match}$에 대한 조인은 내부 조인이고, $\mathsf{OptionalMatch}$에 대한 조인은 좌외부 조인(left outer join)입니다. 그 직관은 이렇습니다. $\mathsf{Match}$ 절은 매칭되는 패턴이 없으면 아무 결과도 반환하지 않는데, 이는 매칭되지 않은 튜플을 버리는 내부 조인의 동작과 정확히 일치합니다. 반대로 $\mathsf{OptionalMatch}$ 절은 매칭이 없을 때 `null`을 반환하는데, 이는 매칭되지 않은 행을 `null` 값과 함께 포함시키는 외부 조인의 동작과 닮았습니다. 이 대응 관계는 앞서 형식적 기반에서 삼치 논리와 외부 조인 연산자가 신중히 도입된 이유와 정확히 맞물립니다.

### 이전 절에 패턴을 잇는 C-Match2 규칙

**C-Match2 규칙**은 $\mathsf{Match}(C, PP, \phi)$ 형태, 즉 선행 절 $C$ 뒤에 경로 패턴 $PP$가 이어지는 경우를 다룹니다. 먼저 선행 절 $C$를 서브쿼리 $Q\_1$로, 경로 패턴 $PP$를 또 다른 서브쿼리 $Q\_2$로 번역하고, 각각에서 사용된 노드·엣지 변수 집합 $\mathcal{X}\_1$과 $\mathcal{X}\_2$를 수집합니다. 핵심은 두 부분에 공통으로 등장하는 변수 $X$(레이블 $l$)마다 다음과 같은 조인 술어를 생성한다는 점입니다.

$$
\phi'' = \phi' \land \bigwedge\_{(X:l)\in\mathcal{X}\_1\cap\mathcal{X}\_2} T\_1.\mathsf{PK}(R\_l) = T\_2.\mathsf{PK}(R\_l) \quad \text{where } l(\ldots)\to R\_l(\ldots) \in \Phi\_{\mathsf{sdt}}
$$

여기서 $T\_1$과 $T\_2$는 새롭게 도입된(fresh) 별칭이며, 공통 변수의 기본 키가 일치하도록 강제하는 조인 술어 $T\_1.K\_1 = T\_2.K\_1$가 결합됩니다. 이렇게 함으로써 절의 서로 다른 부분에 나타난 같은 변수의 출현들이 그들의 기본 키 위에서 조인되어 SQL 번역에서 동일한 튜플을 가리키게 됩니다. 최종 결과는 $\rho\_{T\_1}(Q\_1) \bowtie\_{\phi''} \rho\_{T\_2}(Q\_2)$의 형태가 됩니다. **C-OptMatch 규칙**은 이와 거의 동일하되, 내부 조인 $\bowtie$ 대신 좌외부 조인을 사용한다는 점만 다릅니다.

### 변수 스코프를 재설정하는 C-With 규칙

**C-With 규칙**은 $\mathsf{With}(C, \overline{Y}, \overline{Z})$ 절을 SQL의 이름 변경(renaming) 연산으로 번역합니다. 구체적으로 $\Pi\_{\rho\_{L[\overline{Z}/\overline{Y}]}(L)}(Q)$라는 질의를 생성하는데, 이는 열을 투영하면서 옛 이름 $\overline{Y}$를 새 이름 $\overline{Z}$로 치환합니다. 이는 앞서 형식적 기반에서 $\mathsf{With}$ 절이 중간 결과를 다음 단계로 전달하면서 변수 스코프를 재설정하는 역할을 한다고 설명한 부분의 구체적인 SQL 실현입니다.

이 규칙들을 직원-부서 예제에 적용해 보면 한층 명확해집니다. 앞서 다룬 매칭 절 $\mathsf{Match}([(\texttt{n},\texttt{EMP}),(\texttt{e},\texttt{WORK\_AT},\rightarrow),(\texttt{m},\texttt{DEPT})],\top)$에 C-Match1 규칙을 적용하면, 패턴 안의 모든 변수 $\mathcal{X}=\{(\texttt{n}:\texttt{EMP}),(\texttt{e}:\texttt{WORK\_AT}),(\texttt{m}:\texttt{DEPT})\}$가 수집되고, 다음과 같은 SQL 질의로 번역됩니다.

$$
\rho\_{\mathtt{n}}(\mathtt{emp})\bowtie\_{\mathtt{n.id}=\mathtt{e.SRC}}\rho\_{\mathtt{e}}(\mathtt{work\_at})\bowtie\_{\mathtt{e.TGT}=\mathtt{m.dnum}}\rho\_{\mathtt{m}}(\mathtt{dept})
$$

C-Match1 규칙은 $\mathsf{Match}(PP, \phi)$를 패턴 번역 결과 $Q$에 술어 $\phi'$을 적용한 선택 연산 $\sigma\_{\phi'}(Q)$로 옮기므로, 술어가 $\top$(참)인 이 경우 전체 절은 $\sigma\_{\top}(\cdots)$의 형태로 마무리됩니다.

선택적 매칭의 예도 살펴볼 가치가 있습니다. 위의 매칭 절을 선행 절 $C$로 삼고 $PP=[(\texttt{m},\texttt{DEPT})]$인 $\mathsf{OptMatch}(C, PP, \phi)$를 생각해 봅시다. 이때 $\mathcal{X}\_1=\{(\texttt{n}:\texttt{EMP}),(\texttt{e}:\texttt{WORK\_AT}),(\texttt{m}:\texttt{DEPT})\}$이고 $\mathcal{X}\_2=\{(\texttt{m}:\texttt{DEPT})\}$이므로, 두 집합이 공유하는 노드는 $(\texttt{m},\texttt{DEPT})$입니다. `dept`의 기본 키가 `dnum`이므로, 절은 좌외부 조인 $\rho\_{T\_1}(Q\_1)⟕\_{T\_1.\mathtt{dnum}=T\_2.\mathtt{dnum}}\rho\_{T\_2}(Q\_2)$로 번역되며, 여기서 $Q\_2 = \sigma\_\phi(\rho\_{\mathtt{m}}(\mathtt{dept}))$입니다.

## 경로 패턴(path pattern)의 번역

패턴 번역은 $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash PP \overset{\textsf{pattern}}{\longrightarrow} \mathcal{X}, Q'$ 형태의 판단을 사용하며, 귀납적 구조를 따릅니다. 두 개의 핵심 규칙이 있습니다.

**기저 사례인 PT-Node 규칙**은 단일 노드 패턴을 다룹니다. 노드 변수 $X$를 SDT $\Phi\_{\mathsf{sdt}}$로부터 도출된 관계형 스키마의 대응 테이블 $R'\_l$로 사상하고, 이를 $X$로 이름을 변경하여 $\rho\_X(R\_l)$을 산출합니다.

**귀납 사례인 PT-Path 규칙**은 새로운 노드 $(X\_2, l\_2)$가 기존 부분 패턴 $PP$를 확장하는 더 복잡한 패턴을 다룹니다. 이 규칙은 $PP$의 머리(head)에 있는 연결 노드 $(X\_3, l\_3)$를 식별하고, 노드 $X\_1$, $X\_2$, $X\_3$에 대응하는 테이블 $R'\_{l\_1}, R'\_{l\_2}, R'\_{l\_3}$을 찾아낸 뒤 적절한 조인을 결정합니다. 핵심은 엣지 테이블 $R'\_{l\_2}$의 외래 키 $\mathsf{fk}\_s$와 $\mathsf{fk}\_t$를 출발·도착 노드 테이블의 기본 키에 연결하는 조인 술어를 구성하는 것입니다.

$$
\phi = (R\_{l\_2}.\mathsf{fk}\_s = \mathsf{PK}(R\_{l\_1})) \qquad \phi' = (R\_{l\_2}.\mathsf{fk}\_t = \mathsf{PK}(R\_{l\_3}))
$$

여기서 주목할 미묘한 전제 조건이 있습니다. 규칙은 $\xi(\Psi\_R) \Rightarrow \phi \land \phi'$, 즉 유도 스키마의 무결성 제약이 이 두 외래 키 조인 술어를 함의해야 한다는 조건을 요구합니다. 이는 앞서 엣지 규칙이 생성한 외래 키 제약이 바로 이 조인의 타당성을 떠받친다는 것을 형식적으로 보장하는 장치입니다. 결과적으로 패턴은 $\rho\_{X\_1}(R\_{l\_1}) \bowtie\_\phi \rho\_{X\_2}(R\_{l\_2}) \bowtie\_{\phi'} Q'$의 형태로 번역됩니다. 이 규칙을 직원-부서 경로 패턴 $[(\texttt{n},\texttt{EMP}),(\texttt{e},\texttt{WORK\_AT},\rightarrow),(\texttt{m},\texttt{DEPT})]$에 적용하면, 먼저 PT-Node로 노드 패턴 $(\texttt{m},\texttt{DEPT})$로부터 $\rho\_{\texttt{m}}(\texttt{DEPT})$를 얻고, 이를 토대로 PT-Path를 적용해 앞서 본 세 테이블의 조인 질의를 귀납적으로 완성하게 됩니다.

### 번역의 정확성을 떠받치는 두 정리

이 모든 번역 규칙이 단순한 휴리스틱이 아니라 형식적으로 올바름을 보장한다는 것이 이 절의 이론적 정점입니다. **번역의 건전성(Soundness, Theorem 5.7)** 은 그래프 스키마 $\Psi\_G$ 위의 Cypher 질의 $Q$가 $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash Q \overset{\textsf{query}}{\longrightarrow} Q'$로 번역되면, 그 결과 $Q'$이 표준 변환기 $\Phi\_{\mathsf{sdt}}$를 기준으로 $Q$와 동치($Q \simeq\_{\Phi\_{\mathsf{sdt}}} Q'$)임을 단언합니다. 즉 번역기가 만들어 낸 SQL 질의는 결코 의미를 왜곡하지 않습니다. 한편 **번역의 완전성(Completeness, Theorem 5.8)** 은 Featherweight Cypher 문법이 받아들이는 임의의 Cypher 질의 $Q$에 대해, 반드시 그에 대응하는 SQL 질의 $Q'$이 존재하여 번역 관계가 성립함을 보장합니다. 다시 말해 번역 규칙이 다루는 Cypher 조각 안에서는 어떤 질의도 번역에 실패하지 않습니다. 이 두 정리가 합쳐져, 앞선 전체 알고리즘의 두 번째 단계가 "증명 가능하게 동치인 SQL을 항상 생성한다"는 강한 보장을 부여하는 것입니다.

## SQL 동치성 검사로의 환원

알고리즘의 마지막 단계는 트랜스파일된 질의를 활용해 원래 문제를 두 SQL 질의 사이의 동치성 검사로 환원하는 것입니다. 이 작업은 `ReduceToSQL` 프로시저로 형식화되며, 그 핵심은 **잔여 데이터베이스 변환기(residual database transformer)** $\Phi\_{\mathsf{rdt}}$를 추론하는 데 있습니다.

이 추론은 놀랍도록 단순한 구문적 치환(syntactic substitution)으로 이루어집니다. SDT의 모든 절이 $P\_1(\ldots) \to P\_0(\ldots)$의 형태를 띤다는 점에 착안하여, 다음과 같은 치환 사상 $\sigma$를 정의합니다.

$$
\sigma \leftarrow \{P\_1 \mapsto P\_0 \mid P\_1(\ldots) \to P\_0(\ldots) \in \Phi\_{\mathsf{sdt}}\}
$$

그리고 사용자가 제공한 변환기 $\Phi$ 안의 모든 $P\_1$ 출현을 $P\_0$로 바꾸어 잔여 변환기를 얻습니다.

$$
\Phi\_{\mathsf{rdt}} \leftarrow \Phi[\sigma]
$$

직관적으로 풀어 보면, SDT는 "그래프 레이블 $P\_1$"을 "유도 스키마의 테이블 $P\_0$"으로 대응시키는 규칙들의 모음입니다. 따라서 원래 사용자 변환기 $\Phi$가 그래프 레이블을 통해 목표 스키마를 기술하던 것을, 그래프 레이블 자리를 유도 스키마 테이블 이름으로 갈아끼우면, 곧바로 "유도 스키마의 인스턴스를 목표 스키마의 인스턴스로 변환"하는 잔여 변환기가 완성됩니다. 이렇게 두 변환기를 합성하는 우아한 방식 덕분에, 복잡한 별도 추론 없이 단 한 줄의 치환으로 환원이 가능해집니다. 이 잔여 변환기 개념은 앞선 동기 예제에서 유도 스키마와 목표 스키마 사이의 "차이"를 메우던 장치를 형식적으로 구현한 것입니다.

잔여 변환기가 두 관계형 스키마 사이의 관계를 명시하므로, 이제 기존의 SQL 동치성 검사 도구를 그대로 호출하여 $\mathsf{CheckSQL}(\Psi\_R, Q\_R, \Psi\_R', Q\_R', \Phi\_{\mathsf{rdt}})$를 수행할 수 있습니다. 이 단계가 바로 [VeriEQL](https://doi.org/10.1145/3649849)의 심볼릭 튜플 기반 유계 검증이나 [Mediator](https://doi.org/10.1145/3158144)의 쌍모의(bisimulation) 기반 연역적 검증 같은 기성 도구가 투입되는 지점입니다. 특히 Mediator가 도입한 쌍모의 불변식(bisimulation invariant)을 통한 스키마 마이그레이션 동치성 검증 기법은, 서로 다른 두 관계형 스키마 $\Psi\_R$과 $\Psi\_R'$을 변환기로 연결한 채 동치성을 따지는 이 문제 설정과 정확히 들어맞습니다.

### 전체 방법론의 정확성 보장

마지막 두 정리는 이 환원 전략 전체가 올바름을 형식적으로 못 박습니다. **건전성(Soundness, Theorem 5.9)** 은 $\mathsf{CheckSQL}$이 잔여 변환기 $\Phi\_{\mathsf{rdt}}$로 연결된 SQL 질의들에 대한 건전한 동치성 검사 절차라고 할 때, `CheckEquivalence`$(\Psi\_G, Q\_G, \Psi\_R, Q\_R, \Phi)$가 $\top$을 반환하면 실제로 $Q\_G \simeq\_\Phi Q\_R$이 성립함을 보장합니다. 즉 알고리즘이 "동치"라고 선언하면 그것은 진실입니다. 거꾸로 **완전성(Completeness, Theorem 5.10)** 은 $\mathsf{CheckSQL}$이 완전한 절차라고 할 때, $Q\_G \simeq\_\Phi Q\_R$이 성립하면 반드시 `CheckEquivalence`가 $\top$을 반환함을 보장합니다. 다시 말해 두 질의가 정말로 동치라면 알고리즘은 결코 그것을 놓치지 않습니다.

여기서 중요한 점은, 이 두 정리가 모두 "$\mathsf{CheckSQL}$이 건전하다/완전하다면"이라는 조건부 형태로 진술된다는 것입니다. 이는 Graphiti의 전체 정확성이 백엔드 SQL 검사기의 능력에 위임된다는 뜻인데, 바로 이 설계 덕분에 SQL 동치성 검증 분야에서 축적된 어떤 도구든 플러그인처럼 갈아끼울 수 있는 모듈성을 얻게 됩니다. 일반적인 SQL 동치성 검사가 결정 불가능(undecidable)하다는 근본적 한계 때문에 완전한 백엔드를 항상 가질 수는 없지만, 유계 모델 검사기(반박에 강함)와 연역적 검증기(무계 증명에 강함)를 조합함으로써 실용적으로 검증과 반박 양쪽에서 강력한 성능을 끌어낼 수 있습니다. 이로써 이질적인 그래프-관계형 질의 동치성 문제는, 형식적으로 보장된 트랜스파일레이션과 단 한 줄의 잔여 변환기 추론을 거쳐, 완전히 동질적인 SQL 동치성 문제로 환원되어 마무리됩니다.
# 실험적 평가: Graphiti는 실제로 버그를 찾아내는가

앞선 절들에서 그래프-관계형 질의 동치성 문제를 형식적으로 정의하고, 이를 트랜스파일레이션과 잔여 변환기를 거쳐 순수 SQL 동치성 문제로 환원하는 알고리즘을 살펴보았습니다. 이제 그 이론이 실제 도구 **Graphiti**로 구현되었을 때 얼마나 쓸모가 있는지를 세 가지 실험을 통해 검증할 차례입니다. 여기서 중요한 점은, Graphiti의 검증 방법론이 Cypher-SQL 동치성 검사를 순수 SQL 동치성 검사로 환원하기 때문에, 그 결과가 결국 어떤 SQL 백엔드를 쓰느냐에 따라 달라진다는 것입니다. 따라서 첫 번째 실험에서는 유계 모델 검사기(bounded model checker)인 VeriEQL을, 두 번째 실험에서는 연역적 검증기(deductive verifier)인 Mediator를 백엔드로 사용하며, 마지막으로 트랜스파일레이션 결과의 품질 자체를 평가합니다.

## 벤치마크의 구성

평가는 총 410쌍의 SQL-Cypher 질의에 대해 수행되었으며, 이 질의들은 여섯 가지 출처에서 수집되었습니다. 우선 사용자들이 SQL을 Cypher로(혹은 그 반대로) 번역하는 방법을 묻는 StackOverflow 게시글 12개, 공식 Neo4j 가이드를 포함한 튜토리얼 예제 26개, SQL 분석 관련 학술 논문에서 가져온 7개의 예제가 있습니다. 또한 VeriEQL 논문의 세 데이터셋에서 각각 20개씩 무작위로 표본을 추출해 Cypher 경험이 최소 3개월 이상인 사람들에게 동치인 Cypher 질의를 작성하게 한 60개의 벤치마크, 그리고 Mediator 평가 집합에서 가져온 100개의 벤치마크가 포함됩니다.

특히 Mediator 벤치마크는 흥미로운 변환 과정을 거칩니다. 원래 Mediator 벤치마크는 서로 다른 스키마 $\Psi\_1$, $\Psi\_2$ 위의 SQL 질의 쌍 $(Q\_1, Q\_2)$로 구성되는데, 이를 그래프 스키마 $\Psi\_1'$, $\Psi\_2'$ 위의 Cypher 질의 $G\_1$, $G\_2$를 만든 뒤 $(G\_1, Q\_2)$와 $(G\_2, Q\_1)$ 형태의 교차 쌍으로 변환합니다. 여기서 $\Psi\_1$, $\Psi\_2$는 각각 $\Psi\_1'$, $\Psi\_2'$로부터 유도된 관계형 스키마입니다. 즉 그래프 질의와 관계형 질의가 한 쌍을 이루도록 재구성하여 Graphiti가 다룰 수 있는 형태로 만든 것입니다.

마지막으로 가장 현대적인 관심사를 반영하는 출처가 **GPT-Translate**입니다. GPT 같은 대규모 언어 모델이 코딩과 트랜스파일레이션에 점점 더 많이 쓰이고 있다는 점에 착안하여, 앞선 다섯 범주의 SQL 질의를 GPT로 Cypher로 변환한 205개의 벤치마크를 추가했습니다. 이는 자동 번역에서 발생하는 오류를 Graphiti가 얼마나 잘 잡아내는지를 평가하기 위한 것입니다.

각 벤치마크의 SQL·Cypher 질의 크기와, 그래프-관계형 관계를 기술하는 데이터베이스 변환기(Transformer)의 크기는 다음과 같이 정리됩니다. 크기는 AST(추상 구문 트리) 노드 수로 측정됩니다.

| 데이터셋 | #쌍 | SQL 크기(평균) | Cypher 크기(평균) | 변환기 크기(평균) |
|---|---|---|---|---|
| StackOverflow | 12 | 32.5 | 54.9 | 3.3 |
| Tutorial | 26 | 25.8 | 31.2 | 8.3 |
| Academic | 7 | 46.9 | 75.0 | 6.7 |
| VeriEQL | 60 | 42.2 | 65.8 | 6.7 |
| Mediator | 100 | 18.6 | 33.8 | 5.1 |
| GPT-Translate | 205 | 28.2 | 46.0 | 5.9 |
| Total | 410 | 28.2 | 45.7 | 5.9 |

여기서 주목할 만한 실용적 통찰은 데이터베이스 변환기를 작성하는 부담이 크지 않다는 점입니다. 논문 저자 중 한 명이 각 질의 쌍에 대해 스키마 설명을 바탕으로 변환기를 직접 구성했는데, 평균적으로 변환기 하나는 약 5.9개의 규칙으로 이루어지며 작성에 약 1분 정도가 걸립니다. 이는 [관계형-그래프 직접 매핑 연구](https://www.semanticscholar.org/paper/towards-complete-direct-mapping)에서 강조하듯, 스키마 매핑을 수작업으로 명세하는 것이 현실적으로 충분히 다룰 만한 작업임을 뒷받침하며, Graphiti가 사용자에게 명시적 변환기를 요구하는 설계 선택을 정당화합니다. 모든 실험은 Intel Core i7-8750H 프로세서와 32GB 메모리를 갖춘 노트북에서 Debian 12 운영체제로 수행되었습니다.

## 유계 모델 검사 백엔드(VeriEQL)를 이용한 평가

첫 번째 실험은 [VeriEQL](https://doi.org/10.1145/3649849)을 SQL 동치성 검사 백엔드로 사용합니다. VeriEQL은 심볼릭 튜플(symbolic tuple) 위에서 질의 의미론을 SMT 논리식으로 인코딩하는 유계 모델 검사기로, 심볼릭 테이블의 크기 한계(size bound)를 하이퍼파라미터로 요구합니다. 그런데 적절한 한계 값을 사전에 추정하기란 어렵기 때문에, 실험에서는 10분의 시간 제한을 두고 반례를 찾거나 시간이 소진될 때까지 한계 값을 점진적으로 늘려 갑니다. 반박된(refuted) 각 벤치마크에 대해 Graphiti는 VeriEQL이 생성한 관계형 반례를 활용하여 그래프 스키마 위의 반례를 구성합니다. 즉 SQL 수준에서 발견된 "두 질의가 다른 결과를 내는 데이터"를 다시 그래프 세계의 구체적인 반례로 끌어올리는 것입니다.

| 데이터셋 | #쌍 | #비동치 | 평균 검사 한계 | 평균 반박 시간(초) |
|---|---|---|---|---|
| StackOverflow | 12 | 1 | 19.2 | 0.6 |
| Tutorial | 26 | 2 | 17.7 | 56.2 |
| Academic | 7 | 1 | 12.5 | 5.4 |
| VeriEQL | 60 | 6 | 47.2 | 8.5 |
| Mediator | 100 | 0 | 33.2 | N/A |
| GPT-Translate | 205 | 27 | 18.7 | 25.9 |
| Total | 410 | 34 | 19.6 | 23.4 |

위 표에서 "평균 검사 한계" 열은 10분 시간 제한에 도달했을 때 심볼릭 테이블의 평균 크기(행 수 기준)를 나타냅니다. 결과를 보면 Graphiti는 410개 벤치마크 중 34개에 대해 동치성을 반박했으며, 반례를 찾는 데 평균 23.4초가 걸렸습니다. 나머지 376개에 대해서는 평균 크기 19.6의 심볼릭 테이블을 갖는 데이터베이스 인스턴스 범위 안에서 반례가 없음을 보이는 유계 검증을 수행했습니다.

### 발견된 버그의 의미

발견된 34개의 버그를 모두 수작업으로 검토한 결과, 도구가 생성한 모든 반례가 실제 오류(true positive)에 해당함이 확인되었습니다. 예상대로 GPT가 생성한 질의는 사람이 작성한 질의보다 오류 확률이 높았는데, GPT로 트랜스파일된 질의 중 13%에서 동치성에 대한 반례가 발견되었습니다. 이는 GPT가 SQL을 Cypher로 변환할 때 의미론적 버그를 끼워 넣을 수 있으며, Graphiti가 이러한 버그를 효과적으로 식별할 수 있음을 보여 줍니다. 개발자들이 코딩 작업에 대규모 언어 모델의 도움을 점점 더 많이 받는 추세를 감안하면, 이 결과는 LLM 생성 질의에 의존하는 개발자에게 Graphiti가 실질적 가치를 지님을 시사합니다.

더 놀라운 점은, 전문가가 작성한 것으로 여겨지는 StackOverflow, Tutorial, Academic, VeriEQL 범주에서도 잘못된 번역이 발견되었다는 사실입니다. 특히 SQL에 익숙한 개발자가 Cypher를 배우도록 돕기 위해 만들어진 Neo4j 공식 튜토리얼 예제에서조차 버그가 발견되었습니다. 이 튜토리얼은 동치라고 의도된 여러 SQL-Cypher 질의 쌍을 담고 있는데, 그중 하나에 대해 Graphiti가 실제로 결과가 다른 반례를 찾아낸 것입니다. 이러한 사례들은 앞서 동기 예제에서 보았던 "집계와 경로 매칭의 미묘한 의미론 차이"가 실제 현장에서 얼마나 흔하게 잠복해 있는지를 잘 드러냅니다.

### 거짓 음성(false negative)의 평가

VeriEQL이 유계 모델 검사기라는 점 때문에, 10분 시간 제한 안에 반박되지 않은 벤치마크라도 여전히 버그를 포함할 수 있습니다. 이 빈도를 가늠하기 위해 반박되지 않은 50쌍의 질의를 표본으로 뽑아 번역이 올바른지 수작업으로 검토했습니다. 그 결과 50개 중 48개는 실제로 올바른 번역이었지만, 2개(둘 다 GPT-Translate 범주)는 번역이 잘못되었음에도 Graphiti가 시간 제한 안에 반례를 찾지 못했습니다. 따라서 VeriEQL 백엔드를 사용한 Graphiti는 이론적인 건전성(soundness) 보장을 제공하지는 않지만, 버그 발견에 유용하며 잘못된 번역을 놓칠 확률이 낮음(수작업 검토 기준 약 4%)을 알 수 있습니다.

종합하면, 유계 모델 검사 백엔드를 사용한 Graphiti는 GPT로 트랜스파일된 205개 SQL 질의 중 27개의 버그를 식별했고, 동치라고 의도된 205개의 수작업 작성 질의 쌍 중에서도 7개의 불일치를 찾아냈는데, 여기에는 실제 현장에서 수집한 3개와 수작업 번역에서 나온 4개가 포함됩니다.

## 연역적 검증기(Mediator) 백엔드를 이용한 평가

두 번째 실험은 [Mediator](https://doi.org/10.1145/3158144)를 백엔드로 사용합니다. Mediator는 서로 다른 스키마 위의 SQL 애플리케이션을 추론하기 위한 SMT 기반 연역적 검증기로, 두 질의 사이의 쌍모의 불변식(bisimulation invariant)을 추론하는 방식으로 동작합니다. VeriEQL과 달리 Mediator는 테이블 크기에 제한이 없는 완전한(full-fledged) 검증을 수행할 수 있다는 장점이 있지만, 집계(aggregation)나 외부 조인(outer join)이 없는 제한된 SQL 부분 집합만 지원한다는 한계가 있습니다. 또한 VeriEQL과 달리 반례를 생성해 비동치를 입증하지는 못합니다. 따라서 Mediator를 백엔드로 쓸 때 Graphiti는 동치성을 증명하거나 "Unknown"을 반환하는 두 가지 결과만 낼 수 있습니다. 이 실험에서도 벤치마크당 10분의 시간 제한이 적용되었습니다.

| 데이터셋 | #쌍 | #지원 | #검증 | #Unknown | 평균 시간(초) |
|---|---|---|---|---|---|
| StackOverflow | 12 | 1 | 1 | 0 | 1.0 |
| Tutorial | 26 | 1 | 1 | 0 | 0.2 |
| Academic | 7 | 0 | 0 | 0 | N/A |
| VeriEQL | 60 | 0 | 0 | 0 | N/A |
| Mediator | 100 | 100 | 77 | 23 | 20.5 |
| GPT-Translate | 205 | 94 | 73 | 21 | 23.5 |
| Total | 410 | 196 | 152 | 44 | 16.8 |

"#지원" 열에서 보듯, 전체 벤치마크의 약 절반(410개 중 196개)이 Mediator가 지원하는 SQL 단편 안에 들어가므로, 평가는 이 부분 집합에 대해 수행되었습니다. 전체적으로 Graphiti는 이 196개 중 77.6%를 검증할 수 있었으며, 평균 실행 시간은 16.8초였습니다. Graphiti가 수행하는 구문 지향적 트랜스파일레이션 자체는 거의 시간이 들지 않으므로, 검증 시간의 대부분은 생성된 검증 조건을 해소하기 위한 SMT 질의가 차지합니다.

### 검증 실패 사례의 정성 분석

Mediator 백엔드로 검증되지 않은 44개 벤치마크를 수작업으로 검토한 결과, 그중 2개는 사실 VeriEQL로 반박된 것이어서 애초에 검증되어서는 안 되는 경우였습니다. 남은 42개 중 14개는 Mediator가 10분 시간 제한 안에 검증을 완료하지 못했고, 마지막 28개는 종료되었으나 "Unknown"을 반환했습니다. 실패 원인에 대한 통찰을 얻기 위해서는 Mediator가 두 질의 사이의 귀납적 쌍모의 불변식을 추론해야 한다는 점에 주목해야 합니다. [Mediator 연구](https://doi.org/10.1145/3158144)가 도입한 이 기법은 두 데이터베이스 상태 사이에 불변식 관계를 유지하는 방식으로 동치성을 증명하는데, 긴 조인 체인(join chain)을 포함하는 질의의 경우 대응하는 쌍모의 불변식이 매우 복잡해질 수 있습니다. 이는 SMT 질의를 비싸게 만들어 시간 초과를 일으키거나, 필요한 불변식이 Mediator의 추론 능력 범위를 벗어나게 만듭니다. 그럼에도 Mediator가 지원하는 196개의 SQL 질의 중 약 80%에 대해 (Cypher, SQL) 질의 쌍의 동치성을 증명할 수 있었다는 점은, 무계(unbounded) 검증이 가능한 강력한 보완책을 Graphiti가 갖추었음을 보여 줍니다.

여기서 두 백엔드의 상보적 관계가 분명히 드러납니다. VeriEQL은 반례 생성을 통한 반박(refutation)에 강하지만 유계 검증만 가능한 반면, Mediator는 반례를 만들지 못하지만 크기 제한 없는 완전한 증명(verification)이 가능합니다. 두 도구를 조합함으로써 Graphiti는 버그를 찾는 일과 동치성을 입증하는 일 양쪽 모두에서 실용적 성능을 끌어냅니다.

## 트랜스파일레이션 품질 평가

이 연구의 주된 목표는 그래프 질의와 관계형 질의의 동치성 검사이지만, 그 부산물로 그래프 데이터베이스 질의를 유도 스키마 위의 관계형 질의로 트랜스파일할 수 있다는 추가적 이점이 있습니다. 세 번째 실험은 이 트랜스파일레이션이 얼마나 효율적인 SQL을 만들어 내는지를 평가합니다.

먼저 트랜스파일 속도를 보면, Graphiti는 410개 질의 전부를 성공적으로 트랜스파일했으며 평균·중앙값·최댓값 트랜스파일 시간은 각각 6.3, 3.0, 180.2밀리초였습니다. 즉 트랜스파일레이션은 실용적으로 매우 빠릅니다. 이는 [OpenCypherTranspiler 같은 기존 도구가 410개 중 284개(69%)를 아예 지원하지 못하는 것](부록 비교 참조)과 대조되는, Graphiti의 건전성 보장을 동반한 완전한 커버리지를 나타냅니다.

다음으로 트랜스파일된 질의의 품질을 평가하기 위해, 수작업으로 작성된 SQL 질의와 Graphiti가 트랜스파일한 결과의 실행 시간을 비교했습니다. 그런데 이 평가에는 두 가지 어려움이 있습니다. 첫째, 원본 Cypher와 SQL 질의가 모두 주어진 "정답(ground truth)" 쌍은 일부 범주에만 존재합니다. 둘째, 이 질의들을 실제로 실행할 데이터베이스 인스턴스가 없습니다. 첫 번째 문제는 원본 Cypher와 그에 대응하는 SQL이 모두 주어진 StackOverflow, Tutorial, Academic 범주의 벤치마크로만 평가를 한정하여 해결했고, 두 번째 문제는 모의(mock) 데이터베이스 인스턴스를 생성하여 효율을 측정함으로써 해결했습니다.

구체적인 절차는 다음과 같습니다. 스키마 $\Psi\_R$ 위의 SQL 질의 $Q\_R$와 Cypher 질의 $Q\_G$를 가진 각 벤치마크에 대해, Graphiti로 $Q\_G$를 유도 스키마 $\Psi\_R'$ 위의 SQL 질의 $Q\_R'$로 트랜스파일합니다. 그런 다음 $\Psi\_R$와 $\Psi\_R'$ 위에 각각 데이터베이스 $R$과 $R'$을 생성하되, 잔여 변환기 관계 $\Phi\_{\mathsf{rdt}}(R') = R$이 성립하도록 보장합니다. 실행 시간의 변동성을 고려하기 위해 각 테이블에 1만 개의 튜플로 시작하여 10배씩 최대 100만 개까지 크기를 늘려 가며, 수작업 SQL 질의가 10초 안에 실행되는 가장 큰 크기를 선택합니다. 이 방식으로 36개 벤치마크는 100만 튜플, 나머지 9개는 1만에서 100만 사이의 크기를 갖게 되었습니다. 마지막으로 $Q\_R'$을 $R'$에서, $Q\_R$을 $R$에서 실행한 시간을 측정합니다.

| 데이터셋 | #쌍 | 트랜스파일 평균(초) | 수작업 평균(초) | 더 빠름 | (1x, 1.1x] | (1.1x, 1.2x] | (1.2x, ∞) |
|---|---|---|---|---|---|---|---|
| StackOverflow | 12 | 1.9 | 1.8 | 41.7% | 8.3% | 50.0% | 0.0% |
| Tutorial | 26 | 2.3 | 1.7 | 19.2% | 11.5% | 46.2% | 23.1% |
| Academic | 7 | 2.4 | 3.2 | 71.4% | 0.0% | 14.3% | 14.3% |
| Total | 45 | 2.2 | 2.0 | 33.3% | 8.9% | 42.2% | 15.6% |

위 표가 요약하듯, 트랜스파일된 질의는 33.3%의 벤치마크에서 수작업 작성 질의보다 더 빨랐습니다. 나머지 중 8.9%는 1.1배 이하의 미미한 속도 저하를, 42.2%는 1.1배에서 1.2배 사이의 저하를, 15.6%는 1.2배를 초과하는 저하를 보였습니다. 즉 절반 이상(51.1%)의 벤치마크에서 1.2배 이내의 속도 저하에 머물렀다는 의미입니다. 이러한 결과는 레거시 시스템, 자원이 제한된 환경, 데이터 통합처럼 그래프-관계형 질의 변환이 필요한 실제 상황에서 Graphiti의 자동 트랜스파일레이션이 충분히 실용적일 수 있음을 보여 줍니다. 트랜스파일레이션이 밀리초 단위로 이루어지면서도, 그 결과물이 사람이 손으로 작성한 질의에 견줄 만한 실행 효율을 갖춘다는 점이 이 실험의 핵심 발견입니다.
# 관련 연구

## SQL을 위한 자동 추론

Cypher와 SQL 질의 사이의 동치성 검증이라는 본 논문의 목표는, 그 마지막 단계에서 결국 관계형 질의들 사이의 동치성 검사로 귀결됩니다. 따라서 SQL 질의에 대한 자동 추론 연구가 가장 직접적인 토대를 이룹니다. 여기서 먼저 기억해 둘 근본적 사실은, 두 SQL 질의의 동치성을 일반적으로 판정하는 문제가 결정 불가능(undecidable)하다는 점입니다(Trakhtenbrot, 1950). 이 결정 불가능성 때문에 모든 질의를 항상 정확히 판정하는 완벽한 알고리즘은 존재할 수 없으며, 선행 연구들은 이 한계를 우회하기 위해 세 가지 서로 다른 전략을 택해 왔습니다.

첫 번째 부류는 SQL의 결정 가능한 부분집합(decidable subset)에 한정하여 그 안에서는 완전하게 동작하는 결정 절차(decision procedure)를 제안합니다(Aho et al., 1979; Chandra and Merlin, 1977; Green, 2011). 두 번째 부류는 결정 불가능한 더 넓은 부분집합을 대상으로, 건전하지만 불완전한(sound but incomplete) 알고리즘을 제시합니다. 즉 "동치"라고 답하면 그것은 항상 옳지만, 동치인데도 증명하지 못하는 경우가 있을 수 있습니다(Zhou et al., 2019, 2022; Chu et al., 2017b, 2018). 세 번째 부류는 유계 검증(bounded verification)을 수행하여 일정 크기 이하의 인스턴스 범위에서 SQL 질의의 버그를 찾아내는 데 집중하며, Cosette(Chu et al., 2017a), Qex(Veanes et al., 2010), 그리고 본 논문에서 백엔드로 활용된 VeriEQL(He et al., 2024)이 여기에 속합니다. 이 밖에 질의뿐 아니라 갱신(update)까지 포함하는 관계형 데이터베이스 애플리케이션을 검증하는 연구도 존재합니다(Wang et al., 2018). 본 논문이 그래프-관계형 질의 검증을 한 쌍의 관계형 질의 동치성 검사로 환원한다는 점은 전략적으로 중요한데, 이로써 위 영역에서 앞으로 이루어질 어떠한 발전도 그대로 흡수하여 활용할 수 있기 때문입니다.

## 데이터베이스 인스턴스 간 이전

서로 다른 스키마 사이에서 데이터를 이전(migration)하는 문제에 관한 선행 연구도 다수 존재합니다(Jin et al., 2017; Yaghmazadeh et al., 2016, 2018; Feng et al., 2017; Martins et al., 2019; Wang et al., 2017, 2020). 이 가운데 본 논문과 가장 밀접한 것은 그래프 데이터베이스와 관계형 데이터베이스 사이의 데이터 이전을 자동화하는 Dynamite(Wang et al., 2020)입니다. 그러나 Dynamite는 데이터베이스의 내용을 옮기는 데에만 유용하며, 질의 자체를 트랜스파일하는 데에는 사용할 수 없다는 한계가 있습니다. 또한 SQL 맥락에서 질의 트랜스파일 문제를 다룬 연구들도 있습니다(Wang et al., 2019; Curino et al., 2008, 2013). 본 논문에서 사용한 데이터베이스 변환기 개념은 Dynamite와 Miller et al.(2000)의 선행 연구에서 영감을 받았지만, Cypher에서 SQL로의 트랜스파일 절차를 형식화하고 이를 형식적 동치성 검사에 활용한 것은 본 논문이 처음입니다.

## 데이터 표현 리팩토링

프로그램이나 명세를 한 데이터 표현에서 다른 데이터 표현으로 리팩토링하는 데이터 표현 리팩토링(data representation refactoring)이라는 관련 연구 흐름도 있습니다(Pailoor et al., 2024, 2021; Chen et al., 2022; Cheung et al., 2013; Ge et al., 2012; Laddad et al., 2022; Wang et al., 2013; Rahmani et al., 2021; Delaware et al., 2015). 예컨대 Solidare(Pailoor et al., 2024)는 스마트 컨트랙트를 서로 다른 추상 데이터 타입(ADT) 사이에서 리팩토링하고, QBS(Chen et al., 2022)는 컬렉션을 다루는 Java 프로그램을 SQL 질의로 변환합니다. 그래프 스키마와 관계형 스키마를 서로 다른 데이터 표현으로 볼 수 있으므로, 본 논문의 질의 트랜스파일 문제 역시 일종의 데이터 표현 리팩토링으로 간주할 수 있습니다. 다만 기존 기법 중 그래프와 관계형 데이터 사이의 질의 트랜스파일 문제를 다룬 것은 없습니다. 더 나아가 본 논문은 이러한 리팩토링을 검증하는 새로운 방법론을 제시한다는 의의도 갖는데, 표현 $R$에서 $R'$로 직접 가는 대신 구문 지향적 번역을 가능하게 하는 보조 표현 $R''$를 도입하여 문제를 단순화한다는 점입니다. 이처럼 한 겹의 간접 계층(layer of indirection)을 두는 방법론은 이 맥락에서 이전에 탐구된 바가 없습니다.

## 그래프 데이터베이스 질의 언어

그래프 데이터베이스와 그래프 질의 언어의 의미론적 기초에 관한 오랜 연구 흐름이 존재합니다(Francis et al., 2018; Hartig and Pérez, 2018; van Rest et al., 2016; Deutsch et al., 2022; AGE, 2024; Rodriguez, 2015; Halpin and Cheney, 2011). 이들 연구를 관통하는 통합적 통찰은, 그래프 데이터베이스 스키마를 관계형 데이터베이스 스키마의 개체-관계 다이어그램(ER Diagram)을 그래프로 표현한 것으로 볼 수 있다는 점입니다. 이러한 공통 기반 덕분에 여러 그래프 질의 언어들은 의미론적으로나 구문적으로나 Cypher와 유사하며, 따라서 본 논문의 방법론은 Cypher 이외의 다른 그래프 질의 언어로도 비교적 쉽게 적응될 수 있을 것으로 기대됩니다.

## 데이터베이스 질의 테스팅

데이터베이스 질의를 테스트하는 연구도 관련 분야입니다. 여기에는 동일 입력에 대한 서로 다른 구현의 결과를 비교하는 차등 테스팅(differential testing)(McKeeman, 1998; Hua et al., 2023; Zheng et al., 2022)과, 입력을 변형해도 유지되어야 할 성질을 검사하는 메타모픽 테스팅(metamorphic testing)(Chen et al., 2020; Rigger and Su, 2020; Jiang et al., 2024; Mang et al., 2024)을 통해 데이터베이스 관리 시스템의 버그를 찾는 연구, SQL 질의가 포함된 프로그래밍 과제를 채점하기 위한 변이 기반 테스팅(Chandra et al., 2015), 그리고 잘못된 SQL 질의를 설명하기 위한 출처 기반(provenance-based) 기법(Miao et al., 2019)이 포함됩니다. 이와 달리 Graphiti는 그래프 질의를 동치인 SQL 질의로 트랜스파일한 뒤 기존 SQL 동치성 검사 도구를 활용하므로, 이러한 테스팅 연구와 상보적이며 그 발전으로부터 이득을 얻을 수 있습니다.

## Cypher 질의 트랜스파일

Cypher 질의를 SQL과 유사한 언어로 번역하는 기존 도구도 일부 존재합니다(Liang, 2025; Team, 2025). 본 논문과 가장 관련 깊은 것은 OpenCypherTranspiler(Liang, 2025)로, Cypher 질의를 먼저 논리적 계획(logical plan)으로 변환한 뒤 이를 관계형 질의로 표현합니다. 그러나 이 도구는 Cypher의 제한된 부분집합만 지원하며, 번역된 질의에 대한 건전성 보장이 없습니다. 또 다른 도구인 Kuzu(Team, 2025)는 Cypher 인터페이스를 통해 관계형 데이터베이스 위에서 그래프 질의를 실행할 수 있으며, Cypher 질의를 관계형 데이터베이스의 논리적 계획과 유사한 중간 표현으로 컴파일합니다. 다만 Kuzu는 Cypher를 SQL로 트랜스파일하지 않고 그 중간 표현을 데이터베이스 위에서 직접 실행한다는 점에서 본 논문과 다릅니다. 이에 반해 Graphiti는 트랜스파일 과정에서 건전성을 보장하며, 더 넓은 범위의 Cypher 질의를 지원합니다.
## 한계와 향후 연구 방향

논문은 제안된 기법이 강력하면서도 현재 범위에 명확한 경계가 있음을 솔직하게 인정합니다. Graphiti의 현재 버전은 SQL과 Cypher의 특정 부분 집합—즉 앞서 형식적 기반에서 정의한 Featherweight SQL과 Featherweight Cypher—에 초점을 맞추고 있으며, 현대 질의 언어가 제공하는 모든 기능을 아직 포괄하지는 못합니다. 가장 두드러진 예가 Cypher의 **가변 길이 패턴 매칭(variable-length pattern matching)** 입니다. 이는 `(a)-[*1..5]->(b)`처럼 길이가 사전에 고정되지 않은 경로, 즉 한 노드에서 다른 노드까지 임의의 단계 수를 거쳐 도달하는 패턴을 찾는 기능을 말합니다. 이러한 도달 가능성(reachability) 기반 질의는 본질적으로 재귀적 성격을 띠기 때문에, 논문이 다루는 핵심 SQL 조각—재귀를 포함하지 않는 관계 대수와 그 위의 집계·정렬—만으로는 표현할 수 없습니다. 앞서 그래프 질의 언어를 형식화할 때 `shortestPath`나 무제한 길이 경로 질의를 명시적으로 제외했던 것이 바로 이 한계의 근원입니다.

그럼에도 논문은 이러한 제한적 선택이 충분히 정당하다고 주장합니다. 그 근거는 두 가지입니다. 첫째, 그래프 질의와 관계형 질의 사이의 동치성을 추론하는 문제 자체가 이전까지 전혀 연구된 바 없는 미개척 영역이라는 점입니다. 일반적인 SQL 동치성 검사조차 결정 불가능(undecidable)하다는 근본적 한계를 안고 있는 상황에서, 두 이질적 데이터 모델을 잇는 추론을 처음으로 시도하려면 다루기 쉬우면서도 표현력 있는 핵심 조각부터 단단히 다지는 것이 합리적인 출발점이 됩니다. 즉 선택된 단편은 "모든 것을 한 번에"가 아니라 "확실히 올바른 토대를 먼저"라는 전략의 산물입니다. 둘째, 실증적 근거가 이 선택을 뒷받침합니다. 앞서 살펴본 평가에서 StackOverflow, 튜토리얼, 학술 논문, 그리고 실제 현장에서 수집된 다양한 벤치마크에 대해 Graphiti가 의미 있는 결과를 냈다는 사실은, 비록 일부 SQL·Cypher 질의가 현재 범위 밖에 있더라도 선택된 부분 집합이 실용적 사용 사례를 다루기에 충분히 표현력 있음을 보여 줍니다.

향후 연구는 트랜스파일레이션 규칙과 백엔드 동치성 검증기를 확장하여 추가 기능을 지원하는 방향으로 자연스럽게 이어질 수 있습니다. 여기서 주목할 점은 Graphiti의 모듈식 설계가 이러한 확장을 구조적으로 뒷받침한다는 것입니다. 앞서 강조했듯 전체 정확성이 백엔드 SQL 검사기의 능력에 위임되어 있으므로, SQL 동치성 검증 분야에서 이루어지는 어떠한 발전—예컨대 재귀 질의나 윈도우 함수를 다룰 수 있는 새로운 검사기의 등장—도 그대로 흡수하여 활용할 수 있습니다.

## 결론

논문은 그래프 데이터베이스 질의와 관계형 데이터베이스 질의 사이의 자동 추론 기법을 제안했다는 점에서 분명한 의의를 가집니다. 그 기여는 크게 세 갈래로 정리됩니다. 먼저 그래프 질의와 관계형 질의 사이의 동치성을 형식적으로 정의하고, 이를 토대로 Cypher 질의를 이른바 **유도 관계형 스키마(induced relational schema)** 위의 SQL 질의로 변환하는 구성에 의한 올바름(correct-by-construction)이 보장되는 트랜스파일레이션 전략을 제시했습니다. 이어서 이 번역 접근법을 활용하면, SQL을 위해 이미 개발되어 온 자동 추론 기법을 그대로 끌어와 임의의 스키마 위에서 그래프 질의와 SQL 질의의 동치성을 검사할 수 있음을 보였습니다. 이는 근본적으로 다른 두 데이터 모델을 동시에 추론해야 하는 난제를, 잘 연구된 동질적 SQL 동치성 문제로 환원하는 우아한 전략의 결실입니다.

구현체인 Graphiti에 대한 평가는 이 기법이 실제 현장에서 수집된 Cypher와 SQL 질의를 대상으로 두 가지 측면에서 유용함을 입증했습니다. 하나는 참조 SQL 구현과 동치이도록 의도된 Cypher 질의에 숨어 있는 미묘한 버그를 들춰내는 능력이고, 다른 하나는 Cypher와 SQL 질의 사이의 완전한 동치성을 검증하는 능력입니다. 다시 말해 Graphiti는 "동치임을 증명하는" 검증(verification)과 "동치가 아님을 밝히는" 반박(refutation) 양쪽에서 모두 실용적 가치를 지니며, 특히 사람이 작성한 전문가용 자료에까지 잠복해 있던 번역 오류를 자동으로 찾아낼 수 있다는 점이 인상적입니다.

향후 연구로 논문은 그래프 데이터베이스와 관계형 데이터베이스 사이의 데이터베이스 변환기를 명세하기 위한 **그래픽 인터페이스(graphical interface)** 의 개발을 구상합니다. 이는 스키마 매핑 시각화에 관한 선행 연구, 즉 [Robertson et al.](https://doi.org/10.1145/1054972.1055032)이 제안한 이질적 스키마 간 대응 관계를 직관적이고 사용자 친화적인 형태로 표현하는 시각화 기법에서 영감을 받은 것입니다. 이 선행 연구는 복잡한 스키마 매핑을 시각적으로 표현함으로써 매핑 명세 작업의 인지적 부담을 줄이는 데 초점을 맞추었는데, Graphiti의 맥락에서는 사용자가 변환기를 직접 작성해야 하는 수작업 부담을 덜어 주는 인터페이스 계층으로 발전시킬 수 있습니다. 앞선 평가에서 변환기 하나가 평균 약 5.9개 규칙·약 1분의 작성 시간으로 충분히 다룰 만한 수준이긴 했지만, 시각적 명세 도구가 더해진다면 동치성 검증을 위해 사용자가 들이는 수작업 노력을 한층 더 줄여 더 직관적이고 접근성 높은 경험을 제공할 수 있을 것입니다. 다만 시각화 자체는 형식적 검증을 대체하지 못하며, 어디까지나 변환기 명세 단계를 보완하는 도구로 자리매김한다는 점에서, Graphiti의 핵심인 자동 추론 파이프라인과 상보적 관계에 놓입니다. 논문은 이 방향을 앞으로의 유망한 연구 영역으로 전망하며 마무리됩니다.
# 부록: Featherweight Cypher의 표시적 의미론

앞서 형식적 기반을 다룰 때 Featherweight Cypher의 문법을 소개하고 그 표시적 의미론(denotational semantics)이 부록에서 형식적으로 기술된다고 언급했습니다. 이 부록은 바로 그 약속을 이행하는 부분으로, 질의·반환 질의·절·경로 패턴·표현식·술어 각각이 입력 그래프에 대해 정확히 어떤 값을 계산하는지를 함수형 조합자(functional combinator)를 사용해 빈틈없이 정의합니다. 여기서 핵심이 되는 조합자는 리스트의 각 원소에 함수를 적용하는 $\mathsf{map}$, 조건을 만족하는 원소만 남기는 $\mathsf{filter}$, 누적값을 왼쪽부터 접어 나가는 $\mathsf{foldl}$, 리스트의 첫 원소를 꺼내는 $\mathsf{head}$, 두 리스트를 짝지어 묶는 $\mathsf{zip}$입니다. 이들은 함수형 프로그래밍에서 흔히 쓰이는 고차 함수로, 의미론을 간결하고 수학적으로 엄밀하게 표현하기 위한 도구입니다. 한 가지 유념할 점은 이 정의들이 실행을 위한 구현 지침이 아니라 정확성을 증명하기 위한 명세(specification) 수준의 정의라는 사실입니다. 따라서 효율보다는 의미의 명료함이 우선시됩니다.

본격적인 정의에 앞서 보조 함수들의 역할을 짚어 두면 이해가 한결 수월합니다. $\mathsf{ite}(\phi, E\_1, E\_2)$는 조건 $\phi$가 참이면 $E\_1$, 아니면 $E\_2$로 평가되는 if-then-else 표현식입니다. $\mathsf{dedup}(L)$은 리스트 $L$에서 중복 원소를 모두 제거하고, $\mathsf{hasAgg}(\overline{E})$는 표현식 리스트 $\overline{E}$ 안에 집계 표현식이 있는지를 검사합니다. $\mathsf{merge}(G\_1, G\_2)$는 두 그래프를 하나로 합치며, $\mathsf{Nodes}(G)$는 그래프 $G$의 모든 노드를 돌려줍니다. $\mathsf{Nullify}(G)$는 $G$와 동형(isomorphic)이되 모든 값이 $\mathsf{Null}$로 설정된 그래프를 만들어 내고, $\mathsf{Subgraphs}(G, [NP\_1, EP, NP\_2])$는 단순 패턴 $NP\_1, EP, NP\_2$에 매칭되는 $G$의 모든 부분 그래프를 반환합니다. 특히 이 $\mathsf{Subgraphs}$ 함수는 한 홉(single-hop)짜리 단순 패턴에 매칭되는 부분 그래프를 찾아 주는 원자적 구성 요소로, 부록에서 그 구체적 정의는 암묵적으로 남겨 두지만 모든 재귀적 경로 매칭이 이 함수 위에 쌓여 올라간다는 점에서 매우 중요합니다.

## 질의의 의미론: 그래프에서 테이블로

가장 바깥 계층인 질의 $Q$의 의미론은 타입 $\llbracket Q\rrbracket :: \mathsf{Graph} \to \mathsf{Table}$을 가집니다. 즉 질의는 속성 그래프를 입력받아 테이블을 산출하는 함수로 해석되며, 이는 앞서 형식적 기반에서 질의를 "그래프 데이터베이스 인스턴스로부터 테이블로 가는 함수"로 본 관점을 그대로 구현한 것입니다.

정렬을 담당하는 $\mathsf{OrderBy}$의 의미론은 다음과 같이 정의됩니다.

$$
\llbracket\mathsf{OrderBy}(R,k,b)\rrbracket\_G = \mathsf{foldl}(\lambda xs.\lambda \_.\, xs \mathbin{+\!+} [\mathsf{MinTuple}(E,b,\llbracket R\rrbracket\_G - xs)],\ [],\ \llbracket R\rrbracket\_G)
$$

여기서 보조 함수 $\mathsf{MinTuple}$과 비교 함수 $\mathsf{Cmp}$는 다음과 같이 정의됩니다.

$$
\mathsf{MinTuple}(E,b,xs) = \mathsf{foldl}(\lambda x.\lambda y.\mathsf{ite}(\mathsf{Cmp}(E,b,x,y),x,y),\ \mathsf{head}(xs),\ xs)
$$

$$
\mathsf{Cmp}(E,b,x\_1,x\_2) = (\mathsf{lookup}(x\_1,k) < \mathsf{lookup}(x\_2,k)) = b
$$

이 정의를 직관적으로 풀어 보면, $\mathsf{OrderBy}$는 일종의 **선택 정렬(selection sort)** 처럼 동작합니다. 매 단계마다 아직 결과 리스트 $xs$에 들어가지 않은 튜플들($\llbracket R\rrbracket\_G - xs$) 가운데 정렬 기준에서 "가장 앞서는" 튜플 하나를 $\mathsf{MinTuple}$로 골라내어 결과 끝에 덧붙이는 일을 반복합니다. $\mathsf{MinTuple}$ 자체도 $\mathsf{foldl}$로 리스트를 훑으며 비교 함수 $\mathsf{Cmp}$에 따라 더 앞서는 원소를 계속 유지합니다. 비교 함수 $\mathsf{Cmp}$는 키 $k$에 대한 두 튜플의 값을 비교하되, 불리언 플래그 $b$가 오름차순·내림차순을 결정합니다. 즉 $(\mathsf{lookup}(x\_1,k) < \mathsf{lookup}(x\_2,k))$의 결과가 $b$와 같은지를 따져 정렬 방향을 조절하는 것입니다. 이러한 $\mathsf{foldl}$ 기반 정의는 개념적으로 $O(n^2)$의 비교를 수행하지만, 이는 어디까지나 정확성을 증명하기 위한 명세적 정의이지 실제 구현이 그렇게 동작해야 한다는 뜻은 아닙니다.

합집합 연산의 의미론은 SQL과 유사하게 정의됩니다.

$$
\llbracket\mathsf{Union}(Q\_1,Q\_2)\rrbracket\_G = \mathsf{dedup}(\llbracket\mathsf{UnionAll}(Q\_1,Q\_2)\rrbracket\_G)
$$

$$
\llbracket\mathsf{UnionAll}(Q\_1,Q\_2)\rrbracket\_G = \llbracket Q\_1\rrbracket\_G \mathbin{+\!+} \llbracket Q\_2\rrbracket\_G
$$

즉 $\mathsf{UnionAll}$은 두 질의 결과 리스트를 단순히 이어 붙이는($+\!+$) 백(bag) 합집합이고, $\mathsf{Union}$은 그 결과에 $\mathsf{dedup}$을 적용하여 중복을 제거한 집합 합집합입니다. 이는 앞서 합집합 연산이 SQL의 대응물과 유사하다고 설명한 부분의 형식적 근거가 됩니다.

## 반환 질의의 의미론: 집계 유무에 따른 분기

반환 질의 $R$의 의미론 역시 타입 $\llbracket R\rrbracket :: \mathsf{Graph} \to \mathsf{Table}$을 가지며, $\mathsf{Return}(C, \overline{E}, \overline{k})$의 핵심은 집계 표현식의 유무에 따라 두 갈래로 갈린다는 점입니다.

$$
\llbracket\mathsf{Return}(C,\overline{E},\overline{k})\rrbracket\_G = \mathsf{ite}\big(\neg\mathsf{hasAgg}(\overline{E}),\ \mathsf{map}(\lambda g.\mathsf{map}(\lambda(E,k).(k,\llbracket E\rrbracket\_{G,[g]}),\mathsf{zip}(\overline{E},\overline{k})),\llbracket C\rrbracket\_G),\ \ldots\big)
$$

집계가 없는 경우($\neg\mathsf{hasAgg}(\overline{E})$)에는, 절 $C$를 평가해 얻은 부분 그래프 리스트 $\llbracket C\rrbracket\_G$의 각 부분 그래프 $g$에 대해 표현식 리스트 $\overline{E}$를 평가하여 하나의 행을 만듭니다. $\mathsf{zip}(\overline{E}, \overline{k})$로 각 표현식과 열 이름을 짝지은 뒤, 각 쌍 $(E, k)$를 평가 결과 $\llbracket E\rrbracket\_{G,[g]}$와 열 이름 $k$의 쌍으로 변환하는 것입니다. 여기서 표현식이 단일 부분 그래프 $[g]$ 위에서 평가된다는 점에 주목할 필요가 있습니다.

반면 집계가 있는 경우의 의미론은 그룹화를 도입합니다.

$$
\mathsf{map}(\lambda gs.\mathsf{map}(\lambda(E,k),(k,\llbracket E\rrbracket\_{G,gs}),\mathsf{zip}(\overline{E},\overline{k})),\mathsf{Groups})
$$

이때 그룹들의 집합 $\mathsf{Groups}$는 다음과 같이 정의됩니다.

$$
\mathsf{Groups} = \mathsf{map}(\lambda V.\mathsf{filter}(\lambda g.\llbracket\overline{A}\rrbracket\_{G,[g]}=V,\llbracket C\rrbracket\_G),\mathsf{dedup}(\overline{V}))
$$

$$
\overline{A} = \mathsf{filter}(\lambda E.\neg\mathsf{IsAgg}(E),\overline{E}), \qquad \overline{V} = \mathsf{map}(\lambda g.\llbracket\overline{A}\rrbracket\_{G,[g]},\llbracket C\rrbracket\_G)
$$

이 정의가 말하는 바를 단계적으로 풀어 보면 이렇습니다. 먼저 $\overline{A}$는 표현식 리스트 $\overline{E}$ 가운데 집계가 아닌($\neg\mathsf{IsAgg}$) 표현식들만 걸러낸 것으로, 곧 그룹화 키(grouping key) 역할을 합니다. 이는 앞서 트랜스파일레이션의 Q-Agg 규칙에서 "집계가 아닌 표현식을 그룹화 키로 자동 식별한다"고 설명한 부분과 정확히 일치합니다. 다음으로 $\overline{V}$는 각 부분 그래프에 대해 이 그룹화 키들을 평가한 값들의 리스트이고, $\mathsf{dedup}(\overline{V})$로 서로 다른 키 값들 $V$를 추려냅니다. 마지막으로 각 고유한 키 값 $V$마다 그 값을 갖는 부분 그래프들을 $\mathsf{filter}$로 모아 하나의 그룹 $gs$를 형성합니다. 이렇게 만들어진 각 그룹 $gs$ 전체에 대해 표현식을 평가($\llbracket E\rrbracket\_{G,gs}$)하므로, 집계 표현식이 그룹 단위로 올바르게 계산될 수 있습니다. 집계가 없을 때는 단일 부분 그래프 $[g]$ 위에서, 집계가 있을 때는 그룹 $gs$ 위에서 표현식을 평가한다는 대비가 이 정의의 핵심입니다.

## 절의 의미론: 그래프에서 부분 그래프 리스트로

절 $C$의 의미론은 타입 $\llbracket C\rrbracket :: \mathsf{Graph} \to [\mathsf{Graph}]$를 가집니다. 즉 절은 입력 그래프를 부분 그래프들의 리스트로 변환하며, 각 부분 그래프는 입력 그래프에 대한 패턴 매칭의 결과에 대응합니다.

단일 매칭과 연쇄 매칭의 의미론은 다음과 같습니다.

$$
\llbracket\mathsf{Match}(PP,\phi)\rrbracket\_G = \mathsf{filter}(\lambda g.\llbracket\phi\rrbracket\_{G,[g]}=\top,\llbracket PP\rrbracket\_G)
$$

$$
\llbracket\mathsf{Match}(C,PP,\phi)\rrbracket\_G = \mathsf{filter}(\lambda g.\llbracket\phi\rrbracket\_{G,[g]}=\top,\mathsf{map}(\lambda g\_1.\mathsf{map}(\lambda g\_2.\mathsf{merge}(g\_1,g\_2),\llbracket PP\rrbracket\_G),\llbracket C\rrbracket\_G))
$$

단일 매칭은 경로 패턴 $PP$에 매칭되는 부분 그래프들 가운데 술어 $\phi$를 만족하는($\llbracket\phi\rrbracket\_{G,[g]} = \top$) 것만 $\mathsf{filter}$로 남깁니다. 연쇄 매칭은 선행 절 $C$의 결과 부분 그래프 $g\_1$들과 경로 패턴 $PP$의 결과 $g\_2$들을 $\mathsf{merge}$로 모두 결합한 뒤 같은 술어 필터를 적용합니다. 이렇게 모든 조합을 병합하는 동작이 바로 앞서 트랜스파일레이션에서 연쇄 매칭이 내부 조인(inner join)으로 번역되던 의미론적 근거입니다.

선택적 매칭 $\mathsf{OptMatch}$의 의미론은 가장 미묘하므로 천천히 살펴볼 가치가 있습니다.

$$
\llbracket\mathsf{OptMatch}(C,PP,\phi)\rrbracket\_G = \mathsf{foldl}(\lambda gs.\lambda g.gs \mathbin{+\!+} \mathsf{ite}(|v\_1(g)|=0,v\_2(g),v\_1(g)),\ [],\ \llbracket C\rrbracket\_G)
$$

여기서 두 보조 함수 $v\_1$과 $v\_2$가 정의됩니다.

$$
v\_1(g) = \mathsf{filter}(\lambda x.\llbracket\phi\rrbracket\_{G,[x]}=\top,\ \mathsf{map}(\lambda g'.\mathsf{merge}(g,g'),\mathsf{filter}(\lambda g''.g\cap g'' \neq \emptyset,\llbracket PP\rrbracket\_G)))
$$

$$
v\_2(g) = [\mathsf{merge}(g,\mathsf{Nullify}(\mathsf{head}(\llbracket PP\rrbracket\_G)))]
$$

이 정의의 직관은 이렇습니다. 선행 절 $C$의 각 부분 그래프 $g$에 대해, 먼저 $v\_1(g)$가 패턴 $PP$를 매칭해 $g$와 교집합이 비어 있지 않은($g \cap g'' \neq \emptyset$, 즉 공유하는 노드/엣지가 있는) 부분 그래프들을 찾아 $g$와 병합하고 술어 $\phi$로 거릅니다. 만약 이렇게 매칭된 결과가 하나라도 존재하면($|v\_1(g)| \neq 0$) 그것을 그대로 사용합니다. 그러나 매칭이 전혀 없으면($|v\_1(g)| = 0$) $v\_2(g)$로 넘어가, 패턴의 한 부분 그래프를 $\mathsf{Nullify}$로 모든 값을 $\mathsf{Null}$로 만든 뒤 $g$와 병합한 결과를 대신 사용합니다. 바로 이 "매칭 실패 시 $\mathsf{Null}$로 채운다"는 동작이 SQL의 좌외부 조인(left outer join) 의미론과 정확히 대응하며, 앞서 외부 조인 연산자와 삼치 논리가 신중히 도입된 이유를 형식적으로 뒷받침합니다.

$\mathsf{With}$ 절의 의미론은 이름 변경을 수행합니다.

$$
\llbracket\mathsf{With}(C,\overline{X},\overline{Y})\rrbracket\_G = \mathsf{map}(\lambda(N,E,P,T).(N,E,P[\overline{X}\mapsto\overline{Y}],T[\overline{X}\mapsto\overline{Y}]),\llbracket C\rrbracket\_G)
$$

각 부분 그래프 $(N, E, P, T)$에 대해 속성 함수 $P$와 타입 함수 $T$ 안의 변수 이름 $\overline{X}$를 $\overline{Y}$로 치환($P[\overline{X}\mapsto\overline{Y}]$)하는 것으로, 앞서 $\mathsf{With}$가 변수 스코프를 재설정하는 역할을 한다고 설명한 부분의 구체적 실현입니다.

### OptMatch의 동작을 보여 주는 구체적 예제

앞서 정의한 $v\_1$, $v\_2$의 동작을 손에 잡히게 보여 주는 것이 Example A.1입니다. 직원 A는 CS 부서에 연결된 `:WORK_AT` 엣지가 존재하지만, 직원 B에는 대응하는 부서 엣지가 없는 그래프 인스턴스를 고려합니다. 이 인스턴스에 대해 다음과 같은 Cypher 질의를 실행한다고 합시다.

$$
\mathsf{Return}(\mathsf{OptMatch}(\mathsf{Match}([(\mathsf{n},\mathsf{EMP})],\top),\ [(\mathsf{n},\mathsf{EMP}),(\mathsf{e},\mathsf{WORK\_AT},\rightarrow),(\mathsf{m},\mathsf{DEPT})],\top),\ [\mathtt{n.name},\mathtt{m.dname}],[\mathtt{n.name},\mathtt{m.dname}])
$$

이 질의를 실행하면 직원 이름과 부서 이름을 담은 테이블이 산출되는데, A 행에는 부서 `CS`가 채워지지만 B 행의 `dname` 항목은 $\mathsf{Null}$이 됩니다. 그 이유가 바로 $\mathsf{OptMatch}$의 의미론에 있습니다. 직원 B를 나타내는 노드에서는 나가는 `WORK_AT` 엣지가 없으므로 패턴 $[(\mathsf{n},\mathsf{EMP}),(\mathsf{e},\mathsf{WORK\_AT},\rightarrow),(\mathsf{m},\mathsf{DEPT})]$에 대한 매칭 $v\_1(g)$가 비어 있게 됩니다. 따라서 $|v\_1(g)| = 0$ 분기가 작동하여 $v\_2(g)$, 즉 $\mathsf{Nullify}$로 부서 부분을 $\mathsf{Null}$로 채운 부분 그래프가 결과에 포함됩니다. 만약 일반 $\mathsf{Match}$였다면 B에 대한 행은 아예 사라졌을 것이므로, 이 예제는 선택적 매칭이 어떻게 "매칭이 없어도 행을 보존하되 빈 부분을 $\mathsf{Null}$로 채우는" 외부 조인 동작을 구현하는지를 선명하게 드러냅니다.

## 경로 패턴의 의미론

경로 패턴 $PP$의 의미론은 타입 $\llbracket PP\rrbracket :: \mathsf{Graph} \to [\mathsf{Graph}]$를 가지며, 노드 패턴과 확장된 경로 패턴 두 형태에 대해 정의됩니다.

단일 노드 패턴 $NP$의 의미론은 다음과 같습니다.

$$
\llbracket NP\rrbracket\_G = \mathsf{map}(\lambda n.(\{n\},\emptyset,\{(X,k)\mapsto P(n,k)\mid k\in\mathsf{keys}(T(n))\},\{X\mapsto(l,\mathsf{keys}(T(n)))\}),\ [n\in N\mid \mathsf{label}(T(n))=l])
$$

여기서 $(N,E,P,T) = G$이고 $(X,l) = NP$입니다. 직관적으로 이는 그래프 $G$의 노드 가운데 레이블이 $l$인 노드 $n$들을 모두 골라($\mathsf{label}(T(n)) = l$), 각각을 단일 노드 부분 그래프 $(\{n\}, \emptyset, \ldots)$로 만드는 것입니다. 이때 노드 집합은 $\{n\}$ 하나뿐이고 엣지 집합은 비어 있으며($\emptyset$), 속성 함수는 변수 $X$와 키 $k$의 쌍을 원래 값 $P(n,k)$로 대응시키고, 타입 함수는 $X$를 레이블 $l$과 키 집합으로 사상합니다. 결과적으로 노드 패턴 매칭의 출력은 타입 $l$의 노드를 하나씩 담은 단일 노드 부분 그래프들의 리스트가 됩니다.

확장된 경로 패턴 $NP, EP, PP'$의 의미론은 귀납적으로 정의됩니다.

$$
\llbracket NP,EP,PP'\rrbracket\_G = \mathsf{foldl}(\lambda gs.\lambda g'.gs \mathbin{+\!+} \mathsf{map}(\lambda g.\mathsf{merge}(g,g'),\mathsf{filter}(\lambda g''.g'\cap g'' \neq \emptyset,\mathsf{Subgraphs}(G,[NP,EP,NP']))),\ [],\ \llbracket PP'\rrbracket\_G)
$$

여기서 $NP' = \mathsf{head}(PP')$입니다. 이 정의는 나머지 경로 패턴 $PP'$를 먼저 평가하여 얻은 각 부분 그래프 $g'$에 대해, 그 $PP'$의 머리 노드 $NP'$를 연결점으로 삼아 단순 패턴 $[NP, EP, NP']$에 매칭되는 부분 그래프들을 $\mathsf{Subgraphs}$로 찾고, $g'$와 노드/엣지를 공유하는($g' \cap g'' \neq \emptyset$) 것들만 골라 $g'$와 병합합니다. 이렇게 $\mathsf{Subgraphs}$라는 원자적 단순 패턴 매칭 위에 $\mathsf{foldl}$로 한 홉씩 쌓아 올려 긴 경로 패턴 전체를 귀납적으로 구성하는 것이 핵심입니다.
## 표현식의 의미론: 부분 그래프 리스트에서 값으로

표현식 $E$의 의미론은 앞선 두 계층과 달리 타입 $\llbracket E\rrbracket :: \mathsf{Graph} \to [\mathsf{Graph}] \to \mathsf{Value}$를 가집니다. 즉 입력 그래프 $G$와 더불어 **부분 그래프들의 리스트** $gs$를 추가 인자로 받아 하나의 값을 계산합니다. 여기서 $gs$는 경로 패턴 매칭의 결과를 나타내며, 표현식은 이 매칭 결과 위에서 평가됩니다. 가장 단순한 두 기저 사례는 다음과 같습니다.

$$
\llbracket k\rrbracket\_{G,gs} = \mathsf{lookup}(\mathsf{head}(gs),k) \qquad \llbracket v\rrbracket\_{G,gs} = v
$$

속성 키 $k$는 매칭된 부분 그래프 리스트의 첫 번째 원소($\mathsf{head}(gs)$)에서 그 키의 값을 조회하여 평가됩니다. 여기서 의미론이 보장하는 한 가지 불변식이 중요합니다. 속성 키 $\llbracket k\rrbracket$에 전달되는 $gs$는 항상 길이가 1인 리스트라는 점인데, 따라서 $\mathsf{head}(gs)$로 단일 부분 그래프를 안전하게 꺼낼 수 있습니다. 값 상수 $v$는 문맥과 무관하게 그 자신으로 평가됩니다.

술어를 값으로 변환하는 $\mathsf{Cast}(\phi)$의 의미론은 삼치 논리를 그대로 반영합니다.

$$
\llbracket\mathsf{Cast}(\phi)\rrbracket\_{G,gs} = \mathsf{ite}(\llbracket\phi\rrbracket\_{G,gs}=\mathsf{Null},\mathsf{Null},\mathsf{ite}(\llbracket\phi\rrbracket\_{G,gs}=\top,1,0))
$$

즉 술어 $\phi$가 $\mathsf{Null}$로 평가되면 결과도 $\mathsf{Null}$이고, 참이면 $1$, 거짓이면 $0$으로 변환됩니다. 술어가 가질 수 있는 세 가지 값 각각에 대응하는 캐스팅 결과가 정확히 정의되어 있음을 볼 수 있습니다.

### 집계 연산자의 의미론과 Null 처리

집계 연산자들은 SQL과 유사하되, $\mathsf{Null}$을 다루는 방식이 핵심입니다. 먼저 $\mathsf{Count}$의 의미론을 봅시다.

$$
\llbracket\mathsf{Count}(E)\rrbracket\_{G,gs} = \mathsf{ite}\Big(\bigwedge\_{g\in gs}\llbracket E\rrbracket\_{G,[g]}=\mathsf{Null},\ \mathsf{Null},\ \mathsf{foldl}(+,0,\mathsf{map}(\lambda g.\mathsf{ite}(\llbracket E\rrbracket\_{G,[g]}=\mathsf{Null},0,1),gs))\Big)
$$

이 정의는 두 단계로 작동합니다. 먼저 그룹 $gs$의 **모든** 원소에 대해 $E$가 $\mathsf{Null}$로 평가되면($\bigwedge\_{g\in gs}\llbracket E\rrbracket\_{G,[g]}=\mathsf{Null}$) 전체 결과를 $\mathsf{Null}$로 만듭니다. 그렇지 않으면 각 부분 그래프 $g$에 대해 $E$가 $\mathsf{Null}$이면 $0$을, 아니면 $1$을 부여한 뒤 $\mathsf{foldl}(+, 0, \cdots)$으로 모두 더합니다. 즉 $\mathsf{Null}$이 아닌 값의 개수만 세는 것으로, SQL의 `COUNT`가 `NULL`을 무시하는 동작과 정확히 일치합니다.

$\mathsf{Sum}$도 동일한 골격을 따릅니다.

$$
\llbracket\mathsf{Sum}(E)\rrbracket\_{G,gs} = \mathsf{ite}\Big(\bigwedge\_{g\in gs}\llbracket E\rrbracket\_{G,[g]}=\mathsf{Null},\ \mathsf{Null},\ \mathsf{foldl}(+,0,\mathsf{map}(\lambda g.\mathsf{ite}(\llbracket E\rrbracket\_{G,[g]}=\mathsf{Null},0,\llbracket E\rrbracket\_{G,[g]}),gs))\Big)
$$

$\mathsf{Count}$와의 유일한 차이는 $\mathsf{Null}$이 아닌 원소에 대해 $1$ 대신 그 실제 값 $\llbracket E\rrbracket\_{G,[g]}$를 더한다는 점입니다. $\mathsf{Null}$ 원소는 $0$으로 치환되어 합에 기여하지 않습니다.

평균 $\mathsf{Avg}$는 위 두 연산을 조합하여 간결하게 정의됩니다.

$$
\llbracket\mathsf{Avg}(E)\rrbracket\_{G,gs} = \llbracket\mathsf{Sum}(E)\rrbracket\_{G,gs} / \llbracket\mathsf{Count}(E)\rrbracket\_{G,gs}
$$

즉 합을 개수로 나눈 것이며, $\mathsf{Null}$을 제외한 값들의 평균이 자연스럽게 얻어집니다.

최솟값과 최댓값은 $\mathsf{foldl}$의 초깃값과 누적 함수만 바꾸어 정의됩니다.

$$
\llbracket\mathsf{Min}(E)\rrbracket\_{G,gs} = \mathsf{ite}\Big(\bigwedge\_{g\in gs}\llbracket E\rrbracket\_{G,[g]}=\mathsf{Null},\ \mathsf{Null},\ \mathsf{foldl}(\mathbf{min},+\infty,\mathsf{map}(\lambda g.\mathsf{ite}(\llbracket E\rrbracket\_{G,[g]}=\mathsf{Null},+\infty,\llbracket E\rrbracket\_{G,[g]}),gs))\Big)
$$

$$
\llbracket\mathsf{Max}(E)\rrbracket\_{G,gs} = \mathsf{ite}\Big(\bigwedge\_{g\in gs}\llbracket E\rrbracket\_{G,[g]}=\mathsf{Null},\ \mathsf{Null},\ \mathsf{foldl}(\mathbf{max},-\infty,\mathsf{map}(\lambda g.\mathsf{ite}(\llbracket E\rrbracket\_{G,[g]}=\mathsf{Null},-\infty,\llbracket E\rrbracket\_{G,[g]}),gs))\Big)
$$

$\mathsf{Min}$은 초깃값으로 $+\infty$를 두고 $\mathsf{Null}$ 원소를 $+\infty$로 치환하여 최솟값 계산에서 자연스럽게 배제하며, $\mathsf{Max}$는 대칭적으로 $-\infty$를 사용합니다. 이렇게 "중립 원소(neutral element)"를 $\mathsf{Null}$의 자리에 채우는 기법은 $\mathsf{Null}$이 결과에 영향을 주지 않도록 하는 함수형 정의의 우아한 관용구입니다.

마지막으로 산술 연산은 다음과 같이 정의됩니다.

$$
\llbracket E\_1 \oplus E\_2\rrbracket\_{G,gs} = \llbracket E\_1\rrbracket\_{G,gs} \oplus \llbracket E\_1\rrbracket\_{G,gs}
$$

여기서 한 가지 짚어 둘 점이 있습니다. 원문의 우변에는 $E\_1$이 두 번 등장하는데, 이는 명백히 $E\_2$의 오식으로 보입니다. 의도된 정의는 $\llbracket E\_1\rrbracket\_{G,gs} \oplus \llbracket E\_2\rrbracket\_{G,gs}$, 즉 두 피연산자 표현식을 각각 평가한 뒤 산술 연산자 $\oplus$를 적용하는 것입니다. 원문을 직접 참조하는 독자가 혼동하지 않도록 이 점을 명시해 둡니다.

## 술어의 의미론과 삼치 논리

술어 $\phi$의 의미론은 타입 $\llbracket\phi\rrbracket :: \mathsf{Graph} \to [\mathsf{Graph}] \to \mathsf{Bool} \cup \{\mathsf{Null}\}$을 가집니다. 반환 타입이 단순한 불리언이 아니라 $\mathsf{Bool} \cup \{\mathsf{Null}\}$이라는 점이 핵심으로, 이것이 바로 앞서 강조한 **삼치 논리(three-valued logic)**의 형식적 토대입니다. 가장 단순한 기저 사례부터 보면, 참과 거짓은 문맥과 무관하게 자기 자신으로 평가됩니다.

$$
\llbracket\top\rrbracket\_{G,gs} = \top \qquad \llbracket\bot\rrbracket\_{G,gs} = \bot
$$

비교 술어와 $\mathsf{IsNull}$은 다음과 같습니다.

$$
\llbracket E\_1 \odot E\_2\rrbracket\_{G,gs} = \llbracket E\_1\rrbracket\_{G,gs} \odot \llbracket E\_1\rrbracket\_{G,gs}
$$

$$
\llbracket\mathsf{IsNull}(E)\rrbracket\_{G,gs} = \mathsf{ite}(\llbracket E\rrbracket\_{G,gs}=\mathsf{Null},\top,\bot)
$$

(여기서도 비교 술어 우변의 $E\_1$ 중복은 앞선 산술 연산과 마찬가지로 $E\_2$의 오식으로 읽는 것이 자연스러우며, 의도된 의미는 두 표현식 $E\_1$, $E\_2$를 평가해 비교 연산자 $\odot$로 견주는 것입니다.) $\mathsf{IsNull}(E)$는 표현식 $E$가 $\mathsf{Null}$로 평가되면 $\top$을, 아니면 $\bot$을 돌려주어 $\mathsf{Null}$ 여부를 명시적으로 검사하는 통로를 제공합니다.

멤버십 술어와 존재 술어는 다음과 같이 정의됩니다.

$$
\llbracket E\in\overline{v}\rrbracket\_{G,gs} = \bigvee\_{v\in\overline{v}}\llbracket E\rrbracket\_{G,gs}=v
$$

멤버십 술어는 $E$의 평가 값이 값 리스트 $\overline{v}$의 어느 원소와 같은지를 논리합($\bigvee$)으로 검사합니다.

### 존재 술어와 기본 키 정렬

존재 술어 $\mathsf{Exists}(PP)$의 의미론은 특히 주목할 가치가 있습니다.

$$
\llbracket\mathsf{Exists}(PP)\rrbracket\_{G,gs} = \bigvee\_{g\in\llbracket PP\rrbracket\_G}\bigwedge\_{K\in\overline{K}}\llbracket K\rrbracket\_{G,gs}=\llbracket K\rrbracket\_{G,[g]} \quad \text{where } \overline{K}=[\mathsf{PK}(\mathsf{head}(PP)),\mathsf{PK}(\mathsf{last}(PP))]
$$

이 정의를 풀어 보면, 경로 패턴 $PP$에 매칭되는 어떤 부분 그래프 $g$가 존재하여($\bigvee\_{g}$), 그 $g$가 현재 문맥 $gs$와 **양 끝 노드의 기본 키**에서 일치하는지를 따집니다. 비교 대상이 되는 키 집합 $\overline{K}$는 패턴의 머리($\mathsf{head}(PP)$)와 꼬리($\mathsf{last}(PP)$) 노드의 기본 키($\mathsf{PK}$)로 구성됩니다. 즉 패턴의 시작 노드와 끝 노드가 현재 매칭 문맥의 대응 노드와 동일한 개체를 가리키는지를 기본 키 일치로 확인하는 것입니다. 이는 앞서 형식적 기반에서 $\mathsf{Exists}$ 술어가 "투영된 서브쿼리에 대한 SQL의 `IN` 술어"로 번역된다고 예고한 부분의 의미론적 근거이며, 동기 예제에서 중첩 $\mathsf{Match}$ 대신 $\mathsf{Exists}$를 쓰면 이중 집계 버그가 사라진다는 사실의 형식적 출처이기도 합니다. 핵심은 $\mathsf{Exists}$가 "경로가 몇 개 있는가"가 아니라 "그러한 경로가 하나라도 있는가"만을 따지기 때문에, 동일한 끝점에 이르는 여러 경로가 중복 집계를 일으키지 않는다는 점입니다.

마지막으로 논리 결합 연산자들의 의미론입니다.

$$
\llbracket\phi\_1\land\phi\_2\rrbracket\_{G,gs} = \llbracket\phi\_1\rrbracket\_{G,gs}\land\llbracket\phi\_2\rrbracket\_{G,gs}
$$

$$
\llbracket\phi\_1\lor\phi\_2\rrbracket\_{G,gs} = \llbracket\phi\_1\rrbracket\_{G,gs}\lor\llbracket\phi\_2\rrbracket\_{G,gs} \qquad \llbracket\neg\phi\rrbracket\_{G,gs} = \neg\llbracket\phi\rrbracket\_{G,gs}
$$

논리곱·논리합·부정은 각각 하위 술어의 평가 결과에 대응하는 논리 연산을 적용합니다. 다만 이때의 논리 연산은 통상의 이치 논리가 아니라 $\top$, $\bot$, $\mathsf{Null}$ 세 값 위에서 정의된 삼치 논리라는 점이 결정적입니다. 앞서 형식적 기반에서 설명했듯, 특수한 단축 규칙으로 $\bot \land \mathsf{Null} = \bot$과 $\top \lor \mathsf{Null} = \top$이 성립합니다. 직관적으로 보면, 논리곱에서 한쪽이 이미 거짓이면 다른 쪽이 미지($\mathsf{Null}$)여도 전체가 거짓으로 확정되고, 논리합에서 한쪽이 이미 참이면 다른 쪽이 미지여도 전체가 참으로 확정되기 때문입니다. 그 외의 경우, 즉 어느 한 인자라도 $\mathsf{Null}$이면서 위 단축 규칙이 적용되지 않는 경우에는 결과가 $\mathsf{Null}$이 됩니다. 이러한 삼치 논리 규칙은 SQL의 `NULL` 처리 방식과 정확히 일치하며, $\mathsf{OptMatch}$가 매칭 실패 시 만들어 내는 $\mathsf{Null}$ 값이 후속 술어 평가에서 SQL의 외부 조인 의미론과 어긋나지 않도록 보장하는 마지막 퍼즐 조각입니다.

이로써 Featherweight Cypher의 모든 구문 범주—질의, 반환 질의, 절, 경로 패턴, 표현식, 술어—에 대한 표시적 의미론이 완성되었습니다. 이 형식적 의미론은 앞서 전개한 트랜스파일레이션의 건전성·완전성 정리가 정확히 무엇을 보존해야 하는지를 규정하는 기준점이 되며, 그래프 질의의 추상적 동작을 빠짐없이 수학적으로 못 박음으로써 SQL로의 번역이 의미를 왜곡하지 않음을 증명할 수 있는 토대를 제공합니다.
## 부록: Cypher 표현식과 술어의 트랜스파일레이션

앞서 본문에서 질의·절·경로 패턴 수준의 번역 규칙을 다루고, 또 다른 부록에서 Featherweight Cypher의 표시적 의미론을 빠짐없이 정의했습니다. 이제 마지막으로 남은 조각은 그 두 세계를 잇는 가장 미세한 단위, 곧 **표현식(expression)과 술어(predicate)를 SQL로 번역하는 규칙**입니다. 이들은 본문의 절·패턴 규칙이 호출하는 하위 번역 단계로, 예컨대 `Match` 절의 필터 조건 $\phi$나 `Return` 절의 출력 표현식 $E$가 SQL의 대응물로 바뀔 때 바로 이 규칙들이 작동합니다. 흥미로운 점은, 대부분의 규칙이 거의 항등(identity)에 가까울 만큼 단순하다는 사실입니다. Cypher의 표현식·술어 문법이 SQL의 그것과 구조적으로 매우 닮아 있기 때문에, 대부분은 하위 구성 요소를 재귀적으로 번역한 뒤 그대로 같은 형태로 재조립하면 됩니다.

### 표현식 번역 규칙

표현식 번역은 $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash E \overset{\textsf{expr}}{\longrightarrow} E'$ 형태의 판단을 사용합니다. 즉 표준 변환기 $\Phi\_{\mathsf{sdt}}$와 유도 스키마 $\Psi\_R$가 주어졌을 때 Cypher 표현식 $E$가 SQL 표현식 $E'$으로 사상됨을 뜻합니다. 가장 기본이 되는 두 규칙은 전제가 전혀 없는, 곧 분자가 비어 있는 공리(axiom) 형태입니다.

$$
\frac{}{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash k \overset{\textsf{expr}}{\longrightarrow} k}\ \text{(E-Prop)} \qquad \frac{}{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash v \overset{\textsf{expr}}{\longrightarrow} v}\ \text{(E-Value)}
$$

**E-Prop 규칙**은 속성 키 $k$를 아무런 변형 없이 그대로 SQL 속성 $k$로 번역하고, **E-Value 규칙**은 값 상수 $v$를 그대로 $v$로 옮깁니다. 직관적으로 이는 "그래프에서 노드의 속성을 읽는 일"이 유도 스키마에서는 "대응 테이블의 같은 이름 열을 읽는 일"과 정확히 같기 때문입니다. 앞서 표준 변환기가 노드·엣지의 속성 키를 동명의 테이블 열로 사상하도록 설계되었으므로, 속성 키 이름이 그대로 보존되는 이 단순성이 자연스럽게 따라옵니다.

나머지 표현식 규칙들은 모두 하위 표현식을 번역한 뒤 같은 연산자로 재조립하는 **합동(congruence) 규칙**입니다.

$$
\frac{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash \phi \overset{\textsf{pred}}{\longrightarrow} \phi'}{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash \mathsf{Cast}(\phi) \overset{\textsf{expr}}{\longrightarrow} \texttt{Cast}(\phi')}\ \text{(E-Pred)}
$$

**E-Pred 규칙**은 술어를 값으로 변환하는 $\mathsf{Cast}(\phi)$를 다루는데, 먼저 내부 술어 $\phi$를 술어 번역 규칙으로 $\phi'$으로 옮긴 다음 SQL의 `Cast`로 감쌉니다. 이는 표현식 번역이 술어 번역을 호출하는 상호 재귀(mutual recursion) 구조를 보여 주는 대목입니다.

$$
\frac{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash E \overset{\textsf{expr}}{\longrightarrow} E'}{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash \mathsf{Agg}(E) \overset{\textsf{expr}}{\longrightarrow} \texttt{Agg}(E')}\ \text{(E-Agg)}
$$

**E-Agg 규칙**은 집계 표현식 $\mathsf{Agg}(E)$를 내부 표현식 $E$를 번역한 뒤 같은 집계 연산자로 재구성합니다. 여기서 $\mathsf{Agg}$는 앞서 정의한 $\{\mathsf{Count}, \mathsf{Avg}, \mathsf{Sum}, \mathsf{Min}, \mathsf{Max}\}$ 중 하나이며, Cypher의 각 집계 함수가 SQL의 동명 집계 함수로 직접 대응되기 때문에 번역이 단순해집니다.

$$
\frac{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash E\_1 \overset{\textsf{expr}}{\longrightarrow} E\_1' \quad \Phi\_{\mathsf{sdt}}, \Psi\_R \vdash E\_2 \overset{\textsf{expr}}{\longrightarrow} E\_2'}{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash E\_1 \oplus E\_2 \overset{\textsf{expr}}{\longrightarrow} E\_1' \oplus E\_2'}\ \text{(E-Arith)}
$$

**E-Arith 규칙**은 산술 연산 $E\_1 \oplus E\_2$를 두 피연산자를 각각 번역한 뒤 같은 산술 연산자로 결합합니다. 이렇게 표현식 번역 규칙 전체가 "기저 사례(E-Prop, E-Value)는 항등, 나머지는 합동"이라는 깔끔한 구조를 이루기에, 별다른 어려움 없이 기계적으로 적용됩니다.

### 술어 번역 규칙

술어 번역은 $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash \phi \overset{\textsf{pred}}{\longrightarrow} \phi'$ 형태의 판단을 사용합니다. 술어 규칙 역시 대부분 표현식 규칙과 같은 합동 패턴을 따릅니다. 참과 거짓은 공리로서 그대로 번역되며(**P-True**, **P-False**), 비교 술어는 두 표현식을 번역한 뒤 같은 비교 연산자로 견줍니다.

$$
\frac{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash E\_1 \overset{\textsf{expr}}{\longrightarrow} E\_1' \quad \Phi\_{\mathsf{sdt}}, \Psi\_R \vdash E\_2 \overset{\textsf{expr}}{\longrightarrow} E\_2'}{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash E\_1 \odot E\_2 \overset{\textsf{pred}}{\longrightarrow} E\_1' \odot E\_2'}\ \text{(P-Logic)}
$$

여기서 비교 연산자 $\odot$가 그대로 보존되는 **P-Logic 규칙**은, 술어 번역이 다시 표현식 번역을 호출하는 또 다른 상호 재귀 지점입니다. 같은 맥락에서 **P-IsNull 규칙**은 $\mathsf{IsNull}(E)$를 내부 표현식 $E$를 번역한 뒤 SQL의 `IsNull`로 감싸고, **P-In 규칙**은 멤버십 술어 $E \in \overline{v}$를 $E' \in \overline{v}$로 옮깁니다. 부정과 논리 결합도 마찬가지로 단순한 합동 규칙입니다. **P-Not 규칙**은 $\neg\phi$를 $\neg\phi'$으로, **P-AndOr 규칙**은 $\circ \in \{\land, \lor\}$인 논리 결합 $\phi\_1 \circ \phi\_2$를 양쪽 술어를 각각 번역한 뒤 $\phi\_1' \circ \phi\_2'$으로 재조립합니다.

### 유일하게 까다로운 규칙: P-Exists

이처럼 거의 모든 규칙이 항등이나 합동에 가까운 가운데, **오직 한 규칙만이 본질적으로 비자명(non-trivial)합니다**. 바로 $\mathsf{Exists}$ 술어를 번역하는 **P-Exists 규칙**입니다. 이 규칙이 까다로운 이유는, 단순히 하위 구성 요소를 재귀적으로 번역하는 데 그치지 않고 경로 패턴 전체를 하나의 SQL 서브쿼리로 번역한 다음 그것을 멤버십 검사로 다시 엮어야 하기 때문입니다.

$$
\frac{\begin{array}{c}(X\_1, l\_1) = \text{head}(PP) \quad (X\_2, l\_2) = \text{last}(PP) \\ l\_1(\ldots) \to R\_{l\_1}(\ldots) \in \Phi\_{\mathsf{sdt}} \quad l\_2(\ldots) \to R\_{l\_2}(\ldots) \in \Phi\_{\mathsf{sdt}} \\ \Phi\_{\mathsf{sdt}}, \Psi\_R \vdash PP \overset{\textsf{pattern}}{\longrightarrow} \mathcal{X}, Q \quad \overline{a} = [\mathsf{PK}(R\_{l\_1}), \mathsf{PK}(R\_{l\_2})]\end{array}}{\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash \text{Exists}(PP) \overset{\textsf{pred}}{\longrightarrow} \overline{a} \in \Pi\_{\overline{a}}(Q)}\ \text{(P-Exists)}
$$

이 규칙을 단계적으로 풀어 보면 이렇습니다. 먼저 경로 패턴 $PP$의 머리 노드 $(X\_1, l\_1)$와 꼬리 노드 $(X\_2, l\_2)$를 각각 $\text{head}$와 $\text{last}$로 식별합니다. 그런 다음 표준 변환기 $\Phi\_{\mathsf{sdt}}$를 통해 이 두 레이블 $l\_1$, $l\_2$에 대응하는 관계형 테이블 $R\_{l\_1}$, $R\_{l\_2}$를 찾아냅니다. 이어서 경로 패턴 $PP$ 자체를 본문에서 다룬 패턴 번역 규칙으로 SQL 서브쿼리 $Q$로 옮기고, 양 끝 노드 테이블의 기본 키들을 모은 속성 리스트 $\overline{a} = [\mathsf{PK}(R\_{l\_1}), \mathsf{PK}(R\_{l\_2})]$를 구성합니다. 최종 결과는 $\overline{a} \in \Pi\_{\overline{a}}(Q)$, 즉 **부모 질의에서 쓰이는 양 끝 노드의 기본 키 쌍이 서브쿼리 $Q$가 반환하는 결과 집합 안에 있는지를 검사하는 SQL의 `IN` 술어**입니다.

이 번역이 의미적으로 올바른 이유를 직관적으로 이해하는 것이 중요합니다. Cypher의 의미론에서 $\mathsf{Exists}(PP)$는 그래프 안에 패턴 $PP$의 매칭이 **하나라도 존재하는지**만을 검사하는 술어입니다. 앞서 표시적 의미론에서 이 술어가 양 끝 노드의 기본 키 일치를 논리합($\bigvee$)으로 따진다고 정의되었던 부분이 여기서 정확히 SQL로 실현됩니다. 즉 패턴을 SQL 조인 질의 $Q$로 펼쳐 놓고, 그 결과에서 양 끝점의 기본 키만 투영($\Pi\_{\overline{a}}$)한 뒤, 현재 문맥의 기본 키 쌍이 그 집합에 속하는지를 멤버십으로 묻는 것입니다. 멤버십 검사는 본질적으로 "있다/없다"만을 따지므로, 같은 끝점에 이르는 경로가 여러 개여도 중복 집계가 일어나지 않습니다. 이것이 바로 앞선 동기 예제에서 중첩된 `MATCH` 대신 `EXISTS`를 사용하면 이중 집계 버그가 사라지는 현상의 형식적 근원입니다.

### P-Exists 규칙의 구체적 적용: Example B.1

이 추상적인 규칙이 실제로 어떻게 작동하는지는 직원-부서 예제(Example B.1)를 통해 손에 잡히게 확인할 수 있습니다. 앞서 다룬 표준 데이터베이스 변환기 $\Phi\_{\mathsf{sdt}}$ 아래에서, 다음과 같은 Cypher 술어를 생각해 봅시다.

$$
\mathsf{Exists}([(\texttt{n}, \texttt{EMP}), (\texttt{e}, \texttt{WORK\_AT}, \rightarrow), (\texttt{m}, \texttt{DEPT})])
$$

이 술어는 "`EMP` 타입 노드에서 `WORK_AT` 엣지를 따라 `DEPT` 타입 노드로 이어지는 경로가 존재하는가"를 묻습니다. P-Exists 규칙을 적용하면, 머리 노드는 $(\texttt{n}, \texttt{EMP})$이고 꼬리 노드는 $(\texttt{m}, \texttt{DEPT})$이므로, 양 끝 노드의 기본 키로 이루어진 속성 리스트는 $[\texttt{n.id}, \texttt{m.dnum}]$이 됩니다. 그리고 경로 패턴 전체는 앞서 본문의 경로 패턴 번역에서 본 것과 동일한 세 테이블 조인 질의로 펼쳐집니다. 최종 번역 결과는 다음과 같은 SQL 멤버십 술어입니다.

$$
[\texttt{n.id}, \texttt{m.dnum}] \in \Pi\_{[\texttt{n.id}, \texttt{m.dnum}]}\big(\rho\_{\mathtt{n}}(\mathtt{emp}) \bowtie\_{\mathtt{n.id}=\mathtt{e.SRC}} \rho\_{\mathtt{e}}(\mathtt{work\_at}) \bowtie\_{\mathtt{e.TGT}=\mathtt{m.dnum}} \rho\_{\mathtt{m}}(\mathtt{dept})\big)
$$

이 결과를 차근차근 읽어 보면, 서브쿼리의 내부는 본문에서 경로 패턴을 번역할 때 등장했던 바로 그 조인 질의입니다. `emp` 테이블을 `n`으로 이름 변경하고, `work_at` 테이블을 `e`로 이름 변경하여 외래 키 조건 $\mathtt{n.id} = \mathtt{e.SRC}$로 내부 조인하며, 다시 `dept` 테이블을 `m`으로 이름 변경하여 $\mathtt{e.TGT} = \mathtt{m.dnum}$으로 조인합니다. 이 조인 결과에서 양 끝점의 기본 키 $\texttt{n.id}$와 $\texttt{m.dnum}$만을 투영한 집합이 만들어지고, 현재 문맥에서 다루는 직원·부서 식별자 쌍 $[\texttt{n.id}, \texttt{m.dnum}]$이 그 집합에 속하는지를 `IN`으로 검사하는 것입니다. 여기서 외래 키 속성 `SRC`와 `TGT`가 조인 술어를 떠받치는 모습은, 앞서 엣지 규칙이 생성한 외래 키 제약이 패턴 번역의 타당성을 보장한다는 사실과 정확히 맞물립니다.

이로써 표현식과 술어 수준의 트랜스파일레이션 규칙까지 모두 갖추어져, 본문에서 다룬 질의·절·경로 패턴 규칙과 합쳐져 Featherweight Cypher의 모든 구문 범주를 SQL로 빠짐없이 번역할 수 있는 완전한 규칙 집합이 완성됩니다. 거의 모든 규칙이 단순한 항등·합동임에도 불구하고 $\mathsf{Exists}$ 하나만이 패턴 번역과 멤버십 검사를 결합하는 비자명한 처리를 요구한다는 점은, 이 술어가 그래프 질의와 관계형 질의 사이의 의미론적 간극이 가장 첨예하게 드러나는 지점임을 다시 한번 보여 줍니다.
# 부록: 동기 예제의 올바른 Cypher 질의와 수작업 버그 분석

## 동기 예제를 올바르게 고친 Cypher 질의

본문 동기 예제에서 살펴본 바와 같이, 그래프 데이터베이스에 대한 원래 Cypher 질의는 첫 번째 패턴 매칭 결과를 `WITH s` 절로 다음 단계에 넘기는 방식 때문에 SQL 질의와 동치가 아님이 증명되었습니다. 문제의 핵심은 이렇습니다. `WITH` 절은 `s:SENTENCE` 노드를 거르기는 하지만, 그 필터링이 두 번째 `MATCH` 절의 **개별 매치 수준**에서 강제되지는 않습니다. 그 결과 두 번째 `MATCH`가 의도보다 많은 경로를 탐색하게 되고, 이것이 곧 이중 집계(double counting)로 이어집니다.

이를 바로잡은 올바른 Cypher 질의는 필터링을 `EXISTS` 조건 안으로 옮겨, 각 `SENTENCE` 노드가 첫 번째 패턴을 만족하는 경우에만 고려되도록 보장합니다.

```cypher
MATCH (s:SENTENCE)<-[r3:SP]-(p2:PA)<-[r4:CS]-(c2:CONCEPT)
WHERE EXISTS {
  MATCH (c1:CONCEPT {CID: 1})-[r1:CS]->(p1:PA)-[r2:SP]->(s:SENTENCE)
}
RETURN c2.CID, Count(*)
```

여기서 핵심적인 차이는 첫 번째 패턴이 더 이상 `WITH`를 통해 중간 결과로 전파되지 않고, `EXISTS { ... }` 술어 안에 갇혀 "그러한 경로가 존재하는가"만을 검사한다는 점입니다. 앞서 부록에서 살펴본 P-Exists 규칙이 형식적으로 뒷받침하듯, `EXISTS`는 경로의 개수를 세지 않고 양 끝 노드 기본 키의 일치 여부만을 논리합으로 따지므로, 같은 `SENTENCE`에 이르는 여러 경로가 있어도 중복으로 집계되지 않습니다. 이러한 구조적 변경은 동기 예제에서 관찰된 중복 집계를 제거하여 SQL 질의와 동일한 결과를 산출하게 됩니다.

## 수작업으로 작성된 버그 질의에 대한 정성 분석

평가에서 사용된 수작업 작성 벤치마크의 버그들을 직접 검토한 결과, 몇 가지 공통적인 근본 원인이 식별되었습니다. 첫 번째 원인은 방금 살펴본 것처럼 **존재 패턴(existential pattern) 대신 중첩 `MATCH`를 사용**하는 경우입니다.

### 경로 패턴을 OPTIONAL MATCH에 잘못 사용하는 경우

두 번째 원인은 `OPTIONAL MATCH`에서 경로 패턴을 오용하는 경우입니다. Neo4j 공식 튜토리얼에서 가져와 정렬 절을 제거하고 레이블을 재명명한 다음 예제를 살펴보겠습니다. 원본 SQL 질의는 고객 `Drachenblut Delikatessen`에 대해 제품별 판매량을 구하는 것으로, 네 개의 테이블을 좌외부 조인(`LEFT JOIN`)으로 연결합니다.

```sql
SELECT P.ProductName, Sum(OD.UnitPrice * OD.Quantity) AS Volume
FROM Customers AS C
  LEFT JOIN Orders AS O ON C.CustomerID = O.CustomerID
  LEFT JOIN OrderDetails AS OD ON O.OrderID = OD.OrderID
  LEFT JOIN Products AS P ON OD.ProductID = P.ProductID
WHERE C.CompanyName = 'Drachenblut Delikatessen'
GROUP BY P.ProductName
```

튜토리얼이 제시한 대응 Cypher 질의는 단일 `OPTIONAL MATCH`로 세 노드와 두 엣지를 잇는 경로 패턴을 사용합니다.

```cypher
MATCH (C:Customer {CompanyName:'Drachenblut Delikatessen'})
OPTIONAL MATCH (P:Product)<-[OD:OrderDetails]-(:Order)<-[:Purchased]-(C)
RETURN P.ProductName, Sum(OD.UnitPrice * OD.Quantity) AS Volume
```

그러나 이 Cypher 질의는 SQL 질의와 동치가 아닙니다. 문제의 본질은 SQL이 `Customers → Orders → OrderDetails → Products`로 이어지는 **네 개의 좌외부 조인 체인**을 사용하여 단계별로 "없는 `Product`"까지 보존하는 반면, 단일 `OPTIONAL MATCH` 경로 패턴은 이 네 조인을 하나의 선택적 경로로 뭉뚱그리면서 그 경로 내부에서는 사실상 내부 조인을 수행하기 때문입니다. 즉 경로 전체가 매칭되거나 전혀 매칭되지 않는 양자택일이 되어, 중간 단계에서 누락된 `Product`를 SQL 좌외부 조인처럼 개별적으로 보존하지 못합니다. 이 차이는 Cypher를 SQL로 트랜스파일한 결과를 보면 한층 분명해지는데, 트랜스파일된 SQL의 핵심 부분이 원래 SQL의 좌외부 조인 대신 내부 조인을 사용하게 됨이 드러납니다.

```sql
WITH T0 AS (
  SELECT C.CustomerID AS C_CustomerID
  FROM Customer AS C
  WHERE C.CompanyName = 'Drachenblut Delikatessen' ),
T1 AS (
  SELECT P.ProductName AS P_ProductName, OD.Quantity AS OD_Quantity,
         OD.UnitPrice AS OD_UnitPrice, C.CustomerID AS C_CustomerID
  FROM Product AS P JOIN OrderDetails AS OD JOIN Purchased AS PD JOIN Customer AS C
  ON P.ProductID = OD.TGT AND OD.SRC = PD.TGT AND PD.SRC = C.CustomerID ),
T2 AS (
  SELECT T1.P_ProductName AS ProductName, T1.O_Quantity AS Quantity,
         T1.O_UnitPrice AS UnitPrice
  FROM T0 LEFT JOIN T1 ON T0.C_CustomerID = T1.C_CustomerID )
SELECT ProductName, Sum(UnitPrice * Quantity) AS Volume
FROM T2 GROUP BY ProductName;
```

여기서 `T1`을 정의하는 부분(밑줄로 표시된 절)이 네 테이블을 모두 내부 조인(`JOIN`)으로 묶어 버리는 것을 볼 수 있습니다. 원래 SQL이 단계마다 좌외부 조인으로 누락을 보존하던 것과 달리, 이 트랜스파일 결과는 경로 전체가 매칭될 때만 행을 남기므로 의미론이 어긋나게 됩니다.

### 같은 레이블의 서로 다른 노드/엣지를 잘못 사용하는 경우

세 번째 원인은 같은 레이블을 가진 서로 다른 노드나 엣지를 혼동하는 경우입니다. VeriEQL 데이터셋에서 수집된 다음 SQL 질의를 살펴보겠습니다.

```sql
SELECT t0.EmpNo, t0.DeptNo, t1.DeptNo AS DeptNo0
FROM (SELECT EmpNo, EName, DeptNo, DeptNo + EmpNo AS f9
      FROM EMP WHERE EmpNo = 10) AS t0
JOIN (SELECT DeptNo, Name, DeptNo + 5 AS f2
      FROM DEPT) AS t1
ON t0.EmpNo = t1.DeptNo AND t0.f9 = t1.f2
```

한 대학원생이 작성한 대응 Cypher 질의는 다음과 같았습니다.

```cypher
MATCH (t0:EMP {EmpNo: 10})-[:WORK_AT]->(t1:DEPT)
WHERE t1.DeptNo + t0.EmpNo = t1.DeptNo + 5
RETURN t0.EmpNo, t1.DeptNo, t1.DeptNo AS DeptNo0
```

이 Cypher 질의가 SQL과 동치가 아닌 이유는, `t1`과는 **다른 또 하나의 `DEPT` 노드**를 도입하지 못했기 때문입니다. 좀 더 정확히 말하면, 원래 SQL은 `EMP`에서 나온 서브쿼리 `t0`(여기에 `DeptNo` 속성이 포함됨)와 `DEPT`에서 나온 서브쿼리 `t1`을 `t0.EmpNo = t1.DeptNo`와 `t0.f9 = t1.f2` 조건으로 조인합니다. 즉 `t0`의 `DeptNo`(출처는 `EMP` 테이블)와 `t1`의 `DeptNo`(출처는 `DEPT` 테이블)는 서로 다른 속성 출처를 가집니다. 그런데 Cypher 질의는 단일 `WORK_AT` 엣지를 사용함으로써 이 둘을 동일한 노드 `t1`로 취급해 버려, 잘못된 `t1`을 매칭하고 잘못된 필터링 술어를 적용하게 됩니다.

Graphiti는 이 SQL과 Cypher 질의가 동치가 아님을 반증하면서 반례를 제시합니다. 이 반례는 동일한 데이터를 서로 다른 형식으로 담은 관계형 데이터베이스와 그래프 데이터베이스 한 쌍으로 구성됩니다. 관계형 데이터베이스의 `EMP` 테이블은 `(EmpNo=0, EName=A, DeptNo=10)`과 `(EmpNo=10, EName=B, DeptNo=5)` 두 행을, `DEPT` 테이블은 `(DeptNo=5, DName=C)`와 `(DeptNo=10, DName=D)` 두 행을 가집니다. 이 관계형 데이터베이스에서 원래 SQL 질의를 실행하면 한 행 `(10, 5, 10)`을 반환합니다. 구체적으로 `EmpNo = 10`인 행 B에서 `t0.EmpNo = 10`, `t0.DeptNo = 5`가 나오고, `DEPT`의 행 D에서 `t1.DeptNo = 10`이 와서 조인 조건 `t0.EmpNo(=10) = t1.DeptNo(=10)`가 충족됩니다. 반면 같은 데이터를 그래프 형식(`EMP` 레이블의 실선 노드 A·B, `DEPT` 레이블의 점선 노드 C·D를 `WORK_AT` 엣지로 연결)으로 옮겨 Cypher 질의를 실행하면 빈 결과가 나옵니다. Cypher가 `t0`의 `DeptNo`와 `t1`의 `DeptNo`를 단일 노드로 합쳐 버려, 서로 다른 두 부서 개체를 필요로 하는 이 조인을 표현할 수 없기 때문입니다.

OpenCypherTranspiler가 올바르게 번역하지 못하는 구체적 사례를 세 가지 살펴볼 수 있습니다. 첫 번째는 아예 번역 자체가 불가능한 경우입니다.

```cypher
MATCH (N:PERSON)
WITH N.ZIPCODE AS ZIP, Count(*) AS POPULATION
WHERE POPULATION > 3
RETURN ZIP, POPULATION
```

OpenCypherTranspiler는 `Count(*)`나 `Avg(*)` 같은 표현식을 지원하지 않으므로 이 질의를 다룰 수 없습니다. 두 번째는 정의되지 않은 변수를 포함한 잘못된 형태의 SQL을 생성하는 경우입니다.

```cypher
MATCH (X:USR), (U:PIC), (V:PIC), (W:PIC)
WHERE X.USRUID = U.PICUID AND X.USRUID = V.PICUID AND X.USRUID = W.PICUID
  AND W.PRICSIZE = V.PRICSIZE AND U IS NOT NULL AND V IS NULL
RETURN DISTINCT X.USRUID AS XID, X.USRNAME AS XNAME
```

OpenCypherTranspiler는 이 질의를 다음과 같이 번역합니다.

```sql
SELECT DISTINCT __X_USRUID AS XID, …
FROM ( … WHERE … AND ((U) IS NOT NULL) AND … ) AS _proj
```

그런데 이 SQL 질의는 정의되지 않은 테이블 이름 `U`를 사용하므로 유효하지 않습니다. 세 번째 사례는 구문적으로는 유효하지만 의미적으로 부정확한 번역 결과를 산출하는 경우로, 제공된 내용에서는 구체적인 질의가 제시되지 않았으나, 이러한 사례의 존재 자체가 건전성 보장 없는 도구의 근본적 한계를 다시 한번 드러냅니다. 이처럼 Graphiti가 형식적 동치성 보장과 함께 완전한 커버리지를 제공한다는 점은, 단순히 더 많은 질의를 다룰 수 있다는 양적 우위를 넘어 트랜스파일 결과의 신뢰성이라는 질적 차이를 만들어 냅니다.
# 부록: 번역 규칙의 건전성과 완전성에 대한 증명

본문에서 트랜스파일레이션의 건전성(Theorem 5.7)·완전성(Theorem 5.8)과 전체 방법론의 건전성(Theorem 5.9)·완전성(Theorem 5.10)을 정리로 진술했지만, 그 진술이 왜 참인지에 대한 형식적 증명은 미루어 두었습니다. 이 부록은 그 약속을 이행하는 부분으로, 앞서 정의한 표시적 의미론을 토대로 구조적 귀납법(structural induction)을 통해 각 정리를 빈틈없이 증명합니다. 한 가지 흥미로운 출발점으로, 이 부록은 기존 도구 OpenCypherTranspiler가 범하는 또 하나의 구체적 오류를 먼저 제시한 뒤 본격적인 증명으로 넘어갑니다.

## OpenCypherTranspiler의 조인 방향 오류

앞서 OpenCypherTranspiler가 지원 범위가 좁고 건전성 보장이 없음을 살펴보았는데, 여기서는 의미적으로 부정확한 SQL을 만들어 내는 또 다른 사례가 제시됩니다. 다음 Cypher 질의를 생각해 봅시다.

```cypher
MATCH (e1:EMPLOYEES)
OPTIONAL MATCH (e2:EMPLOYEEUNI)-[:IS]->(e1)
RETURN e2.UNIQUE_ID, e1.NAME
```

이 질의의 의도는 분명합니다. 모든 `EMPLOYEES` 노드 `e1`을 보존하면서, 그에 연결되는 `EMPLOYEEUNI` 노드 `e2`가 있으면 가져오고 없으면 `null`로 채우는 것입니다. 즉 `e1` 쪽이 반드시 보존되어야 하는 기준 테이블입니다. 그런데 OpenCypherTranspiler는 이를 다음과 같이 번역합니다.

```sql
SELECT UNIQUE_ID, NAME
FROM (SELECT * FROM EMPLOYEEUNI JOIN IS ON …) AS T1
LEFT JOIN EMPLOYEES AS T2 ON …
```

문제는 좌외부 조인의 방향입니다. 위 번역은 `EMPLOYEEUNI` 측(`T1`)을 왼쪽에 두고 `EMPLOYEES`(`T2`)를 오른쪽에 두어 `LEFT JOIN`을 수행하므로, 보존되는 쪽이 `EMPLOYEEUNI`가 되어 버립니다. 그러나 원래 `OPTIONAL MATCH`의 의미론에서 보존되어야 하는 것은 선행 `MATCH`의 결과인 `EMPLOYEES` 쪽입니다. 따라서 올바른 SQL은 다음과 같아야 합니다.

```sql
SELECT UNIQUE_ID, NAME
FROM (SELECT * FROM EMPLOYEEUNI JOIN IS ON …) AS T1
RIGHT JOIN EMPLOYEES AS T2 ON …
```

즉 `LEFT JOIN`이 아니라 `RIGHT JOIN`이어야 `EMPLOYEES`의 모든 행이 보존됩니다. 이 사례는 앞서 정성 분석에서 살펴본 "`OPTIONAL MATCH`를 내부 조인처럼 잘못 펼치는" 부류와 더불어, 선택적 매칭의 보존 방향을 잘못 잡는 오류가 실제 도구에서 얼마나 쉽게 발생하는지를 보여 줍니다. 동시에 Graphiti의 절 번역 규칙이 왜 좌외부 조인의 보존 방향—선행 절의 결과를 왼쪽에 두고 그것을 보존하는 방향—을 신중히 고정하는지를 역으로 정당화합니다.

## 번역의 건전성 증명 (Theorem F.1)

건전성 정리는 다음을 단언합니다. 그래프 스키마 $\Psi\_G$와 그 위의 Cypher 질의 $Q$, 유도 관계형 스키마 $\Psi\_R = (S, \xi)$, 그리고 표준 변환기 $\Phi\_{\mathsf{sdt}}$가 주어졌을 때, 만약 $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash Q \overset{\textsf{query}}{\longrightarrow} Q'$이 성립하면 $Q$와 $Q'$은 $\Phi\_{\mathsf{sdt}}$를 기준으로 동치, 즉 $Q \simeq\_{\Phi\_{\mathsf{sdt}}} Q'$입니다. 증명은 질의 $Q$의 구조에 대한 귀납법으로 진행되며, 각 질의 형태(Return, OrderBy, Union, UnionAll)가 하나의 경우를 이룹니다.

### 기저 사례: Return 질의

기저 사례는 $Q = \mathsf{Return}(C, \overline{E}, \overline{k})$입니다. 귀납 가정에 의해 절 $C$의 번역 $Q'$에 대해 $\Phi\_{\mathsf{sdt}}(\llbracket C\rrbracket\_G) = \llbracket Q'\rrbracket\_R$이 성립합니다(이는 곧 살펴볼 Lemma F.2가 보장합니다). 증명의 핵심은 본문에서 보았던 집계 유무에 따른 분기를 그대로 따라가며, 양쪽 의미론이 일치함을 보이는 것입니다.

집계가 없는 경우($\neg\mathsf{hasAgg}(\overline{E}) = \top$), 즉 모든 표현식 $e \in \overline{E}$에 대해 $\mathsf{IsAgg}(e) = \bot$인 경우를 먼저 봅니다. 앞서 표시적 의미론에서 정의했듯 $\llbracket\mathsf{Return}(C,\overline{E},\overline{k})\rrbracket\_G$는 절 $C$의 각 부분 그래프 $g\_i$에 대해 표현식 리스트를 평가해 행을 만든 결과의 리스트가 됩니다. 한편 SQL 측에서는 투영의 의미론에 의해 $\llbracket\Pi\_{\rho\_{\overline{k}}(\overline{E'})}(Q')\rrbracket\_R$가 각 튜플 $t\_i$에 대해 대응되는 행을 만듭니다. 여기서 표현식 사이의 전단사 사상 $\pi = \{E\_j \mapsto E\_j' \mid 1 \leq j \leq m\}$이 존재하고, $\Phi\_{\mathsf{sdt}}(g\_i) = t\_i$인 그래프 $g\_i$와 튜플 $t\_i$에 대해 다음이 성립합니다.

$$
\llbracket E\_j\rrbracket\_{G,[g\_i]} = \llbracket E\_j'\rrbracket\_{R,[t\_i]}
$$

이 핵심 등식은 표현식 번역의 정확성을 보장하는 Lemma F.4에 의해 뒷받침됩니다. 즉 그래프 쪽에서 부분 그래프 $g\_i$ 위에 표현식 $E\_j$를 평가한 값과, 관계형 쪽에서 대응 튜플 $t\_i$ 위에 번역된 표현식 $E\_j'$를 평가한 값이 정확히 일치한다는 것입니다. 따라서 두 결과 테이블의 각 행이 일대일로 대응되어 동치가 증명됩니다. 정리하면, $G \sim\_{\Phi\_{\mathsf{sdt}}} R$을 만족하는 모든 $G, R$에 대해 $\Phi\_{\mathsf{sdt}}(\llbracket C\rrbracket\_G) = \llbracket Q\rrbracket\_R$이 성립합니다.

집계가 있는 경우($\neg\mathsf{hasAgg}(\overline{E}) = \bot$)는 그룹화의 일치를 보여야 하므로 한층 미묘합니다. 중복 제거된 그룹화 키 값의 개수 $n\_v = |\mathsf{dedup}(\overline{V})| \leq n = |\overline{V}|$를 두면, 그래프 측 의미론에서 $\mathsf{Groups} = [gs\_1, \ldots, gs\_{n\_v}]$가 형성되고 각 $gs\_i$는 적어도 하나의 그래프를 담은 그룹입니다. SQL 측에서는 `GroupBy`의 의미론에 의해 그룹화 리스트 $\mathsf{Ts}$가 도출됩니다. 여기서 결정적인 관찰은, 집계가 아닌 표현식 리스트에 대한 전단사 사상이 $\pi(\overline{A}) = \overline{A'}$를 만족하고 $\Phi\_{\mathsf{sdt}}(\llbracket C\rrbracket\_G) = \llbracket Q\rrbracket\_R$이므로, 그래프 측에서 $\overline{A}$를 평가해 얻는 고유한 결과의 개수와 SQL 측에서 $\overline{A'}$를 평가해 얻는 개수가 모두 정확히 $n\_v$로 일치한다는 점입니다. 곧 $|\mathsf{Groups}| = |\mathsf{Ts}| = n\_v$가 성립합니다. 각 그룹 $gs\_i$가 대응 튜플 그룹 $ts\_i$에 사상되어 $\Phi\_{\mathsf{sdt}}(\mathsf{Groups}) = \mathsf{Ts}$가 되고, 따라서 그룹 단위의 표현식 평가 결과까지 일치하므로

$$
\Phi\_{\mathsf{sdt}}(\llbracket\mathsf{Return}(C,\overline{E},\overline{k})\rrbracket\_G) = \llbracket\texttt{GroupBy}(Q', \overline{A'}, \rho\_{\overline{k}}(\overline{E'}), \top)\rrbracket
$$

이 성립하여 이 경우의 동치가 증명됩니다.

### 귀납 사례: OrderBy, Union, UnionAll

귀납 사례 $Q = \mathsf{OrderBy}(R\_1, k, b)$에서는 $R\_1$이 Cypher의 $\mathsf{Return}$ 문에 해당합니다. 귀납 가정으로 $\llbracket R\_1\rrbracket\_G = \llbracket Q'\rrbracket\_R$을 얻고, 정렬 키 $k$가 속성 키일 때 Lemma F.4에 의해 $\llbracket k\rrbracket\_{G,gs} = \llbracket k\rrbracket\_{R,\mathcal{T}}$가 성립합니다. 여기서 핵심 논거는 Cypher의 $\mathsf{OrderBy}$와 SQL의 `OrderBy`가 **열의 순서를 바꾸지 않는다**는 점인데, 덕분에 귀납 가정에서 얻은 전단사 사상 $\pi$를 이 경우에도 그대로 재사용할 수 있습니다. 또한 앞서 표시적 의미론에서 정의한 정렬 보조 함수 $\mathsf{MinTuple}$과 SQL의 대응 함수 $\mathtt{MinTuple}$이 정의상 동등하므로, $\Phi\_{\mathsf{sdt}}(\llbracket R\_1\rrbracket\_G) = \llbracket Q'\rrbracket\_{\mathcal{T}}$로부터 정렬 결과의 동치가 따라옵니다.

합집합 사례 $Q = \mathsf{Union}(Q\_1, Q\_2)$에서는 귀납 가정으로 $\llbracket Q\_1\rrbracket\_{G,gs} \simeq\_{\Phi\_{\mathsf{sdt}}} \llbracket Q\_1'\rrbracket\_R$과 $\llbracket Q\_2\rrbracket\_{G,gs} \simeq\_{\Phi\_{\mathsf{sdt}}} \llbracket Q\_2'\rrbracket\_R$을 얻습니다. 합집합도 열 순서를 바꾸지 않으므로 $\pi$를 물려받을 수 있고, 앞서 정의한 $\mathsf{dedup}$과 SQL의 `Distinct`의 정의에 의해 다음 사슬이 성립합니다.

$$
\llbracket\mathsf{Union}(Q\_1,Q\_2)\rrbracket\_G = \mathsf{dedup}(\llbracket\mathsf{UnionAll}(Q\_1,Q\_2)\rrbracket\_G) \simeq\_{\Phi\_{\mathsf{sdt}}} \texttt{Distinct}(\llbracket Q\_1' \uplus Q\_2'\rrbracket\_R) = \llbracket Q\_1' \cup Q\_2'\rrbracket\_R
$$

즉 그래프 측의 중복 제거 합집합이 SQL 측의 `Distinct`를 씌운 백 합집합, 곧 집합 합집합과 정확히 대응합니다. $\mathsf{UnionAll}$ 사례는 더 단순하여, 두 결과 리스트를 이어 붙이는($+\!+$) 연산이 양쪽에서 일치하므로 $\llbracket\mathsf{UnionAll}(Q\_1,Q\_2)\rrbracket\_G \simeq\_{\Phi\_{\mathsf{sdt}}} \llbracket Q\_1' \uplus Q\_2'\rrbracket\_R$이 곧바로 따라옵니다.

## 절 번역의 정확성 (Lemma F.2)

Lemma F.2는 절 수준의 핵심 보조정리로, $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash C \overset{\textsf{clause}}{\longrightarrow} \mathcal{X}, Q$이면 $G \sim\_{\Phi\_{\mathsf{sdt}}} R$인 모든 $G, R$에 대해 $\Phi\_{\mathsf{sdt}}(\llbracket C\rrbracket\_G) = \llbracket Q\rrbracket\_R$이 성립함을 단언합니다(여기서 $\Phi\_{\mathsf{sdt}}$는 리스트의 각 원소에 적용되도록 자연스럽게 확장됩니다). 증명은 절 $C$의 구조에 대한 귀납법입니다.

기저 사례 $C = \mathsf{Match}(PP, \phi)$에서는 패턴 정확성 보조정리 Lemma F.3에 의해 $\Phi\_{\mathsf{sdt}}(\llbracket PP\rrbracket\_G) = \llbracket Q\rrbracket\_R$을 얻고, 술어 정확성 보조정리 Lemma F.5에 의해 $\llbracket\phi\rrbracket\_{G,gs} = \llbracket\phi'\rrbracket\_{R,\mathcal{T}}$를 얻습니다. 이 둘을 결합하면 그래프 측의 $\mathsf{filter}$가 SQL 측의 선택 연산 $\sigma\_{\phi'}$로 정확히 대응됩니다.

$$
\Phi\_{\mathsf{sdt}}(\llbracket\mathsf{Match}(PP,\phi)\rrbracket\_G) = \mathsf{filter}(\lambda t.\llbracket\phi'\rrbracket\_{R,[t]}=\top, \llbracket Q\rrbracket\_R) = \llbracket\sigma\_{\phi'}(Q)\rrbracket\_R
$$

즉 술어를 만족하는 부분 그래프만 걸러 내는 동작이 술어를 만족하는 튜플만 남기는 선택 연산과 일치함을 보입니다.

### 연쇄 매칭과 변수 겹침의 두 경우

귀납 사례 $C = \mathsf{MATCH}(C\_1, PP, \phi)$의 증명은 본문에서 다룬 C-Match2 규칙의 정확성을 형식적으로 떠받칩니다. 여기서 흥미로운 점은 그래프의 $\mathsf{merge}$ 연산과 튜플의 $\texttt{merge}$ 연산이 동일한 기능을 한다는 관찰입니다. 즉 두 그래프 $g\_1, g\_2$에 대해 $\mathsf{merge}(g\_1, g\_2)$에는 중복된 노드·엣지·속성 키가 없으며, $\Phi\_{\mathsf{sdt}}(\mathsf{merge}(g\_1, g\_2)) = \texttt{merge}(t\_1, t\_2)$가 $\Phi\_{\mathsf{sdt}}(g\_1) = t\_1 \land \Phi\_{\mathsf{sdt}}(g\_2) = t\_2$와 동치입니다.

증명은 선행 절 $C\_1$의 변수 집합 $\mathcal{X}\_1$과 패턴 $PP$의 변수 집합 $\mathcal{X}\_2$의 교집합에 따라 두 경우로 나뉩니다. 교집합이 비어 있는 경우($\mathcal{X}\_1 \cap \mathcal{X}\_2 = \emptyset$)에는 $\phi'' = \phi'$이 되고, 두 서브쿼리의 곱이 카티전 곱으로 표현되어 결국 다음이 성립합니다.

$$
\Phi\_{\mathsf{sdt}}(\llbracket\mathsf{MATCH}(C\_1, PP, \phi)\rrbracket\_G) = \llbracket\rho\_{T\_1}(Q\_1) \bowtie\_{\phi''} \rho\_{T\_2}(Q\_2)\rrbracket\_R
$$

여기서 $T\_1, T\_2$는 새로운(fresh) 이름입니다. 핵심은 $\mathsf{filter}$로 술어를 적용한 두 결과의 모든 조합($\mathsf{map}$의 중첩)이 $\sigma\_{\phi'}(\rho\_{T\_1}(Q\_1) \times \rho\_{T\_2}(Q\_2))$와 같고, 이것이 내부 조인 $\bowtie\_{\phi'}$로 정리된다는 점입니다.

반면 교집합이 비어 있지 않은 경우($\mathcal{X}\_1 \cap \mathcal{X}\_2 \neq \emptyset$)에는, 본문에서 강조했던 대로 공통 변수의 기본 키가 일치하도록 강제하는 조인 술어가 추가됩니다.

$$
\phi'' = \phi' \land \bigwedge\_{(X:l)\in\mathcal{X}\_1\cap\mathcal{X}\_2} T\_1.\xi\_{pk}(\Lambda(l)) = T\_2.\xi\_{pk}(\Lambda(l))
$$

이 추가 술어가 바로 "절의 서로 다른 부분에 나타난 같은 변수의 출현들이 동일한 튜플을 가리키도록" 보장하는 장치이며, 결과적으로 $\rho\_{T\_1}(Q\_1) \bowtie\_{\phi''} \rho\_{T\_2}(Q\_2)$로 정리되어 정확성이 증명됩니다.

### 선택적 매칭의 정확성

귀납 사례 $C = \mathsf{OptMatch}(C\_1, PP, \phi)$는 가장 미묘한 부분으로, 앞서 표시적 의미론에서 정의한 두 보조 함수 $v\_1$과 $v\_2$의 동작을 SQL 좌외부 조인으로 환원하는 것이 핵심입니다. 증명은 $\mathsf{Nullify}$ 함수의 정확성에서 출발하는데, $\mathsf{Nullify}(\mathsf{head}(\llbracket PP\rrbracket\_G))$가 패턴의 첫 그래프와 동형이되 모든 속성 키가 $\mathsf{Null}$인 그래프를 돌려주므로, 이를 $\Phi\_{\mathsf{sdt}}$로 변환한 결과는 SQL 의미론에서 사용되는 유일한 널 튜플 $T\_{\mathsf{Null}}$과 같아집니다.

증명은 술어 $|v\_1(g\_i)| = 0$의 참·거짓에 따라 두 경우로 갈립니다. 매칭이 전혀 없는 경우($|v\_1(g\_i)| = 0$)에는 $v\_2(g\_i)$가 작동하여, 대응 튜플 $t\_i$와 널 튜플 $T\_{\mathsf{Null}}$의 곱 $[t\_i] \times [T\_{\mathsf{Null}}]$이 산출됩니다.

$$
\Phi\_{\mathsf{sdt}}(\mathsf{ite}(|v\_1(g\_i)|=0, v\_2(g\_i), v\_1(g\_i))) = [\texttt{merge}(t\_i, T\_{\mathsf{Null}})] = [t\_i] \times [T\_{\mathsf{Null}}]
$$

반대로 매칭이 존재하는 경우에는 $v\_1(g\_i)$가 작동하여 내부 조인과 동일한 형태 $[t\_i] \bowtie\_{\phi' \land \phi''} \llbracket Q\_2\rrbracket\_R$로 정리됩니다. 두 경우를 $\mathsf{foldl}$로 누적하면 전체 절이 좌외부 조인 $\rho\_{T\_1}(Q\_1) ⟕\_{\phi' \land \phi''} \rho\_{T\_2}(Q\_2)$로 번역됨이 증명되는데, 이는 "매칭이 있으면 내부 조인처럼, 없으면 널로 채워 보존"이라는 좌외부 조인의 정의를 정확히 재현합니다. 앞서 OpenCypherTranspiler의 조인 방향 오류와 대비해 보면, Graphiti의 이 규칙이 보존 방향을 정확히 지킨다는 점이 형식적으로 확립됩니다.

마지막으로 $C = \mathsf{With}(C\_1, \overline{Y}, \overline{Z})$ 사례에서는, $P[\overline{Y} \mapsto \overline{Z}]$와 $T[\overline{Y} \mapsto \overline{Z}]$의 정의에 따라 속성 키 이름을 바꾸는 동작이 SQL의 이름 변경 연산 $\rho\_{L[\overline{Z}/\overline{Y}]}$와 정확히 일치함을 보여, $\Phi\_{\mathsf{sdt}}(\llbracket\mathsf{With}(C\_1, \overline{Y}, \overline{Z})\rrbracket\_G) = \llbracket\Pi\_{\rho\_{L[\overline{Z}/\overline{Y}]}}(Q)\rrbracket\_R$이 성립합니다.
## 패턴 번역의 정확성 (Lemma F.3)

앞서 절 번역의 정확성을 다루면서 패턴 정확성 보조정리에 의존했는데, Lemma F.3가 바로 그 토대입니다. 이 보조정리는 $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash PP \overset{\textsf{pattern}}{\longrightarrow} \mathcal{X}, Q$이면 $G \sim\_{\Phi\_{\mathsf{sdt}}} R$인 모든 $G, R$에 대해 $\Phi\_{\mathsf{sdt}}(\llbracket PP\rrbracket\_G) = \llbracket Q\rrbracket\_R$이 성립함을 단언하며, 증명은 경로 패턴 $PP$의 구조에 대한 귀납법으로 진행됩니다.

기저 사례는 단일 노드 패턴 $PP = NP$입니다. $G = (N, E, P, T)$이고 $NP = (X, l)$일 때, 앞서 표시적 의미론에서 정의한 노드 패턴의 의미론에 의해 $\llbracket NP\rrbracket\_G$는 레이블이 $l$인 노드들 각각을 단일 노드 부분 그래프로 만든 리스트가 됩니다. 한편 SQL 측에서는 대응 테이블에 이름 변경을 적용한 튜플 리스트 $\mathcal{T} = \rho\_X(R(\Lambda(l)))$가 도출되며, 각 부분 그래프 $g\_i$와 튜플 $t\_i$ 사이에 $\Phi\_{\mathsf{sdt}}(g\_i) = t\_i$가 성립합니다. 따라서 다음이 곧바로 따라옵니다.

$$
\Phi\_{\mathsf{sdt}}(\llbracket NP\rrbracket\_G) = \rho\_X(R(\Lambda(l))) = \llbracket\rho\_X(\Lambda(l))\rrbracket\_R
$$

즉 그래프에서 레이블 $l$의 노드들을 골라 변수 $X$로 이름을 붙이는 작업이, 관계형에서 대응 테이블 $\Lambda(l)$을 $X$로 이름 변경하는 작업과 정확히 대응합니다.

### 경로 확장의 귀납 단계

귀납 사례 $PP = NP\_1, EP\_2, PP'$는 한층 흥미롭습니다. $NP\_1 = (X\_1, l\_1)$, $EP\_2 = (X\_2, l\_2, d\_2)$, 그리고 나머지 경로의 머리 노드를 $\mathsf{head}(PP') = NP\_3 = (X\_3, l\_3)$로 둡니다. 증명의 핵심은 단순 패턴 $[NP\_1, EP\_2, NP\_3]$에 매칭되는 부분 그래프를 찾는 $\mathsf{SubGraphs}$ 함수가 SQL의 세 테이블 조인으로 정확히 환원된다는 점입니다.

$$
\Phi\_{\mathsf{sdt}}(\mathsf{SubGraphs}(G, [NP\_1, EP\_2, NP\_3])) = \llbracket\rho\_{X\_1}(\Lambda(l\_1)) \bowtie\_{\phi\_1} \rho\_{X\_2}(\Lambda(l\_2)) \bowtie\_{\phi\_2} \rho\_{X\_3}(\Lambda(l\_3))\rrbracket\_R
$$

여기서 조인 술어 $\phi\_1 = \mathsf{link}(\xi, \Lambda(l\_1), \Lambda(l\_2))$와 $\phi\_2 = \mathsf{link}(\xi, \Lambda(l\_2), \Lambda(l\_3))$는 본문에서 정의한 $\mathsf{link}$ 함수로 주어집니다. 이는 앞서 엣지 규칙이 생성한 외래 키 제약을 바탕으로, 엣지 테이블을 그 출발·도착 노드 테이블과 잇는 조인 조건입니다.

증명에서 주목할 미묘한 부분은, 한 노드 패턴 $NP\_1, EP\_2, NP\_3$로부터 도출된 부분 그래프 $g''$와 나머지 경로 $PP'$로부터 도출된 부분 그래프 $g'$를 병합할 때, **고립된 노드 없이 병합되려면 두 부분 그래프의 겹치는 노드가 반드시 $NP\_3$에서 나온 노드여야 한다**는 관찰입니다. $NP\_3$가 두 경로 조각이 만나는 연결점이기 때문인데, 이 조건이 충족되어야만 병합된 그래프가 $\mathsf{SubGraphs}$의 유효한 원소가 됩니다. 이 관찰 덕분에 $g'$와 교집합이 비어 있지 않은 부분 그래프만 골라 병합하는 $\mathsf{filter}$ 동작이, 연결점 노드의 기본 키에서 조인하는 술어 $\phi\_3 = \mathsf{link}(\xi, \Lambda(l\_3), \Lambda(l\_3))$로 정확히 표현됩니다. 최종적으로 $\mathsf{foldl}$ 누적을 통해 전체 경로가 다음과 같이 귀납적으로 조립됩니다.

$$
\Phi\_{\mathsf{sdt}}(\llbracket NP\_1, EP\_2, PP'\rrbracket\_G) = \llbracket\rho\_{X\_1}(\Lambda(l\_1)) \bowtie\_{\phi\_1} \rho\_{X\_2}(\Lambda(l\_2)) \bowtie\_{\phi\_2} Q'\rrbracket\_R
$$

여기서 $Q'$은 귀납 가정으로 얻은 $PP'$의 번역 결과입니다. 즉 한 홉짜리 단순 패턴 매칭 위에 나머지 경로의 번역을 조인으로 이어 붙임으로써, 임의 길이의 경로 패턴 전체가 일련의 조인으로 정확히 번역됨이 증명됩니다.

## 표현식 번역의 정확성 (Lemma F.4)

Lemma F.4는 표현식 수준의 정확성을 보장하며, 앞선 건전성 증명에서 표현식 평가값의 일치를 떠받친 핵심 도구입니다. 이 보조정리는 $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash E \overset{\textsf{expr}}{\longrightarrow} E'$이고 $\forall i.\ \Phi\_{\mathsf{sdt}}(g\_i) = t\_i$일 때 $\llbracket E\rrbracket\_{G,gs} = \llbracket E'\rrbracket\_{R,\mathcal{T}}$가 성립함을 단언합니다. 즉 그래프의 부분 그래프 리스트 $gs$ 위에서 표현식 $E$를 평가한 값과, 대응되는 튜플 리스트 $\mathcal{T}$ 위에서 번역된 표현식 $E'$를 평가한 값이 정확히 같다는 것입니다. 증명은 표현식 $E$의 구조에 대한 귀납법이며, 대부분의 경우가 거의 자명하지만 집계 연산자들의 $\mathsf{Null}$ 처리가 핵심입니다.

두 기저 사례부터 보면, 속성 키 $E = k$는 양쪽 모두 매칭 결과의 첫 원소에서 같은 키를 조회하므로 $\llbracket k\rrbracket\_{G,gs} = \mathsf{lookup}(\mathsf{head}(gs), k) = \mathsf{lookup}(\mathsf{head}(\mathcal{T}), k) = \llbracket k\rrbracket\_{R,\mathcal{T}}$가 성립하고, 값 상수 $E = v$는 문맥과 무관하게 양쪽 모두 $v$로 평가됩니다.

집계 연산자의 경우, 예컨대 $E = \mathsf{Count}(E\_1)$에서는 귀납 가정으로 내부 표현식의 일치 $\llbracket E\_1\rrbracket\_{G,gs} = \llbracket E\_1'\rrbracket\_{R,\mathcal{T}}$를 얻습니다. 앞서 표시적 의미론에서 살펴본 $\mathsf{Count}$의 정의—그룹의 모든 원소가 $\mathsf{Null}$이면 $\mathsf{Null}$, 아니면 $\mathsf{Null}$이 아닌 값의 개수를 세는—가 그래프 측과 SQL 측 양쪽에서 동일한 형태를 띠므로, $\mathsf{foldl}$과 $\mathsf{map}$이 동일한 구조로 작동하여 결과가 일치합니다. $\mathsf{Sum}$, $\mathsf{Min}$, $\mathsf{Max}$도 동일한 논리를 따르되 누적 함수와 중립 원소만 다릅니다. 특히 $\mathsf{Avg}(E\_1)$는 귀납 가정으로 $\mathsf{Sum}$과 $\mathsf{Count}$의 일치를 각각 얻은 뒤 그 비율로 정의되므로, 두 부분의 일치로부터 평균의 일치가 따라옵니다. 산술 연산 $E = E\_1 \oplus E\_2$는 두 피연산자의 평가값이 각각 일치하므로 같은 연산자 $\oplus$를 적용한 결과도 일치합니다. $\mathsf{Cast}(\phi)$의 경우는 술어 정확성 보조정리에 의해 $\llbracket\phi\rrbracket\_{G,gs} = \llbracket\phi'\rrbracket\_{R,\mathcal{T}}$를 얻고, 삼치 논리에 따른 캐스팅 정의가 양쪽에서 동일하므로 결과가 일치합니다.

## 술어 번역의 정확성 (Lemma F.5)

Lemma F.5는 술어 수준의 정확성을 보장하며, $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash \phi \overset{\textsf{pred}}{\longrightarrow} \phi'$일 때 $\llbracket\phi\rrbracket\_{G,gs} = \llbracket\phi'\rrbracket\_{R,\mathcal{T}}$가 성립함을 단언합니다. 참·거짓·비교·논리 결합 같은 대부분의 경우는 하위 구성 요소의 일치로부터 곧바로 따라오는 합동 논증이지만, **존재 술어 $\phi = \mathsf{Exists}(PP)$의 경우가 유일하게 비자명**합니다.

이 경우 증명은 패턴 정확성 보조정리로 $\Phi\_{\mathsf{sdt}}(\llbracket PP\rrbracket\_G) = \llbracket Q\rrbracket\_R$을 얻는 데서 출발합니다. 그런 다음 경로 패턴의 머리 노드 $(X\_1, l\_1)$와 꼬리 노드 $(X\_2, l\_2)$의 기본 키로 이루어진 키 리스트 $\overline{K} = [\mathsf{PK}(\mathsf{head}(PP)), \mathsf{PK}(\mathsf{last}(PP))]$를 두면, $\Phi\_{\mathsf{sdt}}(g\_i) = t\_i$라는 가정 아래에서 양 끝 노드의 기본 키 평가값이 다음과 같이 일치합니다.

$$
\llbracket\overline{K}\rrbracket\_{G,gs} = \llbracket[\xi\_{pk}(\Lambda(l\_1)), \xi\_{pk}(\Lambda(l\_2))]\rrbracket\_{R,\mathcal{T}} = \llbracket\overline{K}\rrbracket\_{R,\mathcal{T}}
$$

이 일치를 바탕으로, 패턴의 모든 부분 그래프에 대해 양 끝 키를 평가한 리스트가 SQL 측의 투영 $\Pi\_{\overline{K}}(Q)$와 정확히 대응함을 보입니다.

$$
\mathsf{map}(\lambda g.\llbracket\overline{K}\rrbracket\_{G,[g]}, \llbracket PP\rrbracket\_G) = \llbracket\Pi\_{\overline{K}}(Q)\rrbracket\_R
$$

최종적으로 $\mathsf{Exists}$의 의미론—"양 끝 노드의 기본 키가 일치하는 부분 그래프가 하나라도 존재하는가"를 논리합으로 따지는—이 $\mathsf{foldl}$을 거쳐 SQL의 멤버십 술어 $\overline{K} \in \Pi\_{\overline{K}}(Q)$로 정확히 환원됨이 증명됩니다.

$$
\llbracket\mathsf{Exists}(PP)\rrbracket\_{G,gs} = \llbracket\overline{K} \in \Pi\_{\overline{K}}(Q)\rrbracket\_{R,\mathcal{T}}
$$

이는 앞서 동기 예제와 부록의 P-Exists 규칙에서 강조했던 핵심—존재 검사가 경로의 개수가 아니라 존재 여부만을 따지므로 중복 집계를 일으키지 않는다는 사실—을 의미론 수준에서 형식적으로 확립합니다.

## 번역의 완전성 증명 (Theorem F.6)

건전성이 "번역된 SQL이 의미를 왜곡하지 않음"을 보장한다면, 완전성 정리(Theorem F.6, 본문 Theorem 5.8)는 그 반대 방향, 즉 **Featherweight Cypher 문법이 받아들이는 임의의 Cypher 질의 $Q$에 대해 반드시 대응하는 SQL 질의 $Q'$이 존재하여 번역 관계 $\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash Q \overset{\textsf{query}}{\longrightarrow} Q'$이 성립함**을 보장합니다. 다시 말해 번역 규칙이 다루는 Cypher 조각 안에서는 어떤 질의도 번역에 실패하지 않는다는 것입니다. 증명은 질의 $Q$의 구조에 대한 귀납법으로, 각 질의 형태마다 적용 가능한 번역 규칙이 존재함을 보입니다.

기저 사례 $Q = \mathsf{Return}(C, \overline{E}, \overline{k})$에서는 절 번역 완전성 보조정리(Lemma F.7)에 의해 절 $C$를 번역한 SQL 질의 $Q'$이 존재하고, 표현식 번역 완전성 보조정리(Lemma F.9)에 의해 각 표현식 $E$에 대응하는 SQL 표현식 $E'$이 존재합니다. 이후 집계 유무에 따라 두 갈래로 나뉘는데, 집계가 없으면($\neg\mathsf{hasAgg}(\overline{E}) = \top$) Q-Ret 규칙으로 투영 질의 $Q'' = \Pi\_{\rho\_{\overline{k}}(\overline{E'})}(Q')$가, 집계가 있으면 Q-Agg 규칙으로 $\texttt{GroupBy}(Q, \overline{A}, \rho\_{\overline{k}}(\overline{E'}), \top)$가 존재함이 보장됩니다. 여기서 $\overline{A} = \mathsf{filter}(\lambda x.\neg\mathsf{IsAgg}(x), \overline{E'})$는 집계를 포함하지 않는 SQL 표현식 리스트, 곧 그룹화 키입니다.

귀납 사례들도 각각 대응 규칙의 존재를 확인합니다. $\mathsf{OrderBy}(R\_1, k, b)$에서는 귀납 가정으로 $R\_1$의 번역이 존재하고, 표현식·술어 완전성 보조정리로 정렬 키와 방향의 번역이 존재하므로 Q-OrderBy 규칙이 적용됩니다. $\mathsf{Union}(Q\_1, Q\_2)$와 $\mathsf{UnionAll}(Q\_1, Q\_2)$에서는 두 하위 질의의 번역이 귀납 가정으로 존재하므로 각각 $Q\_1' \cup Q\_2'$와 $Q\_1' \uplus Q\_2'$를 만드는 Q-Union 규칙이 적용됩니다.

### 절·패턴·표현식·술어의 완전성 보조정리

완전성 정리는 네 개의 완전성 보조정리에 의존하며, 이들은 각각 절(F.7), 패턴(F.8), 표현식(F.9), 술어(F.10)의 번역이 항상 존재함을 보장합니다.

절 완전성 보조정리(Lemma F.7)는 임의의 Cypher 절 $C$에 대해 SQL 질의 $Q$가 존재함을 구조적 귀납으로 보입니다. 기저 사례 $C = \mathsf{Match}(PP, \phi)$에서는 패턴 완전성(F.8)과 술어 완전성(F.10)으로 각각 $Q$와 $\phi'$의 존재를 얻은 뒤 C-Match1 규칙으로 $\sigma\_{\phi'}(Q)$를 구성합니다. 연쇄 매칭 $\mathsf{Match}(C\_1, PP, \phi)$에서는 두 새로운 별칭 $T\_1, T\_2$를 도입하고 공통 변수의 기본 키 일치를 강제하는 새 술어 $\phi''$를 만들어 C-Match2 규칙으로 내부 조인 $\rho\_{T\_1}(Q\_1) \bowtie\_{\phi''} \rho\_{T\_2}(Q\_2)$를 구성합니다. 선택적 매칭 $\mathsf{OptMatch}$의 경우도 동일한 구조이되 좌외부 조인을 사용하며, $\mathsf{With}(C\_1, \overline{Y}, \overline{Z})$의 경우는 C-With 규칙으로 이름 변경 투영 $\Pi\_{\rho\_{L[\overline{Z}/\overline{Y}]}}(Q\_1)$를 구성합니다.

패턴 완전성 보조정리(Lemma F.8)는 단일 노드 패턴에 대해 PT-Node 규칙으로 $\rho\_X(\Lambda(l))$이, 확장된 경로 패턴에 대해 PT-Path 규칙으로 적절한 조인 질의가 존재함을 보입니다. 표현식 완전성(Lemma F.9)과 술어 완전성(Lemma F.10)은 부록에서 다룬 표현식·술어 번역 규칙들(E-Prop, E-Value, E-Pred, E-Agg, E-Arith 및 P-True, P-False, P-Logic, P-IsNull, P-In, P-Exists, P-AndOr, P-Not)이 각 구문 형태를 빠짐없이 다루므로, 어떤 표현식·술어도 번역 대상이 됨을 보장합니다. 특히 P-Exists 규칙에서는 패턴 완전성으로 얻은 서브쿼리 $Q$와 양 끝 노드의 기본 키 리스트 $\overline{a}$로부터 멤버십 술어 $\overline{a} \in \Pi\_{\overline{a}}(Q)$가 구성됩니다.

## 잔여 변환기의 정확성 (Lemma F.11)

지금까지의 정리들이 표준 변환기 $\Phi\_{\mathsf{sdt}}$를 기준으로 한 번역의 정확성을 다루었다면, 이제 전체 방법론을 완성하기 위해서는 유도 스키마와 목표 스키마를 잇는 잔여 변환기 $\Phi\_{\mathsf{rdt}}$의 정확성이 필요합니다. Lemma F.11은 잔여 변환기가 두 관계형 인스턴스를 올바르게 정렬함을 보장합니다. 구체적으로, $D' = \Phi\_{\mathsf{sdt}}(G)$가 유도 스키마 $\Psi\_{R'}$ 위의 인스턴스이고 $D = \Phi(G)$가 목표 스키마 $\Psi\_R$ 위의 인스턴스일 때, 본문에서 단 한 줄의 구문적 치환으로 얻은 잔여 변환기 $\Phi\_{\mathsf{rdt}}$가 $D' \sim\_{\Phi\_{\mathsf{rdt}}} D$를 만족함을 보입니다.

증명은 본문의 ReduceToSQL 알고리즘이 잔여 변환기를 구문적 치환으로 얻는다는 점에 착안하여 노드와 엣지 두 경우로 나뉩니다. 노드의 경우, $\Phi\_{\mathsf{sdt}}$ 안에 레이블 $l$을 유도 스키마 테이블 $l'$로 보내는 절 $l(a\_1, \ldots, a\_n) \to l'(a\_1, \ldots, a\_n)$이 존재하므로, 치환 사상 $\sigma$와 잔여 변환기가 각각 $\sigma \cup \{l \mapsto l'\}$과 $\Phi\_{\mathsf{rdt}} \cup \Phi[l \mapsto l']$로 갱신됩니다. 원래 사용자 변환기 $\Phi$가 그래프 레이블 $l$을 통해 목표 테이블 $R\_l$을 기술하던 절 $l(a\_1, \ldots, a\_n) \to R\_l(a\_1', \ldots, a\_n')$에 치환 $[l \mapsto l']$를 적용하면, 유도 스키마 테이블 $l'$을 목표 테이블 $R\_l$로 보내는 절 $l'(a\_1, \ldots, a\_n) \to R\_l(a\_1', \ldots, a\_n')$이 얻어집니다. 따라서 노드 $N\_l$에 대응하는 유도 인스턴스 $D\_{l'}$와 목표 인스턴스 $D\_{R\_l}$ 사이에 $\Phi\_{\mathsf{rdt}}(D\_{l'}) = D\_{R\_l}$이 성립합니다.

엣지의 경우는 출발 노드 $s$와 도착 노드 $t$가 추가로 관여한다는 점만 다릅니다. 엣지 절 $s(\ldots), l(s, t, a\_1, \ldots, a\_n), t(\ldots) \to s'(\ldots), l'(a\_1, \ldots, a\_n, s, t), t'(\ldots)$에서, $s$와 $t$가 노드 레이블이므로 노드 경우의 결과를 그대로 활용할 수 있습니다. 즉 $s'(\ldots) \to R\_s(\ldots)$와 $t'(\ldots) \to R\_t(\ldots)$가 이미 $\Phi\_{\mathsf{rdt}}$에 존재하므로, 세 치환을 함께 적용한 $\Phi[s \mapsto s', l \mapsto l', t \mapsto t']$가 유도 스키마의 엣지 인스턴스를 목표 인스턴스로 올바르게 사상합니다. 결과적으로 다음이 확립됩니다.

$$
D' \triangleright \Psi\_R' \land D \triangleright \Psi\_R \land \Phi\_{\mathsf{rdt}}(D') = D \Rightarrow D' \sim\_{\Phi\_{\mathsf{rdt}}} D
$$

## 동치성의 전이 (Lemma F.12)

Lemma F.12는 세 변환기—표준 변환기 $\Phi\_{\mathsf{sdt}}$, 사용자 변환기 $\Phi$, 잔여 변환기 $\Phi\_{\mathsf{rdt}}$—를 통해 동치성이 전이됨을 보이는 다리 역할을 합니다. 구체적으로, 그래프 질의 $Q\_G$가 SQL 질의 $Q\_{R'}$로 번역되고($\Phi\_{\mathsf{sdt}}, \Psi\_R \vdash Q\_G \overset{\textsf{query}}{\longrightarrow} Q\_{R'}$), $Q\_G$가 사용자 변환기 $\Phi$를 기준으로 목표 질의 $Q\_R$과 동치($Q\_G \simeq\_\Phi Q\_R$)라면, 번역된 질의 $Q\_{R'}$이 잔여 변환기를 기준으로 $Q\_R$과 동치($Q\_{R'} \simeq\_{\Phi\_{\mathsf{rdt}}} Q\_R$)임을 보장합니다.

증명의 논리는 동치성 정의들을 사슬처럼 연결하는 것입니다. 먼저 $Q\_G \simeq\_\Phi Q\_R$의 정의로부터 $\Phi(G) = D$인 그래프·관계형 인스턴스 쌍에 대해 $\llbracket Q\_G\rrbracket\_G \equiv \llbracket Q\_R\rrbracket\_D$를 얻습니다. 한편 잔여 변환기 정확성(Lemma F.11)으로 $\Phi\_{\mathsf{rdt}}(D') = D$를 얻고, 번역 건전성(Lemma F.1, 즉 Theorem F.1)으로 $\llbracket Q\_G\rrbracket\_G \equiv \llbracket Q\_{R'}\rrbracket\_{D'}$을 얻습니다. 이 세 등식을 결합하면 $\llbracket Q\_R\rrbracket\_D \equiv \llbracket Q\_G\rrbracket\_G \equiv \llbracket Q\_{R'}\rrbracket\_{D'}$이 성립하므로, 결국 $Q\_{R'} \simeq\_{\Phi\_{\mathsf{rdt}}} Q\_R$, 즉 $\forall D', D.\ (D' \triangleright \Psi\_{R'} \land D \triangleright \Psi\_R \land D' \sim\_{\Phi\_{\mathsf{rdt}}} D) \Rightarrow \llbracket Q\_{R'}\rrbracket\_{D'} \equiv \llbracket Q\_R\rrbracket\_D$가 증명됩니다. 이 보조정리가 바로 "이질적인 Cypher-SQL 동치성을 동질적인 SQL-SQL 동치성으로 환원할 수 있다"는 전체 전략의 형식적 정당화입니다.

## 전체 방법론의 건전성과 완전성 (Theorem F.13, F.14)

마지막으로 두 정리가 전체 방법론의 정확성을 못 박습니다. 두 정리 모두 백엔드 SQL 검사기 $\mathsf{CheckSQL}$의 능력을 전제로 한 조건부 형태로 진술된다는 점이 핵심인데, 이는 앞서 본문에서 강조했듯 Graphiti의 전체 정확성이 플러그인 가능한 SQL 검사기에 위임되는 모듈식 설계를 반영합니다.

건전성 정리(Theorem F.13, 본문 Theorem 5.9)는 $\mathsf{CheckSQL}$이 건전한 동치성 검사 절차일 때, `CheckEquivalence`$(\Psi\_G, Q\_G, \Psi\_R, Q\_R, \Phi)$가 $\top$을 반환하면 실제로 $Q\_G \simeq\_\Phi Q\_R$이 성립함을 보장합니다. 증명은 이렇게 흐릅니다. 번역 건전성으로 $Q\_G \simeq\_{\Phi\_{\mathsf{sdt}}} Q\_{R'}$이 성립하므로, $G \sim\_{\Phi\_{\mathsf{sdt}}} D'$인 임의의 쌍에 대해 $\llbracket Q\_G\rrbracket\_G \equiv \llbracket Q\_{R'}\rrbracket\_{D'}$입니다. 또한 `CheckEquivalence`가 $\top$을 반환하면 $\mathsf{CheckSQL}$도 $\top$을 반환하므로, 잔여 변환기로 연결된 $\llbracket Q\_{R'}\rrbracket\_{D'} \equiv \llbracket Q\_R\rrbracket\_D$가 성립합니다. 두 등식을 이으면 $\llbracket Q\_G\rrbracket\_G \equiv \llbracket Q\_{R'}\rrbracket\_{D'} \equiv \llbracket Q\_R\rrbracket\_D$이 되고, 가정 $G \sim\_\Phi D$ 아래에서 $Q\_G \simeq\_\Phi Q\_R$이 증명됩니다. 즉 알고리즘이 "동치"라고 선언하면 그것은 항상 진실입니다.

완전성 정리(Theorem F.14, 본문 Theorem 5.10)는 그 역방향으로, $\mathsf{CheckSQL}$이 완전한 절차일 때 $Q\_G \simeq\_\Phi Q\_R$이 성립하면 반드시 `CheckEquivalence`가 $\top$을 반환함을 보장합니다. 증명은 앞서 확립한 동치성 전이(Lemma F.12)에 직접 기댑니다. $Q\_G \simeq\_\Phi Q\_R$이 성립하면 Lemma F.12에 의해 $Q\_{R'} \simeq\_{\Phi\_{\mathsf{rdt}}} Q\_R$이 따라오고, $\mathsf{CheckSQL}$이 완전하다면 이 SQL 동치성을 반드시 검출하여 $\top$을 반환하므로, 결국 전체 절차도 $\top$을 반환합니다. 다시 말해 두 질의가 정말로 동치라면 알고리즘은 결코 그것을 놓치지 않습니다.

이 두 정리가 합쳐짐으로써, 이질적인 그래프-관계형 질의 동치성 문제는 형식적으로 보장된 트랜스파일레이션과 단 한 줄의 잔여 변환기 추론, 그리고 잘 연구된 SQL 동치성 검사로 완전히 환원되며, 그 환원이 건전하고도 완전함이 빈틈없이 증명됩니다. 다만 일반적인 SQL 동치성 검사가 결정 불가능하다는 근본적 한계 때문에 완전한 백엔드를 항상 가질 수는 없으므로, 실제로는 [VeriEQL](https://doi.org/10.1145/3649849)의 유계 모델 검사(반박에 강함)와 Mediator의 연역적 검증(무계 증명에 강함)을 조합하여 검증과 반박 양쪽에서 실용적 성능을 끌어내게 됩니다. 이로써 부록의 모든 형식적 증명이 본문 정리들의 진리성을 떠받치며, Graphiti의 이론적 토대가 완결됩니다.
